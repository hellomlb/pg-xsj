文件描述符耗尽：让 PostgreSQL 宕机的 OS 级限制 —— 三招诊断，一套修复

2026-06-05 日常运维 文件描述符 FD耗尽 OS层 故障诊断 PgBouncer 连接池 ulimit

一句话总结

一个 PG 实例突然报错 Too many open files，连接断开、查询失败、WAL 写入报错——DBA 第一反应通常是"PG 出了什么问题"。但大多数情况下，故障点完全不在 PG 代码内，根因在 OS 层：内核的文件描述符限制太低了。

这篇文章给出 三步诊断法 + 一套完整修复方案——读完就能用，无需任何中间假设。

问题是什么：文件描述符（FD）是什么

文件描述符是操作系统分配给进程的整数句柄，用于访问文件、网络套接字、管道等一切 I/O 资源。每个 PG 后端进程需要 FD 处理：

  • 客户端连接（每个连接至少 1 个 FD）
  • 数据文件（每个表至少 1 个 FD）
  • 索引文件（每个索引至少 1 个 FD）
  • WAL 文件
  • 临时文件（排序/哈希溢出时）

当所有进程的 FD 总和超过内核上限时，PG 无法打开新文件或接受新连接——外表看起来是 PG 宕机，实则是 OS 在喊"没资源了"。

触发条件：什么情况最容易 FD 耗尽

场景FD 消耗来源风险等级
高并发连接数每个连接 = 1 个 FD，1000 连接 = 1000 FD 起步⭐⭐⭐⭐⭐
复杂查询 + work_mem 不足排序/哈希溢出产生大量临时文件⭐⭐⭐⭐
大量表分区数千分区查询时可能同时打开所有分区文件⭐⭐⭐
系统默认限制 1024Linux 发行版默认 nofile=1024，对 PG 完全不适用⭐⭐⭐⭐⭐

诊断三步法

第一步：查日志——确认是不是 FD 耗尽

在 PG 日志（$PGDATA/log/ 或 syslog）中搜索：

grep -i "Too many open files" $PGDATA/log/*.log
grep -i "could not open file" $PGDATA/log/*.log | head -20

如果看到类似以下消息，基本可以确认是 FD 耗尽：

could not open file "base/16384/2619": Too many open files
could not accept new connection: Too many open files

第二步：查 OS 层限制——看看"天花板"有多低

# 系统级 FD 上限
sysctl fs.file-max

# 已分配 FD / 上限 / 峰值
cat /proc/sys/fs/file-nr
# 输出三列: <全系统已分配FD数> <空闲FD数> <系统级上限(fs.file-max)>
# 注意: 第1列的"已分配"是所有进程之和，不是单个进程

# 当前进程的软/硬限制
ulimit -n       # soft limit
ulimit -Hn      # hard limit

# PG 进程的实际限制（替换为实际 PID）
cat /proc/$(pgrep -f "postgres.*checkpointer" | head -1)/limits | grep "Max open files"

三个层级的 FD 限制，别搞混：

参数层级含义
fs.file-max内核/系统级全系统所有进程 FD 之和的上限
/proc/sys/fs/file-nr 第1列实时状态当前全系统已分配的 FD 总数
ulimit -n / limits.conf nofile进程级单个进程能打开的 FD 数

检查项危险值推荐值
fs.file-max< 65536≥ 262144
ulimit -n (soft)< 4096≥ 65536
已分配 FD 占比> 50%< 30%

第三步：查 PG 进程 FD 实时使用量

# 统计所有 PG 进程的 FD 总数
for pid in $(pgrep -f "^postgres:"); do
    ls /proc/${pid}/fd 2>/dev/null | wc -l
done | awk '{s+=$1} END {print "Total FDs:", s}'

# 找出 FD 使用最高的进程
for pid in $(pgrep -f "^postgres:" | head -30); do
    count=$(ls /proc/${pid}/fd 2>/dev/null | wc -l)
    echo "PID $pid: $count FDs"
done | sort -t: -k2 -n | tail -5

再结合 PG 内置视图确认连接数和临时文件压力：

-- 当前连接数
SELECT count(*) AS total_connections,
       count(*) FILTER (WHERE state = 'active') AS active_connections,
       count(*) FILTER (WHERE state = 'idle') AS idle_connections,
       count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_txn
FROM pg_stat_activity;
-- max_connections 配置
SELECT name, setting, unit
FROM pg_settings
WHERE name IN ('max_connections', 'max_files_per_process');
-- 临时文件使用量
SELECT datname,
       temp_files,
       pg_size_pretty(temp_bytes) AS temp_size,
       stats_reset
FROM pg_stat_database
WHERE datname NOT IN ('template0', 'template1')
ORDER BY temp_bytes DESC;
-- 表分区数量（分区多 → FD 压力大）
SELECT count(*) AS partition_count
FROM pg_inherits i
JOIN pg_class c ON c.oid = i.inhrelid
WHERE c.relkind = 'r';

修复方案

修复分两层——OS 层治标（提上限），PG 层治本（降消耗）。两层必须同时做，只改 limits.conf 不提连接池，早晚还会爆。

OS 层修复：提上限

方案 A：/etc/security/limits.conf（最常用）

postgres    soft    nofile    65536
postgres    hard    nofile    65536

改完要重新登录或重启 PG 进程才能生效（limits.conf 由 PAM 在登录时应用）。

方案 B：/etc/sysctl.conf（系统级上限）

fs.file-max = 262144

执行 sysctl -p 立即生效，重启后保持。

方案 C：systemd 服务文件（推荐，绕过 PAM 依赖）

# /etc/systemd/system/postgresql.service.d/limits.conf
[Service]
LimitNOFILE=65536
systemctl daemon-reload
systemctl restart postgresql

方案 D：Kubernetes (AKS/EKS/GKE)

在 Pod spec 中设置 securityContext 或通过 initContainer 调整：

apiVersion: v1
kind: Pod
spec:
  containers:
  - name: postgres
    securityContext:
      # 注意：部分 K8s 发行版限制此参数需与节点内核参数匹配
    resources:
      limits:
        # 确保 Pod 有足够资源
  # 或通过 privileged initContainer 调整节点 sysctl

K8s 环境下更推荐的做法是在节点层面统一管理 sysctl，而非逐 Pod 调整。

PG 层修复：降消耗

这是治本之策——光提高上限是饮鸩止渴，真正需要做的是减少 FD 消耗。

核心方案：部署 PgBouncer 连接池

PgBouncer 把几百上千个客户端连接合并到几十个后端连接，FD 消耗从 O(Nₑₗᵢₑₙₜₛ) 降到 O(Nₑₐₗ₋cₒₙₙₑcₜᵢₒₙₛ)。

# pgbouncer.ini 关键配置
[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432
pool_mode = transaction          # 事务级连接池，效率最高
default_pool_size = 20           # 每个数据库的后端连接池大小
max_client_conn = 1000           # 最大客户端连接数

[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

部署 PgBouncer 后，将 PG 的 max_connections 降至 100-200：

ALTER SYSTEM SET max_connections = 200;
SELECT pg_reload_conf();

-- 验证
SELECT setting FROM pg_settings WHERE name = 'max_connections';

辅助方案：优化查询，减少临时文件

  • 适当增大 work_mem：减少排序/哈希溢出到磁盘
  • 设置 temp_file_limit：防止单个查询无限创建临时文件
  • 审查分区表查询：确保分区裁剪生效，避免全分区扫描

事后验证

修复后需确认生效：

-- 1. PG 层面
SELECT name, setting
FROM pg_settings

# 2. OS 层面——找个 PG 进程验证

你想解决的问题你该做的改了就能防住吗
"Too many open files" 报错limits.conf nofile=65536暂时防住，治标
连接太多，FD 一直增长PgBouncer / reduce max_connections✅ 治本
临时文件吃 FDwork_mem↑ + temp_file_limit✅
分区太多吃 FD审查分区策略 + 查询优化✅
K8s 环境限制特殊节点级 sysctl + Pod securityContext需要测试
长期监控 FD 趋势Prometheus node_exporter + 告警✅ 早发现早处理

最佳实践

  1. 部署 PgBouncer 比调整 limits.conf 更根本——连接池是 FD 消耗的结构性优化
  2. nofile=65536 应该是 PG 宿主机的标配，不要等爆了再设
  3. 监控 FD 使用趋势，设置告警阈值（如使用率 > 50%）
  4. 每次升级/迁移后重新检查 limits.conf 和 sysctl 配置是否保留
