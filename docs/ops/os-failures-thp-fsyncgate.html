PG 被冤枉的两次：THP 和 fsyncgate —— OS 层故障如何伪装成数据库问题

2026-06-05 日常运维 THP 透明大页 fsync data_sync_retry OS层 故障诊断 p99延迟

一句话总结

有两个 Linux 层故障，故障点完全不在 PG 代码内，但症状 100% 表现为 PG 问题——一个让你的 p99 延迟随机飙到几百毫秒却查不出任何慢 SQL，另一个让你的已提交事务在崩溃重启后凭空消失。它们加起来可能是 PG 运维中最容易被误诊的一对陷阱。

本文不是科普，是武器——每个故障给一个一刀斩方案，读完就能用。

故障① THP 透明大页 —— 「p99 延迟随机飙升，但查不出任何慢 SQL」

典型场景

你接到告警：某个核心接口 P99 延迟突然飙到 800ms。你打开 pg_stat_statements，发现该查询 mean_time 只有 0.3ms。打开 pg_stat_activity，没有锁等待、没有 I/O 等待。CPU 正常，内存正常，没有 swap。

一切 PG 层面的监控都告诉你「没有问题」，但用户说「就是慢」。

这时候你很可能遇到了 THP。

根因：khugepaged 的"好心办坏事"

Linux 内核有一个后台线程叫 khugepaged，它的工作是扫描进程内存，把连续的 512 个 4KB 小页合并成一个 2MB 大页——目的是减少 TLB miss，提高内存访问效率。

问题在于合并过程本身：

khugepaged 遍历进程页表
  → 找到 2MB 对齐的 512 个连续小页
  → 锁住这些页（❗）
  → 把内容拷贝到新的大页
  → 更新页表指针
  → 释放旧小页
  → 解锁

合并期间，PG 后端进程访问被锁定的页
  → 在内核态挂起等待
  → 挂起时间：几十到几百毫秒
  → PG 进程对此毫不知情——无日志、无 wait_event

为什么 PG 特别敏感？

  • PG 是多进程模型，每个连接一个独立进程——所有后端进程都可能被同时挂起
  • THP 的 defrag 模式加剧问题：always 模式下只要有连续页就尝试合并，PG 的排序/哈希操作大量分配临时内存，恰好满足合并条件
  • PG 的 shared_buffers 已经自己管理大块内存，THP 完全没有正面收益

一刀斩

# 即时生效
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# 持久化（二选一）

# 方法 1：grub 内核参数
# /etc/default/grub
GRUB_CMDLINE_LINUX="... transparent_hugepage=never"
grub2-mkconfig -o /boot/grub2/grub.cfg

# 方法 2：systemd service（不影响 grub）
cat > /etc/systemd/system/disable-thp.service << 'EOF'
[Unit]
Description=Disable Transparent Huge Pages
After=sysinit.target local-fs.target
Before=postgresql.service

[Service]
Type=oneshot
ExecStart=/bin/sh -c 'echo never > /sys/kernel/mm/transparent_hugepage/enabled && echo never > /sys/kernel/mm/transparent_hugepage/defrag'

[Install]
WantedBy=multi-user.target
EOF
systemctl enable disable-thp.service

验证

cat /sys/kernel/mm/transparent_hugepage/enabled
# 应输出: always madvise [never]  ← 看方括号位置

cat /sys/kernel/mm/transparent_hugepage/defrag
# 应输出: always madvise [never]

# 确认 khugepaged 不在运行
ps aux | grep '[k]hugepaged'
# 无输出 = 安全

故障② fsyncgate —— 「事务 COMMIT 了，但数据没写进磁盘」

典型场景

数据库正常运行，一切监控指标绿色。某天服务器意外重启（或 SAN 网络抖动、磁盘坏道触发底层错误），重启后发现：部分已提交的事务凭空消失，数据对不上。

你查 WAL 日志——提交记录都在。查 PG 日志——无任何错误。数据就这么没了。

这就是 fsyncgate。

根因：Linux 内核在写入错误后"遗忘"了错误

整个过程只有 5 步：

① PG 执行 INSERT，数据写入 OS 页缓存 → write() 返回成功
② PG 调用 fsync()，要求将脏页刷到磁盘
③ 磁盘驱动层写入失败（SAN 抖动/坏道/NFS 超时）
④ Linux 内核块设备层检测到错误，但——
   ❌ 内核的错误处理逻辑清除了错误标志
   ✅ 后续 fsync() 看到"干净"标志，返回 0（成功）
⑤ PG 收到 fsync() = 成功 → 认为数据已安全落盘 → 提交完成

结果：事务提交成功，但磁盘上什么都没有。

这个 Bug 不是 PG 的。受影响的内核版本横跨多年，触发条件苛刻：必须底层存储返回写入错误 + 内核恰好清除错误状态。SAN/NFS 网络抖动是最常见触发场景，普通本地磁盘也可能因坏道触发。

PG 的应对：宁可崩溃，绝不丢数据

版本 data_sync_retry 行为
PG 9.4 ~ 10 不支持 无条件信任 fsync 返回值 → 可能丢数据
PG 11 ~ 18 on（默认） fsync 失败 → 重试 → 与旧版同风险 → 可能丢数据
PG 11 ~ 18 off ✅ fsync 失败 → 立即 PANIC → 崩溃恢复 → WAL replay 重新写入 → 安全

data_sync_retry = off 的逻辑很简单：既然不能信任 fsync 的返回值，那就不要信任——失败了就崩溃，用 WAL 日志在恢复时重新写。

一刀斩

-- 检查当前值
SHOW data_sync_retry;

-- 如果不是 off：
ALTER SYSTEM SET data_sync_retry = off;
SELECT pg_reload_conf();

-- 再次确认
SHOW data_sync_retry;  -- 应为 off

⚠️ 注意：PG 9.4 ~ 10 不支持此参数，升级是唯一的修复路径。如果还在用这些版本，fsyncgate 的风险是实打实存在的。

不可选的替代方案

方案 行得通？ 为什么
信任硬件 RAID 卡 ❌ RAID 卡也会返回假成功
用 ZFS/Btrfs ✅ 文件系统层 checksum 可以检测
PG data_checksums 部分 能发现已损坏，不能防止丢失

两个故障的共性

维度 THP fsyncgate
故障层 Linux 内存管理 Linux 块设备层
PG 表现 p99 延迟随机飙升 已提交数据静默丢失
PG 日志 完全正常 完全正常
修复位置 OS 参数（一行 echo） PG 参数（一行 ALTER SYSTEM）
影响面 所有 PG 版本 PG 9.4~10（无修复）；PG 11+（未设 off 时）
误诊率 极高（症状完美模仿 PG 慢查询） 极高（症状完美模仿 PG 数据损坏）

两者的共同特征：故障点不在 PG 代码内，但症状 100% 表现为 PG 故障。这是它们容易被误诊的根本原因——出问题时，所有人的第一反应都是"PG 怎么了？"

集成到巡检

两个故障都可以通过脚本自动化检测，已集成到木木 PG 健康巡检 v2.1：

# THP 检测
cat /sys/kernel/mm/transparent_hugepage/enabled  # [never] = 安全
cat /sys/kernel/mm/transparent_hugepage/defrag    # [never] = 安全

# fsyncgate 检测
psql -c "SHOW data_sync_retry;"  # off = 安全

# 检测结果自动入报告，FATAL 级别标红

老木白话

THP 是 Linux 内核在"好心办坏事"——想帮你减少 TLB miss，结果把 PG 进程挂了几百毫秒。关了就行，反正 PG 自己管 shared_buffers，根本不需要大页。

fsyncgate 是 Linux 内核在"报喜不报忧"——磁盘写入失败了内核不说，fsync 还返回成功。PG 只能自救：设 data_sync_retry = off，告诉内核"搞不定就让我崩，我自己从 WAL 恢复"。

这两个问题的本质是一样的：不要默认 OS 层是完美的。数据库运维不只是看 PG 参数，OS 层的坑踩一个就是一个事故。
