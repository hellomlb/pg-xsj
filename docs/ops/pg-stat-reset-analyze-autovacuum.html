pg_stat_reset() 后执行 ANALYZE 对 AUTOANALYZE 和 AUTOVACUUM 的影响

✅ 验证通过 2026-05-29 日常运维 pg_stat_reset ANALYZE AUTOVACUUM 源码分析 pg_stat_all_tables

问题引出

在 PostgreSQL 中，偶尔会看到以下"反直觉"的现象：

-- pg_stat_all_tables 中
n_live_tup = 10000
n_tup_ins  = 0
-- 活元组 > 累计插入？！

n_tup_ins 既然是"累计插入数"，怎么可能小于 n_live_tup？要回答这个问题，需要深入追踪 pg_stat_reset() 和 ANALYZE 的内部行为。

一、pg_stat_reset() 做了什么

从 pg_stat_reset() 开始追踪源码调用链（PG 15.2）：

pg_stat_reset()
  → pgstat_reset_counters()
    → pgstat_reset_matching_entries()
      → shared_stat_reset_contents()

最终在 pgstat_shmem.c:918-919 执行：

memset(pgstat_get_entry_data(kind, header), 0,
       pgstat_get_entry_len(kind));

所有累计计数器被清零，关键字段包括：

字段含义重置后
tuples_inserted累计插入数（n_tup_ins）0
tuples_updated累计更新数（n_tup_upd）0
tuples_deleted累计删除数（n_tup_del）0
n_live_tuples当前活元组数（n_live_tup）0
n_dead_tuples当前死元组数（n_dead_tup）0
inserts_since_vacuumVACUUM 后的插入数0
changes_since_analyzeANALYZE 后的变更数0

二、ANALYZE 执行后如何更新统计

从 pgstat_relation.c:269-343 的 pgstat_report_analyze()：

void pgstat_report_analyze(Relation rel,
                          PgStat_Counter livetuples, PgStat_Counter deadtuples,
                          bool resetcounter)
{
    // 事务内修正（避免重复计数）
    if (pgstat_should_count_relation(rel) && ...)
    {
        for (trans = rel->pgstat_info->trans; trans; trans = trans->upper)
        {
            livetuples -= trans->tuples_inserted - trans->tuples_deleted;
            deadtuples -= trans->tuples_updated + trans->tuples_deleted;
        }
    }

    // 直接设置活/死元组数量（通过实际扫描表得到）
    tabentry->n_live_tuples = livetuples;
    tabentry->n_dead_tuples = deadtuples;

    // 可选重置变更计数器（默认 true）
    if (resetcounter)
        tabentry->changes_since_analyze = 0;
}

关键发现：

  • ANALYZE 会直接扫描表获取精确的 n_live_tuples
  • 但不会重置 tuples_inserted（累计插入计数器）
  • 这就是 n_live_tup > n_tup_ins 的根因

三、解释 n_live_tup > n_tup_ins

执行顺序：

pg_stat_reset()  →  所有计数器归零，n_tup_ins=0, n_live_tup=0
INSERT 43 行      →  n_tup_ins=43, n_live_tup≈43
pg_stat_reset()  →  n_tup_ins=0, n_live_tup=0 (再次归零)
INSERT 366 行     →  n_tup_ins=366, n_live_tup≈409 (43+366)
ANALYZE 触发      →  n_live_tup=409 (扫描表得到)，但 n_tup_ins 不变！

所以现象是 n_live_tup=409 > n_tup_ins=366。

实际 PG 16.11 验证结果：

字段值说明
n_tup_ins366只记录了 reset 后的插入
n_live_tup409ANALYZE 扫描得到的实际值
n_ins_since_vacuum366VACUUM 后的插入数
last_autoanalyze2026-05-29 09:56reset 后触发了 autoanalyze

结论：完全正常，不是 bug。 n_tup_ins 是累计计数器，n_live_tup 是由 ANALYZE/VACUUM 直接测量的快照值。

四、AUTOVACUUM 判断逻辑

从 autovacuum.c:3080-3112 的触发判断：

vactuples = tabentry->n_dead_tuples;          // 死元组数 → VACUUM 判断
instuples = tabentry->inserts_since_vacuum;   // 插入数 → 冻结 VACUUM 判断
anltuples = tabentry->changes_since_analyze;   // 变更数 → ANALYZE 判断

vacthresh = vac_base_thresh + vac_scale_factor * reltuples;
vacinsthresh = vac_ins_base_thresh + vac_ins_scale_factor * reltuples;
anlthresh = anl_base_thresh + anl_scale_factor * reltuples;

触发公式

操作触发条件默认阈值
VACUUM（死元组）n_dead_tuples > 50 + 0.2 × reltuples需要累积足够死元组
VACUUM（插入冻结）inserts_since_vacuum > 1000 + 0.2 × reltuples需要累积足够插入
ANALYZEchanges_since_analyze > 50 + 0.1 × reltuples需要累积足够变更

五、pg_stat_reset() + ANALYZE 的影响评估

功能影响说明
ANALYZE 本身✅ 无影响正常收集统计信息，正确更新 pg_statistic
查询优化器✅ 无影响pg_statistic 表不受 pg_stat_all_tables 影响
基于死元组的 VACUUM✅ 正常n_dead_tuples 被 ANALYZE 正确设置
基于插入的 VACUUM⚠️ 可能延迟inserts_since_vacuum=0，需重新累积才能触发冻结 VACUUM
基于变更的 ANALYZE✅ 正常changes_since_analyze=0，从 0 开始累积

VACUUM 执行后的统计更新（pgstat_report_vacuum）

tabentry->n_live_tuples = livetuples;        // 覆盖为实际值
tabentry->n_dead_tuples = deadtuples;         // 覆盖为实际值
tabentry->inserts_since_vacuum = 0;           // 重置插入计数器

VACUUM 和 ANALYZE 的核心差异：

操作重置 inserts_since_vacuum重置 changes_since_analyze
VACUUM✅ 是❌ 否
ANALYZE❌ 否✅ 是（默认）

六、pg_stat_reset() 后的最佳恢复策略

✅ 推荐：VACUUM ANALYZE

VACUUM ANALYZE my_table;

执行后所有计数器恢复到准确状态：

字段值状态
n_live_tuples实际活元组✅ 准确
n_dead_tuples实际死元组✅ 准确
inserts_since_vacuum0✅ 已重置
changes_since_analyze0✅ 已重置

⚠️ 不推荐：仅 ANALYZE

inserts_since_vacuum 不会被重置，基于插入的冻结 VACUUM 可能延迟。

七、v16 vs v15 的关键差异

以上分析基于 PG 15.2 源码。PG 16+ 的变更：

  1. pg_stat_io 新增：I/O 统计独立到 pg_stat_io 视图
  2. pg_stat_reset_shared('io')：可单独重置 I/O 统计
  3. last_seq_scan / last_idx_scan：新增最后扫描时间戳

但 pg_stat_reset() 对表级统计的清零行为不变，本文结论在 PG 16+ 同样适用。

九、总结

  1. pg_stat_reset() 清零所有累计计数器，包括 n_tup_ins、n_live_tup、inserts_since_vacuum 等
  2. ANALYZE 扫描表获取精确值，设置 n_live_tup/n_dead_tuples，但不恢复累计计数器
  3. n_live_tup > n_tup_ins 是正常现象，说明统计刚被重置或在重置后执行了 ANALYZE
  4. 推荐重置后执行 VACUUM ANALYZE，确保所有计数器恢复到准确状态
  5. 避免频繁调用 pg_stat_reset()，除非有明确的监控重置需求
