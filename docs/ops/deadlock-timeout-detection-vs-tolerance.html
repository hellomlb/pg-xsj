deadlock_timeout：名字骗了你 —— 它不是"容忍死锁的时间"

✅ SQL 验证通过 2026-06-10 日常运维 deadlock_timeout 死锁检测 锁竞争 GUC log_lock_waits 等待图

一句话总结

deadlock_timeout 的名字是一个"小小的谎言"——它不是死锁被容忍的时长，而是锁等待多久之后才触发死锁检测。死锁一旦发现，立刻打破，零容忍。

更令人意外的是：并非所有潜在死锁都以 ERROR: deadlock detected 告终——有些被静默地通过队列重排化解，你永远不会知道。

命名误区：检测频率 ≠ 容忍时长

常见误解实际行为
deadlock_timeout = 1s = 容忍死锁 1 秒等待锁 1 秒后才去检查有没有死锁
发现死锁后等待 timeout 才打破发现死锁立刻打破，零延迟
值越大 = 越能"忍受"死锁值越大 = 检测越迟 = 真实死锁暴露越慢

Christophe Pettus 在原文中直截了当地说："The name is a small lie."

为什么延迟检测？

死锁检测需要构建等待图（wait-for graph）并搜索其中的环——这个操作不便宜。而绝大多数锁等待并非死锁，只是短暂等待另一个事务提交。

如果每次锁等待都运行完整检测 = 大量 CPU 周期花在反复证明"没有死循环"上。

PostgreSQL 选择了乐观策略：

  1. 进程开始等待冲突锁 → 设置 deadlock_timeout 定时器 → 休眠
  2. 常规情况：定时器触发前锁释放 → 进程继续，检测从未运行
  3. 触发检测：定时器触发时仍阻塞 → 运行 CheckDeadLock() 构建等待图

硬边 vs 软边：死锁检测的精妙之处

等待图中每条边表示"此进程正在等待彼进程"，但边有两种根本不同的性质：

边类型来源性质能否改变
硬边（Hard Edge）等待某人已经持有的锁事实：A 要 B 手里的锁，B 不放 A 就走不了❌ 不可改变
软边（Soft Edge）等待队列的排队顺序约定：A 排在 B 后面，仅因为 B 先到✅ 可以重排

硬边示例

-- 会话 A：持有 accounts 表的行锁
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 会话 B：持有 orders 表的行锁
BEGIN;
UPDATE orders SET status = 'paid' WHERE id = 1;

-- 会话 A：现在要 orders 的锁（硬边 A→B）
UPDATE orders SET status = 'processing' WHERE id = 1;  -- 阻塞

-- 会话 B：现在要 accounts 的锁（硬边 B→A）
UPDATE accounts SET balance = balance + 100 WHERE id = 1;  -- 阻塞
-- 等待图：A→B（硬边）+ B→A（硬边）= 纯硬边环 = 真正死锁

纯硬边环 = 真正的死锁，检测器选择触发检测的那个后端进程作为受害者，中止其事务并抛出：

ERROR:  deadlock detected
SQLSTATE: 40P01

软边示例与静默化解

-- 会话 A：等待锁 X，排在队列第 3 位
-- 会话 B：等待锁 X，排在队列第 1 位
-- A 排在 B 后面 → 软边 A→B

-- 如果此时 B 也在等待某个 A 关联的资源 → 形成环
-- 但环中包含软边 → 可以通过重排队列化解
-- 无人牺牲，无人报错

关键洞察：软边只是当前队列排序的产物。改变唤醒顺序，有时可以在不中止任何事务的情况下化解死锁。

💡 你可能从未注意过：PostgreSQL 已经静默地通过队列重排化解了不少潜在死锁。这些"差点死锁"从不出现在日志里，也不会产生 ERROR: deadlock detected——它们被无感地消化了。

默认值 1s：为什么不调也不该调

-- 查看当前值
SHOW deadlock_timeout;

 deadlock_timeout
------------------
 1s
(1 row)
场景建议理由
日常运行保持默认 1s文档指出这是实践中的最小合理值
高负载短事务可适当调高用略延迟真实死锁报告换取更少无效检测开销
想调低以"更快发现死锁"❌ 不建议大多数锁等待 < 1s 自行解决，调低 = 大量无意义的检测运行

调优原则：保持该值高于你的典型事务执行时间，以便普通锁等待能在检测白白运行之前自行解决。

实战技巧：用 deadlock_timeout + log_lock_waits 诊断锁竞争

这是 deadlock_timeout 最有实战价值的用法——不是调优参数，而是诊断绊线。

deadlock_timeout 同时控制 log_lock_waits 的行为：当 log_lock_waits = on 时，任何超过 deadlock_timeout 时间的锁等待都会被记录到日志。

标准诊断流程

-- Step 1：开启锁等待日志
SET log_lock_waits = on;

-- Step 2：临时降低检测阈值（绊线从 1s 降到 100ms）
SET deadlock_timeout = '100ms';

-- Step 3：正常运行业务，观察日志
-- PostgreSQL 日志中出现类似：
--   LOG: process 12345 still waiting for ShareLock on transaction 67890 after 100.123 ms

-- Step 4：排查完毕后恢复
SET deadlock_timeout = '1s';
-- log_lock_waits 可保持 on（生产环境建议长期开启）

为什么 100ms 而不是 1s？

那些真正损害吞吐量的锁等待，通常是那些"在即将被判定为死锁的半秒前就已经自行解决"的等待——它们不会触发 ERROR: deadlock detected，却在 200-800ms 的时间窗口里持续阻塞其他事务。

把 deadlock_timeout 降到 100ms，这些"隐性的锁竞争"就被完整暴露在日志中。

用 ALTER SYSTEM 持久化诊断配置

-- 持久化 log_lock_waits（推荐生产环境长期开启）
ALTER SYSTEM SET log_lock_waits = on;
SELECT pg_reload_conf();

-- 确认生效
SHOW log_lock_waits;
-- 应为 on

-- 注意：deadlock_timeout 是 superuser 参数
-- 临时诊断时用 SET，排查完毕恢复 1s
-- 不要用 ALTER SYSTEM 把 100ms 持久化！

⚠️ 重要提醒：deadlock_timeout 是 superuser 级别参数，普通用户无法修改。SET deadlock_timeout 需要超级用户权限。诊断时用 SET（会话级），排查完自然失效，不要用 ALTER SYSTEM 把 100ms 持久化到生产环境。

完整检测流程图

进程 P 请求锁 L
    │
    ├─ L 无冲突 → 直接获取 ✅
    │
    └─ L 有冲突（被其他事务持有）
         │
         ├─ 设置 deadlock_timeout 定时器
         │
         ├─ 定时器触发前锁释放
         │   └─ 获取锁 ✅（检测从未运行）
         │
         └─ 定时器触发仍阻塞
              │
              ├─ 调用 CheckDeadLock()
              │   └─ 构建等待图，搜索环
              │
              ├─ 无环 → 继续休眠等待
              │
              ├─ 环含软边
              │   ├─ 重排队列化解 → 无人牺牲 ✅（静默）
              │   └─ 无法化解 → 降级为硬边处理
              │
              └─ 纯硬边环
                  └─ 选择受害者 → ERROR: deadlock detected 🔴

与 PG19 pg_stat_lock 视图的联动

PG19 新增 pg_stat_lock 系统视图，含 waits 计数和 wait_time，可直接 SQL 查询锁详情。这为 deadlock_timeout 的诊断提供了更精确的量化支撑：

工具用途PG 版本
deadlock_timeout + log_lock_waits日志级诊断绊线全部
pg_stat_lock（PG19）SQL 级锁等待量化统计19+
pg_stat_activity当前等待锁的会话全部
