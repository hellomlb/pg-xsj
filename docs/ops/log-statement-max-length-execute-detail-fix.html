log_statement_max_length 的漏网之鱼：EXECUTE 的 DETAIL 绕过长度限制

✅ SQL 验证通过 2026-07-10 日常运维 log_statement_max_length log_parameter_max_length 日志截断 EXECUTE prepared statement GUC 上游补丁

⚠️ 验证范围说明：log_statement_max_length 是我们在验证时才发现的——它是 2026-07-10 才合入 master 的全新 GUC。我们的 16.11 / 16.14 / 19beta1 实例都不含这个参数，因此该 bug 无法在现有版本复现。下文所有 SQL 均为真实环境跑通的结果：参数存在性在三版本核查、截断机制在 16.11 + 16.14 双版本实测。

一句话总结

log_statement_max_length 用来限制日志里 SQL 文本的最大长度，但 EXECUTE 在 DETAIL 消息里打印的 prepared statement 原文一直没遵守这个限制——直到 2026-07-10 上游合入补丁修复。由于该参数本身很新，我们手头的版本都不受影响，也无法复现。

背景：这个参数管什么

PostgreSQL 用一组 GUC 控制"日志里 SQL 能记多长"：

参数引入版本管什么
log_statement_max_lengthPG20（2026-07-10 合入 master）日志里语句文本的最大字节数
log_parameter_max_lengthPG15日志里绑定参数值的最大字节数

设成 10，超长的语句在日志里只保留前 10 字节。本文的 bug 出在"语句文本"这一支。

问题：EXECUTE 的 DETAIL 漏了

当你 EXECUTE 一个 prepared statement 时，PostgreSQL 会额外在 DETAIL 消息里把 prepared statement 的原文再打印一遍。Fujii Masao 在 review 时发现：这段 DETAIL 文本根本不走 log_statement_max_length 的截断逻辑。于是哪怕你把限制设得很小，巨大查询的原文照样完整漏进日志。

-- 上游复现（master 修复前）
=# SET log_statement_max_length TO 10;
=# EXECUTE test(1);
LOG:  statement: EXECUTE te
DETAIL:  prepare: PREPARE test AS SELECT * FROM pgbench_accounts WHERE aid = $1;

主语句被截成 EXECUTE te，DETAIL 里的原文却完整无缺——这就是漏网之鱼。

上游修复（提一下）

2026-07-10，Jim Jones 提交、Fujii Masao 合入 master 的两个 commit：

  • 999a841：让 EXECUTE 的 DETAIL 也走统一截断逻辑，并在被截断处追加 ... 标记（与 log_parameter_max_length 风格一致）；0 长度时只输出省略号
  • 5594f20：简化了调用方的冗余判断

（补丁源码细节此处不展开，想深挖可直接翻上面两个 commit。）

我们在真实环境验证

验证一：三个版本都没有这个参数

分别在 PG 16.11、16.14、19beta1 上查 pg_settings：

SELECT name
FROM pg_settings
WHERE name = 'log_statement_max_length';

三版本均返回空集。结论：

版本log_statement_max_lengthlog_parameter_max_length
PG 16.11不存在存在
PG 16.14不存在存在
PG 19beta1不存在存在

这个参数是当天才随补丁进 master 的新 GUC，19beta1 在它之前就已冻结，16.x 更没有。所以该 bug 在我们维护的所有版本上既不存在、也无法复现（因为特性本身尚不存在）。

验证二：截断 + ... 标记机制本身是真的（双版本）

虽然 log_statement_max_length 没有，但它的近亲 log_parameter_max_length（PG15+）在我们三个版本都在。我们在 16.11 和 16.14 上用 extended protocol 实测了它的截断行为：

SET log_statement = 'all';
SET log_parameter_max_length = 5;
SELECT $1 AS v \bind 'abcdefghijk' \g

两版本日志均出现：

LOG:  execute <unnamed>: SELECT $1 AS v
DETAIL:  parameters: $1 = 'abcde...'

abcdefghijk（11 字节）被截成 abcde（5 字节）并追加 ...。这证明 PostgreSQL 早已采用"截断 + ... 标记"的日志风格，上游这次修复只是把语句文本也对齐到同一套逻辑。

对运维的启发

要点说明
日志长度治理要等 PG20若依赖 log_statement_max_length 控制日志体积，需等包含该修复的版本（预计 PG20）发布
现在可用 log_parameter_max_length它早已能截断绑定参数值，对防止手机号/token 等敏感数据泄露到日志尤其有用，建议生产设合理值
排查日志别只盯主语句EXECUTE 的 DETAIL 过去可能携带超长原文；PG20 修复后 prepare: ... 也会被统一截断

小结

  • 问题：EXECUTE 的 DETAIL 绕过 log_statement_max_length 不截断
  • 修复：2026-07-10 合入 master（999a841 + 5594f20）
  • 现状：16.11 / 16.14 / 19beta1 均无此参数，无法复现；同类机制 log_parameter_max_length 双版本实测确凿
  • 待办：PG20 发布后补一篇真实验证
