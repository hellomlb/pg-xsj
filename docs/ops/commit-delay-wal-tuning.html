commit_delay / commit_siblings：WAL 组提交调优——从源码到决策树

✅ SQL 验证通过 2026-05-28 运维实战

概述

PostgreSQL 的 commit_delay 和 commit_siblings 是一对让人又爱又恨的参数。爱它，是因为调对了确实能提升高并发写入吞吐；恨它，是因为九成人调了等于白调——不先测 fsync 就是盲调，最终都会回到默认值。

本文从源码出发，说清楚这两个参数到底在干什么、什么时候该调、怎么调才有效。

先回顾：PG 的组提交（Group Commit）

在讲 commit_delay 之前，需要理解 PG 已经有的组提交机制。即使你不设这两个参数，PG 也会自动把多个并发事务的 WAL 合并成一次 fsync：

事务 A 刷 WAL 时拿到 WALWriteLock
  → 事务 B、C 排队等锁
  → A 发现队列里有人等着
  → A fsync 时一次性把所有排队的 WAL 记录都刷下去
  → B、C 醒来发现自己的 WAL 已经刷好了，直接走人

这就是组提交的核心：一个事务的 fsync 顺带帮所有人刷了。不需要任何额外参数。

那 commit_delay 做什么？它在组提交之上加了一层"再等等"。

源码级机制：它在哪、干什么

关键代码在 src/backend/access/transam/xlog.c 的 XLogFlush() 函数中。流程如下：

XLogFlush()
  → 获取 WALWriteLock
  → 如果三个条件同时满足，则 sleep(commit_delay) 微秒：
      1. commit_delay > 0
      2. fsync = on
      3. 当前活跃事务数 >= commit_siblings
  → issue_xlog_fsync()    ← 真正刷盘
  → 释放 WALWriteLock

三个条件缺一不可：

条件 含义 不满足时
commit_delay > 0 你主动开启了延迟 直接跳过，不 sleep
fsync = on WAL 同步写盘 fsync=off 本来就不刷，sleep 无意义
活跃事务 ≥ commit_siblings 有足够多的"同伴"值得等 等半天没人来，白等

关键洞察：sleep 发生在拿锁之后、fsync 之前。这意味着：

  • 你是 WAL 写入的"领头羊"（拿到了锁）
  • 你 delay 若干微秒，希望更多事务的 WAL 记录在这期间排队
  • 然后你一次性 fsync，帮所有人刷下去

    • 什么时候该调

    很多人上来就调，这是错的。先回答两个问题：

    问题一：WAL 同步是你的瓶颈吗？

    -- PG 14+ 查看 WAL 同步统计
SELECT wal_sync_time, wal_sync,
       wal_sync_time / NULLIF(wal_sync, 0) AS avg_sync_us
FROM pg_stat_wal;

    如果 avg_sync_us 很小（比如 SSD 上几十微秒），那 commit_delay 的收益微乎其微——你 delay 的时间可能比 fsync 本身还长，得不偿失。

    问题二：有真正的并发提交吗？

    单个连接疯狂写入不需要 commit_delay。需要的是多个并发连接同时提交的场景：连接池、Web 应用、微服务批量写入。

    如果两个答案都是"是"，继续往下走。

    调优决策树

    pg_test_fsync → 拿到 fsync 平均耗时
    ↓
查 pg_stat_wal.wal_sync_time → WAL 同步是瓶颈？
    ↓ 否 → 放弃调优（调了也是安慰剂）
    ↓ 是
确认有真正并发提交（看 pg_stat_activity）
    ↓
commit_delay 初始值 = fsync 时间 / 2
commit_siblings：NVMe → 提高门槛（如 10~20）
               HDD  → 降低门槛（如 3~5）
    ↓
迭代测试（pgbench -c N -j N）
    ↓
设太高 → 延迟放大，适得其反

    第一步：测 fsync 速度

    # 在数据库服务器上运行，不要在远程执行
pg_test_fsync

    输出示例（NVMe SSD）：

    5 seconds per test
Compare file sync methods using one 8kB write:
        open_datasync                    22448.940 ops/sec      45 usecs/op
        fdatasync                        22523.490 ops/sec      44 usecs/op
        fsync                            16764.431 ops/sec      60 usecs/op

    这里的 usecs/op 就是单次 fsync 的耗时。假设你的 fsync 方法是 60 微秒，那么 commit_delay 的初始值可以设为 30（微秒）。

    第二步：确认当前配置

    SELECT name, setting, unit, context, short_desc
FROM pg_settings
WHERE name IN ('commit_delay', 'commit_siblings', 'fsync', 'wal_sync_method');

    默认值：commit_delay = 0（关闭），commit_siblings = 5。

    第三步：设参 + 压测

    -- 假设 fsync 耗时 60us，初始设 30us
ALTER SYSTEM SET commit_delay = 30;

-- NVMe 环境，提高 siblings 门槛，避免单连接也被 delay
ALTER SYSTEM SET commit_siblings = 10;

-- 让参数生效（不需要重启）
SELECT pg_reload_conf();

    然后用 pgbench 压测对比：

    # 基准测试（默认值）
pgbench -c 20 -j 4 -T 60 -M prepared bench_db

# 调参后测试
pgbench -c 20 -j 4 -T 60 -M prepared bench_db

    对比指标：TPS、平均延迟、P99 延迟。

    第四步：迭代

    观察 行动
    TPS 提升 + 延迟可接受 微调 commit_delay 继续测试
    TPS 下降 commit_delay 设大了，减半再试
    P99 延迟暴涨 同样说明设大了
    毫无变化 你的瓶颈不在 WAL fsync，放弃调优

    常见误区

    误区一：不看 fsync 就调

    没跑过 pg_test_fsync 就调 commit_delay，相当于不知道路况就开始踩油门。你的磁盘 fsync 只要 20 微秒，你设了个 10000 微秒的 delay，每次提交多等 500 倍时间——纯属自残。

    误区二：在云环境照搬物理机参数

    云盘（EBS、云 SSD）的 fsync 延迟波动远大于本地 NVMe。pg_test_fsync 的结果在不同时间段可能差几倍。在云上，commit_delay 通常不建议调，因为 fsync 延迟的不确定性会让你永远找不到合适的值。

    误区三：commit_siblings 设太小

    commit_siblings = 1 意味着只要有任何其他活跃事务就 delay。但单个活跃事务不代表它会并发提交——它可能在跑一个大查询。设太小会让不相关的查询也被 delay。

    误区四：把 commit_delay 当万能药

    如果你的写入瓶颈是 CPU、锁竞争、或 SQL 执行本身慢，调 commit_delay 不会有任何效果。先用 pg_stat_wal 确认瓶颈在 WAL 同步：

    -- 如果 wal_sync_time 占比很小，瓶颈不在 WAL
SELECT
  wal_write_time AS "WAL 写入耗时(ms)",
  wal_sync_time  AS "WAL 同步耗时(ms)",
  stats_reset
FROM pg_stat_wal;

    实战 checklist

    运维中需要调整 commit_delay 的典型场景：

    场景 建议
    裸金属 + NVMe + 高并发短事务 值得测试，初始值 20~50us
    裸金属 + HDD + 高并发短事务 值得测试，初始值 500~2000us
    云环境（EBS/云 SSD） 不建议调，fsync 延迟波动大
    fsync=off commit_delay 不生效，不用调
    单连接写入 不生效，活跃事务数不够
    synchronous_commit=off commit_delay 仍然生效（fsync 还在）

    小结

    commit_delay 的本质是在 PG 已有组提交之上加一层"再等等"——leader 拿到 WALWriteLock 后、fsync 前 sleep 若干微秒，希望更多事务搭顺风车。调它的前提只有两个：WAL fsync 真的是瓶颈 + 真的有并发提交。不满足任何一个，调了就是安慰剂。

    记住金句：没跑 pg_test_fsync 就调 commit_delay = 盲调，最终都会回到默认值。
