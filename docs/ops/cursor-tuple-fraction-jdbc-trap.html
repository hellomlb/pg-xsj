cursor_tuple_fraction JDBC 陷阱：流式读取 ≠ 服务端游标

✅ 分析验证通过 2026-06-04 日常运维 cursor_tuple_fraction JDBC 游标 执行计划 GUC 协议层

一句话总结

你调了 cursor_tuple_fraction = 1.0 以为优化了全量读取，结果对 JDBC 完全无效——因为 JDBC 驱动默认根本不使用 PostgreSQL 服务端游标。

这个陷阱的本质是：把不同抽象层的行为混为一谈。JDBC 的"批量取" ≠ PG 的"游标"，是两件事。

cursor_tuple_fraction 本身做什么

项目 内容
默认值 0.1（10%）
取值范围 0.0 ~ 1.0
引入版本 PG 8.4
作用范围 仅对 DECLARE CURSOR 查询生效

当查询通过服务端游标执行时，规划器用它来估计"用户实际上会消费多少行"：

cursor_tuple_fraction = 0.1  → "用户只看前 10%"  → 偏向索引扫描（首行快）
cursor_tuple_fraction = 1.0  → "用户会读完所有行" → 偏向顺序扫描（总量快）

对普通 SQL 查询：完全无效。 只有 DECLARE CURSOR ... FETCH ... 这种模式才会用到。

JDBC 的三种读取模式

JDBC 驱动的分块读取有两套完全不同的机制，而大多数人只看到了 setFetchSize() 的表面含义。

模式 触发方式 服务端看到什么 cursor_tuple_fraction 有效？
简单协议
（默认）		 setFetchSize(0) 或不设置 一条普通 SELECT，结果一次性发送 ❌ 无效
扩展协议
（批量模式）		 setFetchSize(N)
（N > 0）		 一个匿名 portal，分批 FETCH N 行 ❌ 仍无效
命名游标 setCursorName() + setFetchSize() DECLARE CURSOR ... FETCH ... ✅ 仅这种有效

协议层：为什么扩展协议不受 cursor_tuple_fraction 影响

关键点在于 JDBC 扩展协议走的是 匿名 portal，不是 DECLARE CURSOR（命名游标）。

-- JDBC 扩展协议批量模式（setFetchSize(N)，N > 0）：
-- 服务端视角：
Parse → Bind → Execute(limit=N) → ... → Execute(limit=N)
-- 服务端：这不是"游标"，这是分批次执行的普通查询
-- cursor_tuple_fraction 不会被规划器参考

-- PG 原生游标模式（必须显式 DECLARE CURSOR）：
DECLARE c CURSOR FOR SELECT ...   ← 游标声明
FETCH N FROM c                    ← 服务端明确知道这是游标
-- 服务端：这是游标，用 cursor_tuple_fraction 调优执行计划

规划器判断是否走游标优化路径，只看一句话：这条查询是不是通过 DECLARE CURSOR 进来的？ 不是 → 忽略 cursor_tuple_fraction。

三个常见误区

误区 真相
"设了 fetchSize 就是在用游标" fetchSize > 0 用的是匿名 portal，不是命名游标；规划器看不到游标声明
"调 cursor_tuple_fraction 能优化 JDBC 批量读取" 除非显式用了 setCursorName()，否则完全无效
"游标默认 0.1 所以我的全量导出很慢" 如果代码没调 setCursorName()，这压根不是游标慢，是没有设 fetchSize 或 LIMIT 导致的其他问题

cursor_tuple_fraction 真正生效的条件

只有当以下所有条件同时满足时，调 cursor_tuple_fraction 才有意义：

  1. SQL 层确实使用了 DECLARE CURSOR ... FETCH ...（命名游标）
  2. JDBC 层显式调用了 setCursorName()
  3. 目标是对"只取前 N 行"和"取完全部"两种场景选择不同的执行计划

如果条件 1 或 2 不满足，cursor_tuple_fraction 对这条查询的执行计划零影响——调了等于白调。

老木白话

流式读取是 JDBC 的批获取，游标是 PG 的服务端机制，两件事。

你告诉 JDBC "每次取 100 行"（fetchSize=100），JDBC 用扩展协议分批拿，PG 规划器根本不知道你在"游标模式"下跑查询，自然不会参考 cursor_tuple_fraction。

只有你显式告诉 JDBC "用命名游标"（setCursorName），它才会发 DECLARE CURSOR 给服务端，规划器才会看 cursor_tuple_fraction。
