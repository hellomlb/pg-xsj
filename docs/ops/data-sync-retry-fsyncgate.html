data_sync_retry：fsyncgate 的遗产 —— PostgreSQL 持久化最深的教训

2026-06-09 日常运维 fsyncgate data_sync_retry 持久化 Linux内核 检查点 DirectIO

一句话总结

data_sync_retry 是 PostgreSQL 持久化历史上最深刻的一次教训——fsyncgate 事件的"疤痕组织"。它的默认值 off（fsync 失败 → PANIC） 不是保守，而是安全的；Linux 上开启 on 的唯一诚实描述是"请静默地丢失我的数据，而不是大张旗鼓地丢失"。

本文不是参数调优指南，而是对 fsyncgate 事件的完整技术拆解——从内核行为到 PG 对策到残留风险到终极方案。

fsyncgate 事件始末

PostgreSQL 原以为的契约

PostgreSQL 写入磁盘分两步：

  • write()：将脏页复制到内核的页面缓存（page cache），速度快但不持久
  • fsync()（在检查点时调用）：强制将文件的所有未刷脏页写入持久化存储

数十年来的共识契约很简单：

  • fsync() 返回成功 → 数据已落盘，WAL 可以回收
  • fsync() 返回失败 → 写入未完成，脏页仍在缓存中，可以重试

PostgreSQL 的检查点逻辑完全建立在第二条之上——

内核实际做了什么

当 Linux 内核尝试将脏页写回设备但写入失败时（坏扇区、精简配置卷空间耗尽、SAN 短暂消失等），内核面临抉择：将永远无法回收的脏页一直保留？这构成无界内存泄漏，内核无法允许。

因此 Linux 采取了实用主义做法：

  • 将页面标记为 clean（干净）
  • 记录发生了一个错误
  • 在下一次对该文件的 fsync() 调用时报告该错误——仅此一次
  • 之后该错误被遗忘，页面可像其他干净页一样被回收重用

更糟糕的是：当时的内核中，错误被传递给第一个调用 fsync() 的文件描述符，且仅该描述符收到。PostgreSQL 的 checkpointer 进程通常重新打开文件而非一直持有描述符——因此错误可能被其他后端进程消费并丢弃，然后 checkpointer 的后续 fsync() 会返回干净的成功。

完整的灾难场景

  1. 检查点运行，写回错误发生
  2. 错误被报告给错误的文件描述符，然后被遗忘
  3. 脏页被标记为 clean，被回收重用
  4. checkpointer 调用 fsync()，获得成功
  5. 检查点被认为已完成，WAL 段被回收
  6. 数据从未写入磁盘，页面缓存中的副本已消失，WAL 中的副本也已消失

正如 Andrew Gierth 在原讨论中所说：数据此时"从已知宇宙中消失"——任何地方都没有它的副本，重试无法恢复已不存在的东西。

data_sync_retry 参数深度解析

值行为安全性
off（默认）fsync() 失败 → PANIC✅ 安全——WAL 重放可恢复
onfsync() 失败 → 重试并继续🔴 Linux 上 = 静默丢数据

为什么 PANIC 是安全的

  1. PANIC → 服务器立即崩溃
  2. 崩溃恢复启动，从最后一个已完成的检查点开始重放 WAL
  3. 因为失败的检查点从未完成，WAL 日志未被回收
  4. WAL 仍包含那些丢失写入的描述
  5. 重放 WAL → 精确重建丢失的数据

反直觉但正确的真相：在此之前，PostgreSQL 对 WAL 文件一直采用这种 PANIC 策略；2018 年的变更只是将其扩展到普通数据文件。PANIC 是安全的响应。

为什么重试是灾难

  1. fsync() 失败 → 重试
  2. 脏页已被内核标记为 clean 并回收
  3. 重试的 fsync() 可能返回成功（因为页面已不存在，没什么可报告的了）
  4. PostgreSQL 认为检查点成功，回收 WAL
  5. 数据实际已丢失，但数据库静默地继续运行

官方文档极其罕见地直言：在 Linux 上开启此选项的唯一诚实描述就是"请静默地丢失我的数据，而不是大张旗鼓地丢失"。

残留风险窗口

即使 data_sync_retry = off，仍有窄窗口

文件被关闭 → 重新打开前 → inode 因内存压力被逐出 → errseq_t 错误记录随 inode 消失

PG12 的缓解

PostgreSQL 12 引入机制：保持有脏数据的文件持续打开，并将其文件描述符直接传递给 checkpointer。这避免了"关闭→重新打开"过程中的错误丢失窗口。

但此变更被判定为过于侵入性，不适合反向移植到旧版本。

Linux 4.13 errseq_t 修补的局限性

Jeff Layton 在 Linux 4.13 引入 errseq_t，为每个 inode 的 address_space 附加一个 32 位的错误+序列值，使写回错误对之后所有调用 fsync() 的描述符都可见，而不仅仅是第一个。

但有三个关键局限：

  1. 数据回不来：可靠报告的错误仍是丢失数据的错误——数据丢了就是丢了
  2. inode 逐出时错误仍然丢失：如果在重新打开文件之前，inode 因内存压力被逐出，记录的错误随 inode 一起消失
  3. 这是报告层修补，不是语义层修补：fsync() 的契约问题在更根本的层面

终极方案：Direct I/O

Dave Chinner、Ted Ts'o 以及 PostgreSQL 开发者们一致认为：真正完整的修复是对主数据路径使用 Direct I/O，完全绕过页面缓存，让数据库自己掌控写回和错误处理。

属性说明
方案对主数据路径使用 Direct I/O，绕过页面缓存
优势数据库自己掌控写回和错误处理，彻底消除 fsyncgate 类问题
进展Andres Freund 评价："a metric ton of work"（巨量工作）
状态截至 2026 年，仍未完全落地

共享文件系统的特殊风险

NFS/CIFS 等共享文件系统在 data_sync_retry 场景下有多层额外风险：

  1. fsync() 语义不可靠：客户端层面报告成功，但服务端并未真正持久化
  2. 错误报告路径更长：客户端内核 → 网络 → 服务端 → 服务端存储，任何一层都可能丢失错误
  3. PANIC 是重要信号：如果运行在 NFS/CIFS 上，检查点时 fsync PANIC 很可能意味着文件系统在"说谎"
