debug_* GUC 家族：PostgreSQL 的 14 个"内脏检查开关"

✅ PG19 Beta1 验证通过 2026-06-11 日常运维 debug_* GUC debug_io_direct Direct I/O debug_parallel_query PG19

一句话总结

PostgreSQL 有 14 个以 debug_ 开头的 GUC 参数，Christophe Pettus 在其 GUC 系列第 10 篇中梳理了其中 12 个——实测 PG19 Beta1 发现原文遗漏了 2 个。这 14 个参数绝大多数是 PG 自身的开发/QA 工具，永远不应在生产环境触碰。唯一值得持续关注的是 debug_io_direct——它披着 debug_ 前缀，却承载着 Direct I/O 的未来。

原文遗漏 2 个参数

Christophe Pettus 原文声称 12 个 debug_ 参数，但 PG19 Beta1 实测有 14 个：

遗漏参数默认值上下文说明推测遗漏原因
debug_exec_backendoffinternal（只读）显示是否以 EXEC_BACKEND 模式编译（Windows / EXEC_BACKEND 构建）极冷门，仅特定构建模式相关
debug_print_raw_parseoffuser输出原始解析树（比 debug_print_parse 更早阶段）可能是 PG19 新增参数

🔍 排查方法：永远不要仅凭文档或博客写参数清单——直接查数据库：

-- 查看你的 PG 版本有多少个 debug_ 参数
SELECT name, setting, context, vartype, short_desc
FROM pg_settings
WHERE name LIKE 'debug_%'
ORDER BY name;

PG19 Beta1 实测输出 14 行，V1 验证通过 ✅

14 个参数完整分类

第一类：只读检测（2 个）—— 能看不能动

这两个参数 context = internal，只能 SHOW，不能 SET。

参数默认值类型含义实测验证
debug_assertionson（调试构建）/ off（正式）bool是否以 --enable-cassert 编译SET 被拒绝 ✅
debug_exec_backend 🆕offbool是否以 EXEC_BACKEND 模式编译SET 被拒绝 ✅
 
-- 检查你的 PG 是什么构建类型
SHOW debug_assertions;
-- 如果是 on → 这是调试构建，不要用于生产

SHOW debug_exec_backend;
-- 如果是 on → EXEC_BACKEND 模式（通常是 Windows 或特殊构建）

💡 实用技巧：如果你接手一个 PG 实例，先 SHOW debug_assertions——如果是 on，说明这个实例是用调试构建跑的，性能会比正式构建差。这不是参数设置问题，是编译选项决定的。

第二类：解析树/计划树输出（5 个）—— 偶尔调试用

这是唯一可能对非核心开发者有用的 debug_ 参数。EXPLAIN 和 auto_explain 已在绝大多数场景取代它们，但当你需要查看字面节点结构（Var、RangeTblEntry 等）时，这些参数提供独一无二的视角。

参数默认值输出内容阶段
debug_print_raw_parse 🆕off原始解析树最早——词法/语法刚完成
debug_print_parseoff分析后解析树语义分析完成
debug_print_rewrittenoff重写后解析树规则系统改写后
debug_print_planoff最终执行计划树规划器输出
debug_pretty_printon格式化上述输出控制缩进/换行
 
-- 临时开启（会话级，用完自动失效）
SET debug_print_plan = on;
SET debug_pretty_print = on;

-- 执行一条查询（输出进服务器日志，不在客户端显示）
SELECT 1 AS test_col;

-- 用完即关
SET debug_print_plan = off;

⚠️ 注意：debug_print_* 的输出进服务器日志（log_destination），不在客户端显示。输出量可能巨大，仅在排查特定问题时按会话临时开启，绝不持久化。

第三类：酷刑测试——节点复制/序列化（3 个）+ 缓存失效（1 个）—— 绝对不用

这 4 个参数是 PG 开发者用来找代码 Bug 的"酷刑测试"。它们会显著改变系统行为，在生产环境打开等于自杀。

参数类型作用后果
debug_copy_parse_plan_treesbool强制复制-比较循环测试每次查询额外复制+比较整棵树
debug_write_read_parse_plan_treesbool序列化-反序列化往返测试每次查询额外序列化+反序列化
debug_raw_expression_coverage_testbool遍历表达式树覆盖率测试强制遍历每个表达式节点
debug_discard_cachesint(0-5)激进失效目录缓存慢几个数量级，5 最激进
 
-- debug_discard_caches 不是布尔！是整数 0-5
SHOW debug_discard_caches;
-- 应为 0

-- 设置不同级别（仅在测试环境！）
SET debug_discard_caches = 1;  -- 最轻度
SET debug_discard_caches = 5;  -- 最激进，极慢

-- 恢复
SET debug_discard_caches = 0;

PV19 实测验证：级别 0-5 全部可设 ✅

第四类：强制特定行为（2 个）—— 测试专用

参数类型枚举值作用实测验证
debug_parallel_queryenumoff / on / regress强制并行计划3 个枚举值全部切换成功 ✅
debug_logical_replication_streamingenumbuffered / immediate强制逻辑复制即时流式2 个枚举值全部切换成功 ✅

debug_parallel_query 的重命名故事

从 force_parallel_mode 重命名为 debug_parallel_query——因为用户一直误以为它能"强制规划器生成有用的并行查询"，实际上它是测试并行代码路径的。

-- PG19 中 force_parallel_mode 已彻底移除
SELECT count(*) FROM pg_settings WHERE name = 'force_parallel_mode';
-- 结果: 0 ✅

-- 正确用法：测试并行代码路径
SET debug_parallel_query = on;
SELECT * FROM some_table WHERE id = 1;
SET debug_parallel_query = off;

-- ❌ 错误想法：想加速查询
-- debug_parallel_query = on 不会让你的查询更快！
-- 想加速？调 max_parallel_workers_per_gather
-- debug_parallel_query 的 3 个模式
SET debug_parallel_query = off;      -- 关闭（默认）
SET debug_parallel_query = on;       -- 强制使用并行
SET debug_parallel_query = regress;  -- 回归测试专用

PG19 实测：3 个枚举值全部切换成功 ✅

第五类：⭐ Direct I/O（1 个）—— 披着 debug_ 前缀的未来

维度说明实测确认
参数debug_io_direct✅
作用启用 O_DIRECT 绕过 OS 页面缓存✅
合法值data、wal、wal_init（可逗号组合）✅ 5 种组合全部 ALTER SYSTEM 通过
设置时机仅服务器启动时（postmaster 上下文）✅ SET 被拒绝
当前状态文档说"会降低性能"——异步 I/O 基础设施仍在落地—
未来展望摆脱 fsync 错误陷阱的公认出路—
 
-- 合法值验证（ALTER SYSTEM + 重启生效）
ALTER SYSTEM SET debug_io_direct = 'data';               -- OK ✅
ALTER SYSTEM SET debug_io_direct = 'wal';                -- OK ✅
ALTER SYSTEM SET debug_io_direct = 'wal_init';           -- OK ✅
ALTER SYSTEM SET debug_io_direct = 'data,wal';           -- OK ✅
ALTER SYSTEM SET debug_io_direct = 'data,wal,wal_init';  -- OK ✅

-- SET 不行（需要重启）
SET debug_io_direct = 'data';
-- ERROR: parameter "debug_io_direct" cannot be changed without restarting the server

-- 恢复
ALTER SYSTEM RESET debug_io_direct;
-- 记得重启才生效！

⭐ 为什么 debug_io_direct 值得关注：它是 PostgreSQL 摆脱 fsyncgate 阴影的公认出路。当 PG 自己管理写回、自己处理 I/O 错误，就不再依赖内核页面缓存——不再被 Linux 脏页丢弃+静默重试的坑困扰。当前性能仍不如传统路径，因为异步 I/O 基础设施还在落地中，但这是未来版本的重点演进方向。

白话总结

14 个 debug_ 参数，按实用性排列：

数量参数白话
3 个能看debug_assertions、debug_exec_backend、debug_pretty_print查看构建信息，不用改
4 个偶尔调试debug_print_raw_parse、debug_print_parse、debug_print_rewritten、debug_print_plan查解析树/计划树细节，按会话开，用完关
1 个未来方向debug_io_directDirect I/O，关注但暂不生产用
6 个酷刑测试debug_copy_parse_plan_trees、debug_write_read_parse_plan_trees、debug_raw_expression_coverage_test、debug_discard_caches、debug_parallel_query、debug_logical_replication_streamingPG 开发者在 buildfarm 里跑的，不在你的生产库跑

Pavlo Golub 说得好：正式集群上别碰任何带 debug_ 前缀的东西。
