dynamic_shared_memory_type：Docker 容器的 64MB 陷阱 —— 并行查询 No space left on device

2026-06-24 日常运维 DynamicSharedMemory Docker K8s 并行查询 共享内存 容器陷阱
✅ 已验证版 · PG 16.11 + PG 16.14 双版本实测

一句话总结

Docker 容器下运行 PostgreSQL 并行查询报 No space left on device，根因不是磁盘满了，而是 Docker 默认只给 /dev/shm 分配 64MB——POSIX 动态共享内存（DSM）无法分配足够空间。修复方案：不要动 postgresql.conf，给容器加 --shm-size=1g。

故障现象

以下错误在大规模数据查询时频繁出现，且只在容器化环境（Docker / K8s）中复现：

ERROR:  could not resize shared memory segment "/PostgreSQL.NNNNN" to NNN bytes: No space left on device
CONTEXT:  parallel worker

排查时先查磁盘：

$ df -h
Filesystem      Size  Used Avail Use% Mounted on
overlay          50G   30G   20G  60% /

磁盘还有 60% 空闲，错误却报 "No space left"。这不是磁盘空间不足，是 tmpfs 空间不足。

dynamic_shared_memory_type 是什么

PostgreSQL 有两类共享内存：

  • 固定共享内存（fixed shared memory）：shared_buffers 等，启动时一次性分配，以后不能增长
  • 动态共享内存（DSM）：运行时按需分配，用完即释放——并行查询、逻辑解码等特性依赖它

dynamic_shared_memory_type 就是选择 DSM 使用哪种操作系统机制。这是个 postmaster 上下文的参数，修改需要重启。

三种实现对比

在 Linux 上（Windows 只有 windows），可选三种：

值 机制 底层 性能 容器兼容性
posix ✅ 默认 POSIX 共享内存 shm_open → /dev/shm (tmpfs) 最快 受限：受 Docker /dev/shm 大小影响
sysv System V 共享内存 shmget → 受内核参数 SHMMAX/SHMALL 限制 较快 可由 --ipc=host 解决，但换了一组限制
mmap 内存映射文件 $PGDATA/pg_dynshmem/ 目录下的文件 最慢 完全兼容，但 OS 可能将 mmap 页回写磁盘

实际验证（PG 16.11 / PG 16.14 / PG 19 Beta1 三版本一致）：

关键发现：windows 和 none 选项只在特定平台暴露。Linux 上的 enumvals 只有 {posix,sysv,mmap}。设置 none 会完全禁用并行查询。

根因链：64MB 的隐形天花板

这个问题是一套完整的根因连锁反应：

① POSIX 共享内存 → ② /dev/shm (tmpfs) → ③ Docker 默认 64MB → ④ 并行哈希连接需要更多 → ⑤ 分配失败 → "No space left on device"

逐层拆解：

层级 说明
① 默认配置 PG 的 dynamic_shared_memory_type 默认 posix，这也是绝大多数云镜像和 Docker 镜像的默认值
② POSIX shm shm_open() 创建的文件描述符指向 /dev/shm（一个 tmpfs 文件系统，数据存在内存中）
③ Docker 默认值 物理机上 /dev/shm 默认是物理内存的 50%。Docker 容器默认只给 64MB——对绝大多数现代数据库应用来说几乎为零
④ 需求估算 并行哈希连接：每个 worker 需要较大共享内存段。一个 hash join + 4 workers 轻松超 64MB
⑤ 报错 shm_open → ftallocate(tmpfs) 失败 → PG 报 "No space left on device"

验证 /dev/shm 大小：

$ docker run --rm postgres:16 bash -c 'df -h /dev/shm'
Filesystem      Size  Used Avail Use% Mounted on
tmpfs            64M     0   64M   0% /dev/shm     ← Docker 默认只有 64MB！

$ docker run --shm-size=1g --rm postgres:16 bash -c 'df -h /dev/shm'
Filesystem      Size  Used Avail Use% Mounted on
tmpfs           1.0G     0  1.0G   0% /dev/shm     ← 扩到 1GB，问题解决
📋 SQL 2：查看 DSM 使用情况（仅 PG 18+）
-- PG 18+ 有 pg_stat_dynamic_shmem 视图
SELECT * FROM pg_stat_dynamic_shmem;

-- PG 13+ 通用手段：查看 pgaudit 日志或扩展日志中是否有 DSM 相关错误
-- 或在 psql 中查看当前并行查询是否在使用 DSM
bug=# SHOW shared_memory_size_in_huge_pages;

-- 如果 DSM 有问题，上面这条也会触发

正确修复方案

方案一：Docker run 时设置（推荐）

docker run --shm-size=1g -d --name pg16 postgres:16

# 对已有容器，重新创建
docker rm pg16
docker run --shm-size=1g ... postgres:16

方案二：Docker Compose

services:
  postgres:
    image: postgres:16
    shm_size: '1g'
    # 或
    shm_size: 1073741824

方案三：Kubernetes

K8s 没有直接等价于 --shm-size 的参数。推荐两种方式：

# 方式 A：emptyDir 挂载为内存介质（替代 /dev/shm）
volumes:
  - name: dshm
    emptyDir:
      medium: Memory
      sizeLimit: 1Gi

# 方式 B：设置 Pod 的 ShareProcessNamespace 并挂载宿主机 /dev/shm
# （不推荐，因为要改宿主机）

方案四（不推荐）：改用 SysV

# 假设 Docker 容器能访问宿主机 ipc 资源
docker run --ipc=host --rm postgres:16

# 但注意：SysV 受 SHMMAX 限制（通常 64MB-256MB），换汤不换药

错误修法（千万别这样做）

错误操作 后果 为什么不对
设置 dynamic_shared_memory_type = sysv 只是换到了 SHMMAX 限制，从 64MB 的坑跳到另一个坑 治标不治本，换限制而不是移除限制
设置 dynamic_shared_memory_type = mmap 文件映射到 $PGDATA/pg_dynshmem/，OS 可能回写磁盘→性能灾难 把本应 100% 内存的操作变成了潜在 I/O 操作
max_parallel_workers_per_gather = 0 禁用所有并行查询，错误消失，但性能大幅下降 自废武功，全表扫描和海量哈希连接变慢 2-10x
设置 dynamic_shared_memory_type = none 直接禁用 DSM，并行查询完全不可用 Linux 上 enumvals 甚至没有 none，硬写可能导致集群无法启动
挂载宿主机 /dev/shm 到容器 -v /dev/shm:/dev/shm 覆盖了 tmpfs 宿主机 /dev/shm 默认是物理内存的 50%，太大 + 影响宿主机稳定性
物理机上增大 /dev/shm 比例 把 /etc/fstab 中 tmpfs 的 size 从 default 改为 size=2g 改物理机配置治标不治本，换一台机器就失效，且影响其他容器
把 shared_buffers 调到很小 幻想减少 DSM 需求，实际上 DSM 和 shared_buffers 完全独立 两套空间，互不相关。DSM 需求由 work_mem 和并行度决定

DSM 需求估算

并行查询的 DSM 峰值需求：

最大 DSM 页数 ≈ concurrent_queries × max_parallel_workers_per_gather × work_mem 等效

举个具体例子：

  • 同时运行 5 个并行哈希连接查询
  • 每个查询使用 4 个并行 worker
  • 哈希表的 work_mem 等效用量约 128MB
  • 峰值：5 × 4 × 128MB = 2.5GB

Docker 默认 64MB 面对上述场景，不到 1/40 的需求量，必然报错。

容器巡检清单

检查客户容器环境时，加入以下检查项：

# 检查项1：/dev/shm 大小
$ docker exec <container> df -h /dev/shm

# 检查项2：PG 日志中是否有 DSM 相关错误
$ docker logs <container> 2>&1 | grep -i "shared memory"

# 检查项3：Docker 配置中的 shm_size
$ docker inspect <container> | jq '.[0].HostConfig.ShmSize'
# 返回字节数，默认 67108864（64MB），应该至少 1073741824（1GB）

背后的设计哲学

为什么默认用了 POSIX 而不是其他方案？

  • 性能：POSIX shm 的 tmpfs 完全是内存操作，没有文件系统开销
  • 安全：进程退出时内核自动清理，不会残留文件
  • 简单：不需要调内核参数（不像 SysV 需要 SHMMAX）
  • 跨平台：POSIX 标准接口，所有类 Unix 系统一致

但在容器化部署中，POSIX 依赖宿主机的 tmpfs 配额——Docker 的 64MB 默认值在 2026 年的主流硬件配置下已经极度不合理。这个默认值来自 Docker 诞生初期（2013 年），从未随硬件进步更新过。
