VACUUM 三阶段与行指针生命周期：页面级别深度解析

2026-07-06 内核研究 VACUUM page-level LP_DEAD heap_page_items INDEX_CLEANUP
✅ 已验证版 · PG 19 Beta1 实测

一句话总结

VACUUM 对有索引的表分三个阶段执行。最容易被误解的是 Phase 3（堆清理）：它只回收 4 字节的行指针槽位，元组数据在 Phase 1 修剪时就已经回收了。VACUUM (INDEX_CLEANUP OFF) 跳过 Phase 2 和 Phase 3，元组空间仍然被回收，但行指针槽位不会释放。

核心认知修正：很多人以为 VACUUM 的"回收空间"发生在 Phase 3，实际上元组字节在 Phase 1 的修剪中就已经被释放了。Phase 3 处理的只是那个 4 字节的 ItemPointerData 槽位。

行指针状态机

每个页面有一个行指针数组（pd_lower 到 pd_upper 之间），每个行指针 4 字节，指向元组在页面中的位置（偏移量 + 长度 + 标志位）。

状态值含义触发
LP_UNUSED 0 空槽位，可被新 INSERT 复用 VACUUM Phase 3 / 无索引表直接修剪
LP_NORMAL 1 指向元组（可能是活元组或已删除的死元组） INSERT / UPDATE
LP_REDIRECT 2 HOT 链头部，指向链中的另一个行指针 HOT 修剪（链头部被剪掉时）
LP_DEAD 3 元组数据已回收，但槽位必须保留
因为索引可能仍引用此 TID		 VACUUM Phase 1 修剪（有索引表）

完整状态转换

-- 有索引的表：两阶段回收
LP_UNUSED(0) ──INSERT──────────────────▶ LP_NORMAL(1)
LP_NORMAL(1) ──prune (Phase 1)─────────▶ LP_DEAD(3)
LP_DEAD(3)   ──index cleanup + 2nd scan▶ LP_UNUSED(0)

-- 无索引的表：一次性回收
LP_UNUSED(0) ──INSERT──────────────────▶ LP_NORMAL(1)
LP_NORMAL(1) ──prune───────────────────▶ LP_UNUSED(0)  -- 跳过 LP_DEAD！

-- HOT 更新
LP_NORMAL(1) ──HOT prune (链头部)───────▶ LP_REDIRECT(2)
LP_NORMAL(1) ──HOT prune (链成员)───────▶ LP_UNUSED(0)
LP_REDIRECT(2)──VACUUM (目标也死时)─────▶ LP_UNUSED(0)

关键点：LP_DEAD 中间状态是为索引而存在的。如果表上没有索引，修剪直接把行指针设为 LP_UNUSED(0)，完全跳过 LP_DEAD。

实测验证：pageinspect 页面级铁证

以下实验在 PG 19 Beta1 上执行，用 pageinspect 扩展直接观测页面 0 的物理变化。测试表有一颗主键索引（隐式唯一索引）：

测试表准备

CREATE EXTENSION pageinspect;

CREATE TABLE vac_test (
    id serial PRIMARY KEY,   -- 主键 = 隐式唯一索引
    data text
) WITH (autovacuum_enabled = off, fillfactor = 100);

-- 插入 50 行（约 88 字节/行，全部在一页内）
INSERT INTO vac_test (data)
SELECT 'row_data_' || LPAD(g::text, 10, '0')
    || '_padding_text_to_make_rows_wide_enough'
FROM generate_series(1, 50) g;

阶段 0：基准（BASELINE）

SELECT lower, upper, special, pagesize,
       pagesize - upper AS tuple_area
FROM page_header(get_raw_page('vac_test', 0));

--  lower | upper | special | pagesize | tuple_area
-- ───────┼───────┼─────────┼──────────┼────────────
--    224 |  3792 |    8192 |     8192 |       4400

SELECT count(*) FILTER (WHERE lp_flags = 0) AS LP_UNUSED,
       count(*) FILTER (WHERE lp_flags = 1) AS LP_NORMAL,
       count(*) FILTER (WHERE lp_flags = 3) AS LP_DEAD
FROM heap_page_items(get_raw_page('vac_test', 0));

--  lp_unused | lp_normal | lp_dead
-- ───────────┼───────────┼─────────
--          0 |        50 |       0

pd_upper = 3792，50 个 LP_NORMAL，元组区 4400 字节。干净的一页。

阶段 1：DELETE 30 行（id % 5 IN (1,2,3)）

DELETE FROM vac_test WHERE id % 5 IN (1,2,3);

SELECT lower, upper FROM page_header(get_raw_page('vac_test', 0));
--  lower | upper
-- ───────┼───────
--    224 |  3792    ← 完全没变！

-- 行指针状态也没有任何变化：仍然是 50 个 LP_NORMAL
-- 但 t_xmax != 0 表明这些行已被删除标记

关键发现：DELETE 不改变页面布局。pd_upper 纹丝不动，元组数据仍在原处，行指针全部 LP_NORMAL。唯一的变化是元组头部的 t_xmax 被标记为删除事务 ID。页面空间回收必须等 VACUUM。

阶段 2：Phase 1 Only — VACUUM (INDEX_CLEANUP OFF)

只执行堆扫描修剪，跳过 Phase 2 索引清理和 Phase 3 堆清理：

VACUUM (INDEX_CLEANUP OFF) vac_test;

SELECT lower, upper,
       pagesize - upper AS tuple_area,
       upper - lower AS usable_gap
FROM page_header(get_raw_page('vac_test', 0));

--  lower | upper | tuple_area | usable_gap
-- ───────┼───────┼────────────┼────────────
--    224 |  6432 |       1760 |       6208   ← pd_upper 从 3792 → 6432！

SELECT count(*) FILTER (WHERE lp_flags = 0) AS LP_UNUSED,
       count(*) FILTER (WHERE lp_flags = 1) AS LP_NORMAL,
       count(*) FILTER (WHERE lp_flags = 3) AS LP_DEAD
FROM heap_page_items(get_raw_page('vac_test', 0));

--  lp_unused | lp_normal | lp_dead
-- ───────────┼───────────┼─────────
--          0 |        20 |      30   ← 30 LP_NORMAL → 30 LP_DEAD

-- 取样观察行指针细节
SELECT lp, lp_off, lp_flags FROM heap_page_items(get_raw_page('vac_test', 0))
WHERE lp <= 5;

--  lp | lp_off | lp_flags
-- ────┼────────┼──────────
--   1 |      0 |        3    ← LP_DEAD: lp_off=0（不再指向元组）
--   2 |      0 |        3
--   3 |      0 |        3
--   4 |   8104 |        1    ← LP_NORMAL: 仍指向元组
--   5 |   8016 |        1
铁证 #1：pd_upper 从 3792 → 6432，元组区从 4400B 缩小到 1760B，回收了 2640 字节元组数据（= 30 死元组 × 88 字节）。同时，30 个行指针从 LP_NORMAL 变为 LP_DEAD（lp_off=0, lp_flags=3），但槽位本身仍然占用 pd_lower 下方的 4 字节空间。

阶段 3：Full VACUUM（三阶段完整）

       pagesize - upper AS tuple_area
FROM page_header(get_raw_page('vac_test', 0));

--  lower | upper | tuple_area
-- ───────┼───────┼────────────
--    224 |  6432 |       1760   ← pd_upper 纹丝不动！

SELECT count(*) FILTER (WHERE lp_flags = 0) AS LP_UNUSED,
       count(*) FILTER (WHERE lp_flags = 1) AS LP_NORMAL,
       count(*) FILTER (WHERE lp_flags = 3) AS LP_DEAD
FROM heap_page_items(get_raw_page('vac_test', 0));

--  lp_unused | lp_normal | lp_dead
-- ───────────┼───────────┼─────────
--         30 |        20 |       0   ← 30 LP_DEAD → 30 LP_UNUSED

-- 取样确认
SELECT lp, lp_off, lp_flags FROM heap_page_items(get_raw_page('vac_test', 0))
WHERE lp <= 5;

--  lp | lp_off | lp_flags
-- ────┼────────┼──────────
--   1 |      0 |        0    ← LP_UNUSED: 槽位已释放！
--   2 |      0 |        0
--   3 |      0 |        0
--   4 |   8104 |        1    ← 活行不受影响
--   5 |   8016 |        1
铁证 #2：pd_upper = 6432 → 6432，完全没有任何变化。Phase 3 唯一做的事情就是 LP_DEAD → LP_UNUSED（行指针 lp_flags 从 3 变为 0），每个槽位仅回收 4 字节（30 × 4 = 120B）。元组数据在 Phase 1 修剪时已经全部回收完毕。

阶段 4：INSERT 复用 LP_UNUSED 槽位

INSERT INTO vac_test (data)
SELECT 'reused_' || LPAD(g::text, 10, '0')
FROM generate_series(1, 5) g;

SELECT lower, upper FROM page_header(get_raw_page('vac_test', 0));
--  lower | upper
-- ───────┼───────
--    224 |  6192    ← pd_upper 从 6432 → 6192（新元组占用 240B）

SELECT count(*) FILTER (WHERE lp_flags = 0) AS LP_UNUSED,
       count(*) FILTER (WHERE lp_flags = 1) AS LP_NORMAL
FROM heap_page_items(get_raw_page('vac_test', 0));

--  lp_unused | lp_normal
-- ───────────┼───────────
--         25 |        25    ← 5 LP_UNUSED 被新 INSERT 复用

新 INSERT 优先复用 LP_UNUSED 槽位，而非在 pd_lower 下方分配新行指针。pd_upper 从 6432 下移到 6192（新元组数据 240B = 5 × 48B），行指针总数保持 50 不变。

两个不可辩驳的铁证：

  1. Phase 1 回收元组字节：pd_upper 从 3792 → 6432（2640B = 30 × 88B），行指针 LP_NORMAL → LP_DEAD（lp_off=0）
  2. Phase 3 只回收行指针槽位：pd_upper 6432 → 6432（纹丝不动），唯一变化是 lp_flags 从 3 → 0（每槽位 4 字节）

INDEX_CLEANUP OFF 的代价

VACUUM (INDEX_CLEANUP OFF) 跳过了 Phase 2 和 Phase 3，这会带来：

好处代价
✅ 跳过索引扫描（最昂贵阶段） ❌ 死索引条目永久累积→ 索引膨胀
✅ 元组空间仍被回收 ❌ LP_DEAD 槽位永不释放
✅ 快速完成 ❌ 索引扫描越来越慢（死条目越来越多）
紧急场景可用，但不要设为默认。如果表有大量索引且 VACUUM 时间不可接受，可以先 INDEX_CLEANUP OFF 快速回收空间，然后在低峰期跑完整 VACUUM。永远不要长期跳过索引清理。

惰性 VACUUM vs VACUUM FULL

维度惰性 VACUUM（常规）VACUUM FULL
锁级别ShareUpdateExclusiveLock
✅ 不阻塞读写		AccessExclusiveLock
❌ 阻塞所有访问
能力回收页面内空间 → 新 INSERT 可复用
截断末尾空页 → 归还 OS		整个表重写 → 紧密打包存活行

文件收缩仅末尾连续空页整个表文件

核心区别：惰性 VACUUM 使空间可复用（表达到稳态大小后不再增长），VACUUM FULL 将内部空闲空间压缩出文件（但代价高昂）。

完整生命周期追踪（实测数值版）

INSERT（50 行，约 88B/行）:
  pd_lower: 24 → 224（+200B = 50 × 4B 行指针）
  pd_upper: 8192 → 3792（-4400B = 50 × 88B 元组数据）
  行指针: 0 LP_UNUSED / 50 LP_NORMAL / 0 LP_DEAD

DELETE（30 行，id % 5 IN (1,2,3)）:
  t_xmax = 880（删除事务 XID）
  pd_lower: 224 → 224  ← 不变
  pd_upper: 3792 → 3792 ← 不变！
  行指针: 0 LP_UNUSED / 50 LP_NORMAL / 0 LP_DEAD ← 全部不变！
  → DELETE 只是标记元组头部，不碰页面布局

VACUUM Phase 1 — VACUUM (INDEX_CLEANUP OFF):
  ✅ 修剪死元组 → pd_upper: 3792 → 6432（回收 2640B = 30 × 88B）
  ✅ FSM 注册回收空间 → INSERT 可立即复用
  行指针: 0 LP_UNUSED / 20 LP_NORMAL / 30 LP_DEAD
  → LP_DEAD: lp_off=0, lp_flags=3, xmin/xmax 已清除
  → pd_lower 不变（224），行指针数组大小不变

VACUUM Phase 2（索引清理）:
  ✅ 扫描 pkey 索引，移除 30 个死 TID 的索引条目
  → 堆页面 pd_lower/upper/行指针状态完全不变化

VACUUM Phase 3（堆清理）:
  ✅ LP_DEAD → LP_UNUSED: lp_flags 3→0（30 个槽位 × 4B = 120B）
  行指针: 30 LP_UNUSED / 20 LP_NORMAL / 0 LP_DEAD
  ✅ VM all_visible 位设置
  → pd_upper: 6432 → 6432 ← 纹丝不动！元组数据 Phase 1 已回收

INSERT 5 行（复用 LP_UNUSED 槽位）:
  pd_upper: 6432 → 6192（新元组 240B = 5 × 48B）
  行指针: 25 LP_UNUSED / 25 LP_NORMAL / 0 LP_DEAD
