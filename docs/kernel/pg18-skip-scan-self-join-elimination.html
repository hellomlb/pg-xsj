一句话总结

AWS 官方博客宣称 PostgreSQL 18 带来三大性能增强，本文在真实 PG18.4 环境上逐一实测，用 PG16.11 + PG16.14 双版本做反例对照，结论：三项全部属实，而且拿到了真实提速数字——

一、Skip Scan：复合索引跳过前导列

是什么

假设你有复合 B-tree 索引 (status, created_date)。在 PG18 之前，如果查询只过滤 created_date 而不带前导列 status，这个索引几乎用不上——优化器往往退化成全表扫描。

PG18 的 Skip Scan：优化器能"跳过前导列 status 的每一个不同值"，对每个值分别做一次索引搜索，相当于把一个复合索引当成多个单列索引来用。前导列基数越低（比如 status 只有 5 个值），收益越大。

实测：500 万行同机基准

CREATE TABLE orders_bench (
    order_id int, status int,        -- status 低基数：只有 5 个值
    created_date date, amount numeric(10,2));
INSERT INTO orders_bench
SELECT g, g % 5, '2024-01-01'::date + (g % 1000), (random()*1000)::numeric(10,2)
FROM generate_series(1, 5000000) g;
CREATE INDEX ob_idx ON orders_bench(status, created_date);  -- 前导列 status
ANALYZE orders_bench;

-- 只过滤非前导列 created_date
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders_bench WHERE created_date = '2024-06-15';

A) PG18 默认（Skip Scan 生效）实测输出：

Bitmap Heap Scan on orders_bench (actual time=2.369..7.946 rows=5000 loops=1)
  Recheck Cond: (created_date = '2024-06-15'::date)
  Heap Blocks: exact=5000
  Buffers: shared hit=5026
  ->  Bitmap Index Scan on ob_idx (actual time=1.036..1.036 rows=5000 loops=1)
        Index Cond: (created_date = '2024-06-15'::date)
        Index Searches: 7                    ← ★ Skip Scan 的实测标志
        Buffers: shared hit=26
 Execution Time: 8.282 ms

B) 强制关闭索引扫描（模拟 PG18 之前只能全表扫描）：

SET enable_indexscan = off; SET enable_bitmapscan = off;
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders_bench WHERE created_date = '2024-06-15';

Gather (actual time=0.736..672.742 rows=5000 loops=1)
  Workers Launched: 2
  Buffers: shared hit=15998 read=15850          ← 扫了 ~31800 个块
  ->  Parallel Seq Scan on orders_bench (actual time=0.564..621.066 rows=1666 loops=3)
        Filter: (created_date = '2024-06-15'::date)
        Rows Removed by Filter: 1665000          ← 每个 worker 白扫 166 万行
 Execution Time: 673.164 ms

A) Skip ScanBitmap Index Scan（Index Searches: 7）50268.3 ms
B) 全表扫描Parallel Seq Scan（2 workers）~31800673 ms

➡️ 约 80× 提速，读取块数降到 1/6，且无需新建任何索引——同一个复合索引，PG18 就能榨出这个收益。

⚠️ 一个必须纠正的认知

很多人以为 Skip Scan 会在执行计划里显示一个叫 Index Skip Scan 的独立节点名。实测证明这是误解：

-- 在 PG18 强制走纯 Index Scan 路径
SET enable_bitmapscan = off; SET enable_seqscan = off;
EXPLAIN (COSTS OFF) SELECT * FROM skip_test WHERE created_date = '2024-01-15';

Index Scan using skip_test_idx on skip_test       ← 就是普通 Index Scan
  Index Cond: (created_date = '2024-01-15'::date)

PG18 和 PG16.14 在这条路径下都显示普通 Index Scan。Skip Scan 的真正标志是 Bitmap/Index Scan 里的 Index Searches: N 计数（N = 跳过前导列后做的分段索引搜索次数）。看计划时别被字面误导。

三版本对照

版本Bitmap Index Scan 是否有 Index Searches
PG 16.11❌ 无该字段
PG 16.14❌ 无该字段
PG 18.4✅ Index Searches: 7

适用场景

  • 前导列基数低（status / type / is_deleted 等少量枚举值）
  • 查询常常只过滤非前导列
  • 非前导列选择率高（能过滤掉大部分行）

二、Self-Join Elimination：自动消除冗余自连接

是什么

ORM（Django / Rails / Prisma）经常生成 FROM t a JOIN t b ON a.id = b.id 这种基于主键的自连接。因为 a.id = b.id 且 id 是主键，a、b 必然是同一行——这个 JOIN 是恒等变换，纯属浪费。

PG18 的规划器能自动识别并把它重写成单次表扫描，客户端零代码改动即可提速。

实测：100 万行主键自连接，同机 on/off

PG18 提供了 enable_self_join_elimination 开关（默认 on），正好用它做同机公平对比：

CREATE TABLE emp_bench (id int PRIMARY KEY, name text, salary numeric);
INSERT INTO emp_bench SELECT g, 'emp_'||g, (random()*100000)::numeric
FROM generate_series(1,1000000) g;
ANALYZE emp_bench;

EXPLAIN (ANALYZE, BUFFERS, COSTS OFF)
SELECT e1.id, e1.name FROM emp_bench e1 JOIN emp_bench e2 ON e1.id = e2.id
WHERE e1.salary > 50000;

A) enable_self_join_elimination = on（PG18 默认，消除）：

Seq Scan on emp_bench e2 (actual time=0.016..494.548 rows=500010 loops=1)
  Filter: (salary > '50000'::numeric)
  Buffers: shared hit=7352                    ← 只扫一遍
 Execution Time: 523.063 ms

B) enable_self_join_elimination = off（模拟 PG18 之前，走 Hash Join）：

Hash Join (actual time=734.286..1758.000 rows=500010 loops=1)
  Hash Cond: (e2.id = e1.id)
  Buffers: shared hit=14704, temp read=3799 written=3799   ← ★ 溢出磁盘！
  ->  Seq Scan on emp_bench e2 ... rows=1000000            ← 扫全表
  ->  Hash (actual time=733.609..733.611 rows=500010)
        Batches: 4  Memory Usage: 7760kB                   ← hash 分 4 批
        ->  Seq Scan on emp_bench e1 ... rows=500010       ← 又扫一遍
 Execution Time: 1787.109 ms
路径执行计划读取块数临时文件耗时
A) SJE 消除单 Seq Scan7352无523 ms
B) Hash JoinHash Join + 双 Seq Scan14704溢出 3799 块1787 ms

➡️ 约 3.4× 提速。更关键的是：SJE 不仅省掉一次扫描，还避免了 Hash Build 的内存/临时文件溢出（off 时 hash 分 4 批、读写各 3799 块临时文件）。表越大、内存越紧张，这个收益越明显。

三版本对照

版本主键自连接的执行计划
PG 16.11Hash Join（双表扫描）
PG 16.14Hash Join（双表扫描）
PG 18.4✅ 单 Seq Scan（自连接被消除）

触发条件

  • inner join，连接键是主键 / 唯一键
  • SELECT / WHERE 未引用内表中无法被外表替代的列
  • 仅 SELECT 生效——UPDATE / DELETE / MERGE 中的自连接不消除（因 EPQ 重评估语义）

三、Autovacuum 三个新 GUC：大表治理

在 PG18.4 上查 autovacuum 相关 GUC 共 17 个，其中三个是 PG16 完全没有的新参数（PG16.11/16.14 实测均为 0）：

新 GUCPG18 默认值作用调整级别
autovacuum_vacuum_max_threshold 100000000 触发 autovacuum 的死元组数硬上限（与表大小无关）。10 亿行大表不用再积 2 亿死元组才触发；防膨胀 + 防 txid 回卷 sighup（免重启）
vacuum_truncate（升为全局 GUC） on 控制 VACUUM 是否截断表尾空页。截断需 AccessExclusiveLock，在备库易致 canceling statement due to conflict with recovery；设 off 免锁。PG18 前只能逐表设 sighup
autovacuum_worker_slots 16 autovacuum worker 的后端槽位数硬上限，与实际 autovacuum_max_workers（默认 3）解耦，高写入期可免重启动态加 worker postmaster 
-- 单表覆盖硬上限
ALTER TABLE orders SET (autovacuum_vacuum_max_threshold = 500000);
-- 大表禁用硬上限，仍按 scale_factor
ALTER TABLE archive_data SET (autovacuum_vacuum_max_threshold = -1);

为什么 max_threshold 重要：老的触发公式是 threshold + scale_factor × reltuples。对 10 亿行大表，scale_factor=0.2 意味着要积累 2 亿死元组才触发 vacuum——表早就膨胀到不像话了。新参数给了一个绝对上限，让大表也能被及时清理。

五、诚实边界

  • 本文所有提速倍数均为 PG18 同机 on/off 对比（enable_* 开关切换新旧行为），公平可信。
  • PG16.11/16.14 仅用于确认执行计划层面的版本差异，不参与跨机耗时比较。
  • AWS 博客的 21× 是 Aurora db.r6g.large 环境数字，与本文 aarch64 本地环境不同，两者都真实但环境各异，不能混谈。
  • 本文数据集为构造数据（500 万 / 100 万行），生产真实收益取决于数据分布、前导列基数、内存配置，建议在你自己的环境复测。
