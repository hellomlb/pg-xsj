PG19 应用时间时态表实测：三特性全部验证通过

2026-06-16 PG19 时态表 SQL:2011 FOR PORTION OF WITHOUT OVERLAPS PERIOD FK

一、是什么

PostgreSQL 19 实现了 SQL:2011 标准的应用时间（Application-Time）时态表。从提出到落地，社区等了 15 年。

大白话：以前你手写的那一堆 BETWEEN + EXCLUDE USING gist + 手动行拆分逻辑，PG19 原生语法搞定。三大核心能力：

能力语法做什么
时间不重叠约束 PRIMARY KEY (id, valid_at WITHOUT OVERLAPS) 引擎级保证同一实体时间线永不重叠
自动行拆分 DML UPDATE/DELETE ... FOR PORTION OF 修改时间段时自动切开历史，无间隙无重叠
时间外键 FOREIGN KEY (x, PERIOD t) REFERENCES y (x, PERIOD t) 引用在整个时间段内必须存在

二、实测环境

版本: PostgreSQL 19beta1 on x86_64-pc-linux-gnu
扩展: btree_gist 1.9（手动编译安装）
      pg_plan_advice, pg_stash_advice, plpgsql

⚠️ 重要前提：WITHOUT OVERLAPS 和 PERIOD FK 依赖 btree_gist 扩展提供非 range 列（如 integer）的 GiST 操作符类。不装 btree_gist 的话只能测 FOR PORTION OF（它是纯 DML 特性，不依赖 GiST 索引）。

三、特性一：WITHOUT OVERLAPS 时间不重叠约束

3.1 建表

CREATE EXTENSION btree_gist;  -- 必须先装

CREATE TABLE salary (
    emp_id    INTEGER NOT NULL,
    emp_name  TEXT NOT NULL,
    amount    NUMERIC(10,2) NOT NULL,
    valid_at  DATERANGE NOT NULL,
    PRIMARY KEY (emp_id, valid_at WITHOUT OVERLAPS)
);
背后的机制：WITHOUT OVERLAPS 内部等价于 EXCLUDE USING gist (emp_id WITH =, valid_at WITH &&)，由 GiST 索引驱动。btree_gist 提供了 integer 列的 GiST "=" 操作符类——这是缺了它就报错 data type integer has no default operator class for access method "gist" 的原因。

3.2 插入数据

INSERT INTO salary VALUES
    (1, '小王', 8000,  '[2020-01-01,2022-01-01)'),
    (1, '小王', 12000, '[2022-01-01,2024-01-01)'),
    (1, '小王', 15000, '[2024-01-01,infinity)');
-- ✅ 3 行插入成功

3.3 重叠插入被自动拒绝

INSERT INTO salary VALUES (1, '小王', 9000, '[2021-06-01,2022-06-01)');
-- ❌ ERROR: conflicting key value violates exclusion constraint "salary_pkey"

实测输出：PG 自动拒绝，因为 [2021-06,2022-06) 与第一行 [2020-01,2022-01) 重叠（2021-06 到 2022-01 重叠区间）。引擎级保证，不需要应用代码检查。

四、特性二：FOR PORTION OF 自动行拆分

这是最震撼的特性。

4.1 初始数据

emp_nameamountvalid_at
小王8000[2020-01-01, 2022-01-01)
小王12000[2022-01-01, 2024-01-01)
小王15000[2024-01-01, ∞)

4.2 UPDATE FOR PORTION OF

2023年7月到12月项目津贴，工资涨到 18000：

UPDATE salary
    FOR PORTION OF valid_at FROM '2023-07-01' TO '2024-01-01'
    SET amount = 18000
    WHERE emp_id = 1;

PG19 Beta1 实测输出：

emp_nameamountvalid_at说明
小王8000[2020-01-01, 2022-01-01)未碰触
小王12000[2022-01-01, 2023-07-01)前半段残余
小王18000[2023-07-01, 2024-01-01)← 新值
小王15000[2024-01-01, ∞)未碰触
发生了什么？
小王第二行 [2022-01, 2024-01) 与目标区间 [2023-07, 2024-01) 重叠。PG 把它从 2023-07-01 处切开：前面保留 12000，重叠部分更新为 18000。第三行起点恰好等于目标终点（daterange 右开，不含 2024-01-01），不受影响。

1 行被切 → 2 行，原子操作，无间隙无重叠。 以前这要写循环 + INSERT + UPDATE 几十行应用代码。

4.3 DELETE FOR PORTION OF

挖掉 2025年6月到10月：

DELETE FROM salary
    FOR PORTION OF valid_at FROM '2025-06-01' TO '2025-10-01'
    WHERE emp_id = 3;

实测：2 行 → 3 行

emp_nameamountvalid_at说明
老张30000[2024-01-01, 2025-06-01)前半段残余
老张30000[2025-10-01, 2026-01-01)后半段残余
老张35000[2026-01-01, ∞)未碰触
原一行为 [2024-01, 2026-01)，被挖掉中间的 [2025-06, 2025-10)，自动裂变为前后两段残余。 一行 DELETE，一行被挖断——无间隙。

五、特性三：PERIOD 时态外键

这是最容易被误解的特性——它不是你想象的那种外键。

5.1 场景

员工分配到项目。被引用表 project 有两行（凤凰项目 + 凤凰二期），联合覆盖了 [2024-01-01, ∞)：

CREATE TABLE project (
    proj_id   INTEGER NOT NULL, proj_name TEXT NOT NULL, valid_at DATERANGE NOT NULL,
    PRIMARY KEY (proj_id, valid_at WITHOUT OVERLAPS)
);

INSERT INTO project VALUES
    (101, '凤凰项目', '[2024-01-01,2025-07-01)'),
    (101, '凤凰二期', '[2025-07-01,infinity)');

5.2 建 PERIOD FK

CREATE TABLE project_assign (
    assign_id INTEGER NOT NULL, emp_id INTEGER NOT NULL,
    proj_id INTEGER NOT NULL, valid_at DATERANGE NOT NULL,
    PRIMARY KEY (assign_id, valid_at WITHOUT OVERLAPS),
    FOREIGN KEY (proj_id, PERIOD valid_at)
        REFERENCES project (proj_id, PERIOD valid_at)
);

5.3 合法引用

-- assign 完全在单行 project 范围内
INSERT INTO project_assign VALUES (1, 1, 101, '[2024-06-01,2025-01-01)');
-- ✅ 成功

5.4 PERIOD FK 的 UNION 语义（重要发现）

-- assign 跨越两行 project，但联合覆盖了整个区间
INSERT INTO project_assign VALUES (2, 1, 101, '[2024-01-01,2026-01-01)');
-- ✅ 成功！（project 两行的 UNION = [2024-01-01, infinity) ⊇ assign）
🔑 核心发现：PERIOD FK 使用多行组合覆盖（UNION 语义）！

assign_id=2 的区间 [2024-01,2026-01) 跨越了 project 的两行（凤凰项目 + 凤凰二期），但两行的 UNION [2024-01,∞) 完全覆盖了它——所以引用合法。

这与普通外键的"一行引用一行"逻辑完全不同：只要被引用表中有匹配 proj_id 的行集合，其时间范围的 UNION 覆盖了引用行的整个期间，就算满足约束。

5.5 真正非法的引用

-- assign 从 2023 年开始，但 project 最早从 2024 年开始
INSERT INTO project_assign VALUES (3, 1, 101, '[2023-06-01,2025-01-01)');
-- ❌ 应被拒绝：2023-06 到 2024-01 没有 project 行覆盖

🧪 本测试环境 project 只覆盖到 [2024-01,∞)，所以任何从 2024-01 之前开始的 assign 都应被 PERIOD FK 拒绝。

六、WITHOUT OVERLAPS 与 btree_gist 依赖关系

整条依赖链是：

WITHOUT OVERLAPS → GiST 索引 → btree_gist 扩展 → contrib 包
PERIOD FK       → 被引用表有 GiST 唯一约束 → 同上

不装 btree_gist 时的报错：

CREATE TABLE t (id int, v daterange, PRIMARY KEY (id, v WITHOUT OVERLAPS));
-- ERROR: data type integer has no default operator class
--        for access method "gist"

为什么 btree_gist 编译安装即可：它是一个纯 SQL 扩展，不依赖操作系统共享库之外的任何东西。手写 CREATE EXTENSION 即可，不需要改 postgresql.conf。

八、生产建议

  1. 装 btree_gist：编译安装或 yum/apt install postgresqlXX-contrib
  2. 时态 UPDATE/DELETE 前加锁：READ COMMITTED 下并发时态 DML 可能丢失效果，先 SELECT ... FOR UPDATE
  3. PERIOD FK 是多行 UNION 语义：不是一行对一行，别用普通外键的思维去理解
  4. PG19 是 Beta：当前为 Beta1，生产等正式版
  5. 时态外键仅 NO ACTION：不支持 CASCADE / SET NULL / SET DEFAULT
  6. 仅支持应用时间：系统时间（谁何时改了数据）仍需要触发器或扩展模拟

九、总结

PG19 应用时间时态表三特性全部验证通过：

  • WITHOUT OVERLAPS — 引擎级时间不重叠保证，重叠插入自动拒绝
  • FOR PORTION OF — 一行 SQL 自动行拆分，原子操作无间隙
  • PERIOD FK — 时间外键使用多行 UNION 覆盖语义，跨越多个被引用行仍合法

从 PG ≤18 到 PG19，最大的变化不是语法糖，而是数据库终于理解了"时间"——它不再是两个普通的日期列，而是一条不能有间隙、不能重叠的时间线。

保险/金融/人力资源场景的核心基础设施。15 年圆梦，值。
