PG19 应用时间时态表实测：FOR PORTION OF 一行 SQL 自动切行

2026-06-16 PG19 时态表 SQL:2011 FOR PORTION OF

一、是什么

PostgreSQL 19 终于等来了 SQL:2011 标准的应用时间（Application-Time）时态表。从提出到落地，社区等了 15 年。

大白话：以前你手写的那一堆 BETWEEN + EXCLUDE USING gist + 手动行拆分逻辑，PG19 原生语法搞定。

时态表让数据库理解时间不是普通数据，而是一条不能重叠的时间线。核心能力有三个：

能力语法做什么
时间不重叠约束 PRIMARY KEY (id, valid_at WITHOUT OVERLAPS) 引擎级保证同一实体时间线永不重叠
自动行拆分 DML UPDATE/DELETE ... FOR PORTION OF 修改时间段时自动切开历史，无间隙无重叠
时间外键 FOREIGN KEY (x, PERIOD t) REFERENCES y (x, PERIOD t) 引用在整个时间段内必须存在

二、实测环境

版本: PostgreSQL 19beta1 on x86_64-pc-linux-gnu
扩展: 仅 pg_plan_advice, pg_stash_advice, plpgsql 等 6 个
注意: btree_gist 未安装（影响 WITHOUT OVERLAPS 和 PERIOD FK）

⚠️ 重要发现：在缺 btree_gist 扩展的情况下，WITHOUT OVERLAPS 和 PERIOD FK 无法创建（它们依赖 GiST 操作符类），但 FOR PORTION OF 不受影响——它是纯 DML 特性，只要有 daterange 列就能用。

三、FOR PORTION OF — 自动行拆分

这是最震撼的特性。场景：员工工资历史，某段时间有特殊津贴。

3.1 建表与初始数据

CREATE TABLE salary (
    emp_id    INTEGER NOT NULL,
    emp_name  TEXT NOT NULL,
    amount    NUMERIC(10,2) NOT NULL,
    valid_at  DATERANGE NOT NULL
);

INSERT INTO salary VALUES
    (1, '小王', 8000,  '[2020-01-01,2022-01-01)'),
    (1, '小王', 12000, '[2022-01-01,2024-01-01)'),
    (1, '小王', 15000, '[2024-01-01,infinity)');
emp_nameamountvalid_at
小王8000[2020-01-01, 2022-01-01)
小王12000[2022-01-01, 2024-01-01)
小王15000[2024-01-01, ∞)

3.2 UPDATE FOR PORTION OF

2023年7月到12月项目津贴，工资涨到 18000：

UPDATE salary
    FOR PORTION OF valid_at FROM '2023-07-01' TO '2024-01-01'
    SET amount = 18000
    WHERE emp_id = 1;

真实执行结果（PG19 Beta1 实测输出）：

emp_nameamountvalid_at说明
小王8000[2020-01-01, 2022-01-01)未碰触，原样
小王12000[2022-01-01, 2023-07-01)前半段残余
小王18000[2023-07-01, 2024-01-01)新值覆盖
小王15000[2024-01-01, ∞)未碰触，原样
发生了什么？
小王原来第二行 [2022-01, 2024-01) 覆盖了 2023-07 到 2024-01 这个目标区间。PG19 自动把这行从 2023-07-01 处切开：前面保留 12000，重叠部分更新为 18000。第三行 [2024-01, ∞) 的起点恰好等于目标终点（daterange 左闭右开，不包含 2024-01-01），所以不受影响。
2 行受影响 → 1 行切为 2 行。所有拆分原子操作。

3.3 DELETE FOR PORTION OF

挖掉 2025年6月到10月这段时间：

DELETE FROM salary
    FOR PORTION OF valid_at FROM '2025-06-01' TO '2025-10-01'
    WHERE emp_id = 2;

初始 2 行：[2024-01-01, 2026-01-01) 和 [2026-01-01, ∞)

实测结果：2 行 → 3 行（中间挖掉 4 个月后分裂为 2 段）

emp_nameamountvalid_at
小李24000[2024-01-01, 2025-06-01)
小李24000[2025-10-01, 2026-01-01)
小李28000[2026-01-01, ∞)
一行 DELETE，一行被挖断，一行没碰。被挖断的行自动裂变为两段残余——以前要写循环 + INSERT + UPDATE 几十行代码。

四、WITHOUT OVERLAPS 与 btree_gist 依赖

WITHOUT OVERLAPS 需要 btree_gist 扩展提供非 range 列（如 integer）的 GiST 操作符类：

-- ❌ btree_gist 未安装时：
CREATE TABLE salary (
    emp_id    INTEGER NOT NULL,
    valid_at  DATERANGE NOT NULL,
    PRIMARY KEY (emp_id, valid_at WITHOUT OVERLAPS)
);
-- ERROR: data type integer has no default operator class for access method "gist"

-- ✅ 安装后正常：
CREATE EXTENSION btree_gist;
-- 上述 CREATE TABLE 成功

这个依赖关系意味着：生产环境部署前，确保 postgresql-contrib 包已安装，否则 WITHOUT OVERLAPS 和 PERIOD FK 都不可用。

五、时间点查询

时态表搭配 range 操作符做时间点查询非常自然：

-- 查 2023年6月的工资
SELECT * FROM salary
WHERE emp_id = 1 AND valid_at @> '2023-06-01'::date;
-- 结果: 小王 | 12000 | [2022-01-01,2023-07-01)

-- 查 2023年10月的工资（津贴期间）
SELECT * FROM salary
WHERE emp_id = 1 AND valid_at @> '2023-10-01'::date;
-- 结果: 小王 | 18000 | [2023-07-01,2024-01-01)

六、完整能力矩阵

特性是否依赖 btree_gist本实例实测
FOR PORTION OF UPDATE❌ 不需要✅ 通过
FOR PORTION OF DELETE❌ 不需要✅ 通过
时间点查询 @>❌ 不需要✅ 通过
WITHOUT OVERLAPS✅ 需要❌ 未安装
PERIOD FK✅ 需要❌ 未安装

七、生产建议

  1. 装 btree_gist：yum install postgresql19-contrib 或 apt install postgresql-19 完整安装
  2. 时态 UPDATE/DELETE 前加锁：READ COMMITTED 下并发时态 DML 可能丢失效果，先 SELECT ... FOR UPDATE
  3. PG19 是 Beta：当前为 Beta1，生产等正式版
  4. 时态外键仅 NO ACTION：不支持 CASCADE / SET NULL / SET DEFAULT
  5. 仅支持应用时间：系统时间（谁何时改了数据）仍需要触发器模拟

八、总结

PG19 的时态表在 核心 DML 体验上已经可用：FOR PORTION OF 在 btree_gist 缺失的情况下工作良好，自动行拆分机制正确。约束层（WITHOUT OVERLAPS、PERIOD FK）需要完整 contrib 包。

从 PG ≤18 到 PG19，最大的变化不是语法糖，而是数据库终于理解了"时间"——它不再是两个普通的日期列，而是一条不能有间隙、不能重叠的时间线。
