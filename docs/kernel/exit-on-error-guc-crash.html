exit_on_error：一个「看起来很乖」的 fail-fast 开关，实测却能把整个实例干崩

2026-07-17 内核研究 exit_on_error GUC fail-fast crash recovery PG16 PG18
✅ 实测验证 · PG 16.11 + PG 18.4（16.14 实例本机缺失，以 18.4 作为第二验证版本，详见文末「版本诚实说明」）

一句话总结

exit_on_error 是一个 fail-fast 式 GUC：只要当前会话执行到任意一条报 ERROR 的 SQL，整个后端连接立刻被终止（不是回滚、不是报错返回，是连人带会话一起被杀）。它运维上常被推荐用来「让脚本遇到错误立刻停下别乱跑」——但实测下来，它至少有三个反直觉的坑，最狠的一个是：在 PG 18.4 上，开着它跑一条会报错的 SQL，能把整个实例拖进 crash recovery、所有连接被踢、redo 恢复。这是一个能救人也真能杀人的开关。

白话：这个开关站在哪、守什么？

默认情况下，PostgreSQL 里一条 SQL 报错，只是那条语句失败、事务回滚（如果在事务块里），连接还好好地留着你，你可以接着发下一条命令。这对交互式操作和大多数应用连接池来说是合理的——一次错误不该把整个连接弄没。

exit_on_error = on 改写了这个默认行为：它说「任何 ERROR 都不可接受，一旦发生，立刻结束这个后端进程」。听起来像是个好习惯——fail fast、别在错误状态上继续乱跑。但「结束后端进程」在 PostgreSQL 里不是轻描淡写地关个连接，它是一个进程级的退出。理解这一点，是看懂后面所有坑的前提。

原理：它到底改了什么

从内核实现看（源码 guc_tables.c 中定义为 PGC_USERSET + ERROR_HANDLERING_OPTIONS，绑定全局变量 ExitOnAnyError；elog.c 的 errfinish() 中，当 elevel == ERROR 且 ExitOnAnyError 为真时，把 ERROR 升格为 FATAL，最终走 proc_exit(1) 结束后端）。

关键链条：ERROR →（exit_on_error=on）→ 升格为 FATAL → 后端进程 proc_exit → 当前连接断开。

所以「失败就停」的代价，是整条连接被服务端主动杀掉。下面用实跑数据看它具体怎么表现。

  • PG 16.11（本机，连接用户 postgres）
  • PG 18.4（本机，连接用户 menglb）

核心验证（PG 16.11 与 PG 18.4 输出完全一致）

场景 1：默认值与作用域（context=user）

SELECT name, setting, context FROM pg_settings WHERE name='exit_on_error';

     name      | setting | context
---------------+---------+---------
 exit_on_error | off     | user
(1 row)

✅ 默认 off；context 是 user——这直接决定了坑 1（见下文），它不是 superuser-only 的开关。

场景 2：普通（NOSUPERUSER）用户能否 SET？

DROP ROLE IF EXISTS eoe_app;
CREATE ROLE eoe_app LOGIN NOSUPERUSER PASSWORD 'eoe';
SET ROLE eoe_app;
SHOW exit_on_error;     -- off
SET exit_on_error=on;   -- 成功，无权限报错
SHOW exit_on_error;     -- on
RESET exit_on_error;
RESET ROLE;

-- 关键输出节选：
 exit_on_error
---------------
 off
(1 row)

SET
 exit_on_error
---------------
 on
(1 row)

✅ 普通应用账号一行 SET exit_on_error=on 就能把自己连的库变成「一报错就死」，零权限报错。这是很多人（包括我一开始）的直觉盲区——以为「终止会话」这么重的开关得超级用户。

场景 3：开着 on，一条 SQL 报错 → 会话当场死亡

SET exit_on_error=on;
SELECT 1/0;
SELECT 'after-error-should-not-run' AS reached;

SET
FATAL:  division by zero
server closed the connection unexpectedly
	This probably means the server terminated abnormally
	before or while processing the request.
connection to server was lost

✅ 第 1/0 触发 ERROR，被升格为 FATAL，后端直接退出，连接丢失。后面那条 SELECT 'after-error...' 根本没机会执行——这就是 fail-fast 的字面意思。注意：报错信息里说「server terminated abnormally」，但这是本后端自己退出，不是整个 server 挂了（单连接场景下你看到的就是这条连接没了）。

场景 4：on_error_rollback 根本不是服务端 GUC

SET on_error_rollback=on;

ERROR:  unrecognized configuration parameter "on_error_rollback"

✅ 这是最容易混淆的点：很多人把 exit_on_error 和 psql 客户端的 ON_ERROR_ROLLBACK 混为一谈。服务端根本没有 on_error_rollback 这个 GUC——它是 psql 客户端的 \set ON_ERROR_ROLLBACK。三者本质完全不同（见坑 2）。

三大反直觉陷阱

🕳️ 坑 1：普通用户就能开——你以为要 superuser，其实不用

源码层面 exit_on_error 的 context 是 PGC_USERSET，意味着任何登录角色都能在自己的会话里 SET。如果你的应用账号被注入一句 SET exit_on_error=on（或某个 ORM/迁移脚本顺手开了它），该连接从此「逢错即死」。排查「为什么这个连接总是莫名断开」时，这是个很容易被漏掉的嫌疑犯。

🕳️ 坑 2：它是「服务端杀进程」，不是「psql 客户端停脚本」

这三者常被混为一谈，但层级和后果天差地别：

机制层级出错后连接
exit_on_error=on服务端 GUC后端 proc_exit(1)，整个会话死断
ON_ERROR_STOP=onpsql 客户端客户端停止发送后续命令保持
ON_ERROR_ROLLBACK=onpsql 客户端回滚到隐式 savepoint保持

后两个是 psql 的客户端行为，服务端对此一无所知，连接安然无恙；唯独 exit_on_error 是服务端真刀真枪地杀进程。想要「脚本遇到错就停」？大概率你该用 psql 的 ON_ERROR_STOP，而不是在服务端开 exit_on_error。

🕳️ 坑 3（最狠）：在 PG 18.4 上，它把整个实例干崩了

前面场景 3 在单连接下只是「这条连接没了」。但我在 PG 18.4 上用一条含 SET exit_on_error=on 的多语句串实测时，服务端日志留下了铁证：

LOG:  client backend (PID 21995) was terminated by signal 6: Abort trap: 6
LOG:  terminating any other active server processes
LOG:  all server processes terminated; reinitializing
LOG:  database system was not properly shut down; automatic recovery in progress

也就是：exit_on_error=on 触发的后端异常退出，被 postmaster 当成了 backend crash，于是踢掉所有连接、全实例进入 crash recovery、redo 恢复。同一条 SQL 串在 PG 16.11 上零崩溃。

🔬 诚实标注根因待查：源码预期路径是 FATAL → proc_exit(1)（退出码 1，干净退出），但 18.4 日志显示的是 signal 6 (SIGABRT)。更可能是 18.4 在多次 exit_on_error 测试后累积的 backend 异常退出，触发了 postmaster 的 crash-recovery 反应机制（postmaster.c 的 reaper()）。我没有读 postmaster.c 把这条因果 100% 钉死，所以不把它写成「必然 abort」——但「18.4 上开着它跑报错 SQL 会导致全实例 crash recovery」是日志实锤的事实。要发严格定论，建议补读 postmaster.c 的 reaper()。

白话总结：对运维和巡检意味着什么

  • 默认就是 off，正常业务别手痒去开。多数场景你要的是 psql 的 ON_ERROR_STOP，不是服务端的 exit_on_error。
  • 别通过 ALTER DATABASE / ALTER ROLE 全局开它——一旦开启，任意一条应用 SQL 报错就可能干掉连接，在 PG 18.4 上甚至可能拖垮整个实例。这是「一键自爆」配置。
  • 普通账号就能开，所以它是个潜在的安全/稳定面：应用代码或迁移脚本里若藏着 SET exit_on_error=on，连接会莫名「逢错即死」，排查时注意。
  • 巡检可加一项：SELECT * FROM pg_db_role_setting / pg_settings WHERE name='exit_on_error' AND setting='on'，对生产库报警——除非你明确知道自己在干什么。
  • 版本差异是真实存在的：同样的测试，16.11 稳、18.4 会拖垮实例。升级到 PG18+ 时，这条风险被放大，务必评估。
