exit_on_error：一个「看起来很乖」的 fail-fast 开关，实测三大反直觉陷阱

2026-07-17 内核研究 exit_on_error GUC fail-fast PG16 PG18 PG19
✅ 实测验证 · 四版本（PG 16.11 / 16.14 / 18.4 / 19beta1）逐场景一致

一句话总结

exit_on_error 是一个 fail-fast 式 GUC：只要当前会话执行到任意一条报 ERROR 的 SQL，整个后端连接立刻被终止（不是回滚、不是报错返回，是连人带会话一起被杀）。它运维上常被推荐用来「让脚本遇到错误立刻停下别乱跑」——但实测下来，它至少有三个反直觉的坑。最容易被低估的，是它「失败后直接杀掉整条会话」这件事本身的代价：不是回滚、不是报错返回，而是连接被服务端主动终结。

白话：这个开关站在哪、守什么？

默认情况下，PostgreSQL 里一条 SQL 报错，只是那条语句失败、事务回滚（如果在事务块里），连接还好好地留着你，你可以接着发下一条命令。这对交互式操作和大多数应用连接池来说是合理的——一次错误不该把整个连接弄没。

exit_on_error = on 改写了这个默认行为：它说「任何 ERROR 都不可接受，一旦发生，立刻结束这个后端进程」。听起来像是个好习惯——fail fast、别在错误状态上继续乱跑。但「结束后端进程」在 PostgreSQL 里不是轻描淡写地关个连接，它是一个进程级的退出。理解这一点，是看懂后面所有坑的前提。

原理：它到底改了什么

从内核实现看（源码 guc_tables.c 中定义为 PGC_USERSET + ERROR_HANDLING_OPTIONS，绑定全局变量 ExitOnAnyError；elog.c 的 errfinish() 中，当 elevel == ERROR 且 ExitOnAnyError 为真时，把 ERROR 升格为 FATAL，最终走 proc_exit(1) 结束后端）。

关键链条：ERROR →（exit_on_error=on）→ 升格为 FATAL → 后端进程 proc_exit → 当前连接断开。

所以「失败就停」的代价，是整条连接被服务端主动杀掉。下面用实跑数据看它具体怎么表现。

  • PG 16.14
  • PG 19beta1

核心验证（PG 16.11 / 16.14 / 18.4 / 19beta1 输出完全一致）

场景 1：默认值与作用域（context=user）

SELECT name, setting, context FROM pg_settings WHERE name='exit_on_error';

     name      | setting | context
---------------+---------+---------
 exit_on_error | off     | user
(1 row)

✅ 默认 off；context 是 user——这直接决定了坑 1（见下文），它不是 superuser-only 的开关。四个版本输出完全相同。

场景 2：普通（NOSUPERUSER）用户能否 SET？

DROP ROLE IF EXISTS eoe_app;
CREATE ROLE eoe_app LOGIN NOSUPERUSER PASSWORD 'eoe';
SET ROLE eoe_app;
SHOW exit_on_error;     -- off
SET exit_on_error=on;   -- 成功，无权限报错
SHOW exit_on_error;     -- on
RESET exit_on_error;
RESET ROLE;

-- 关键输出节选：
 exit_on_error
---------------
 off
(1 row)

SET
 exit_on_error
---------------
 on
(1 row)

✅ 普通应用账号一行 SET exit_on_error=on 就能把自己连的库变成「一报错就死」，零权限报错。这是很多人（包括我一开始）的直觉盲区——以为「终止会话」这么重的开关得超级用户。四版本均验证通过。

场景 3：开着 on，一条 SQL 报错 → 会话当场死亡（四版本一致）

SET exit_on_error=on;
SELECT 1/0;
SELECT 'after-error-should-not-run' AS reached;

SET
FATAL:  division by zero
server closed the connection unexpectedly
	This probably means the server terminated abnormally
	before or while processing the request.
connection to server was lost

✅ 第 1/0 触发 ERROR，被升格为 FATAL，后端直接退出，连接丢失。后面那条 SELECT 'after-error...' 根本没机会执行——这就是 fail-fast 的字面意思。四个版本输出逐字相同。

🔬 断开后立即重连验证实例状态：会话结束后立即重连成功（SELECT 'instance_alive' ... 正常返回），证明仅仅是当前会话被终止，实例本身完好——这是理解「会话被终止」与「实例崩溃」区别的关键。

场景 4：on_error_rollback 根本不是服务端 GUC

SET on_error_rollback=on;

ERROR:  unrecognized configuration parameter "on_error_rollback"

✅ 这是最容易混淆的点：很多人把 exit_on_error 和 psql 客户端的 ON_ERROR_ROLLBACK 混为一谈。服务端根本没有 on_error_rollback 这个 GUC——它是 psql 客户端的 \set ON_ERROR_ROLLBACK。三者本质完全不同（见坑 2）。四版本均验证相同。

三大反直觉陷阱

🕳️ 坑 1：普通用户就能开——你以为要 superuser，其实不用

源码层面 exit_on_error 的 context 是 PGC_USERSET，意味着任何登录角色都能在自己的会话里 SET。如果你的应用账号被注入一句 SET exit_on_error=on（或某个 ORM/迁移脚本顺手开了它），该连接从此「逢错即死」。排查「为什么这个连接总是莫名断开」时，这是个很容易被漏掉的嫌疑犯。

🕳️ 坑 2：它是「服务端杀进程」，不是「psql 客户端停脚本」

这三者常被混为一谈，但层级和后果天差地别：

机制层级出错后连接
exit_on_error=on服务端 GUC后端 proc_exit(1)，整个会话死断
ON_ERROR_STOP=onpsql 客户端客户端停止发送后续命令保持
ON_ERROR_ROLLBACK=onpsql 客户端回滚到隐式 savepoint保持

后两个是 psql 的客户端行为，服务端对此一无所知，连接安然无恙；唯独 exit_on_error 是服务端真刀真枪地杀进程。想要「脚本遇到错就停」？大概率你该用 psql 的 ON_ERROR_STOP，而不是在服务端开 exit_on_error。

🕳️ 坑 3：误开或全局开启，会引发会话被批量终止的连接池雪崩

场景 3 已经证明：开着 exit_on_error=on 时，任意一条报 ERROR 的 SQL 都会让当前会话被服务端杀掉。这本身只是「这条连接没了」。但真正危险的是规模：

  • 如果通过 ALTER DATABASE / ALTER ROLE 全局或角色级开启，那么该库 / 该角色下所有连接都会「逢错即死」。
  • 应用层某条常见 SQL（某个边界值、某个并发冲突）一旦偶尔报错，就会成批杀掉连接，触发连接池耗尽 + 雪崩式重试，足以造成业务中断——而故障表象只是「连接莫名断开」，极难第一时间定位到这个 GUC。

🔬 为什么这是通用风险而非个案：会话被终止（ERROR→FATAL→proc_exit）在四个版本上逐字一致，是 exit_on_error 的固有行为，不依赖任何特定平台或触发条件。真正要在生产环境规避的，正是这种「一错杀全连」的放大效应。

白话总结：对运维和巡检意味着什么

  • 默认就是 off，正常业务别手痒去开。多数场景你要的是 psql 的 ON_ERROR_STOP，不是服务端的 exit_on_error。
  • 别通过 ALTER DATABASE / ALTER ROLE 全局开它——一旦开启，任意一条应用 SQL 报错就会杀掉对应连接。大量连接「逢错即死」本身就会引发应用层连接池耗尽、雪崩式重试，足以造成业务中断。
  • 普通账号就能开，所以它是个潜在的安全/稳定面：应用代码或迁移脚本里若藏着 SET exit_on_error=on，连接会莫名「逢错即死」，排查时注意。
  • 巡检可加一项：SELECT * FROM pg_db_role_setting / pg_settings WHERE name='exit_on_error' AND setting='on'，对生产库报警——除非你明确知道自己在干什么。
