PG19 在线 checksum 切换：告别停机恐惧

2026-07-06 内核研究 PG19 checksum data integrity online operation
✅ 已验证版 · PG 19 Beta1 实测

一句话总结

PG19 把 data_checksums 从布尔值升级为四值状态机，引入 pg_enable_data_checksums() / pg_disable_data_checksums() 两个函数在线切换——不需要停机，不需要 pg_checksums --enable，不需要重建备库。后台 worker 逐页计算 checksum，WAL 日志化自动复制到备库，cost_delay/cost_limit 可控节流。

为什么 checksum 这么重要

checksum 保护的是数据库最底层的数据完整性——当存储控制器返回了错误数据、宇宙射线翻转了某个 bit、或者文件系统静默损坏了一个页面时，PostgreSQL 需要有能力说："等等，这个页面不对。"

没有 checksum 的集群是"盲飞"：任何硬件层面的数据损坏都不会被检测到，系统会拿着错误的数据继续跑，直到某天一个奇怪的查询结果或不完整的索引扫描把你打醒。

此外，checksum 对 pg_rewind 至关重要。HA 切换后旧主库需要 rewind，必要条件之一是 wal_log_hints = on 或 data_checksums = on。只开 wal_log_hints 没有校验功能，只解决 rewind 需要；开 checksum 两者兼得。

历史痛点：13 年的妥协

版本里程碑局限
PG 9.3 (2013)引入 data checksums默认关闭，initdb 时决定，集群生命周期内无法改变
PG 11pg_verify_checksums能检测损坏，但不能切换开关
PG 12pg_checksums --enable/--disable必须完全停机，重写所有页面，中断无法恢复，备库需重建
PG 18默认开启 checksum新集群没问题，但老集群仍需离线切换
PG 19在线切换后台 worker + WAL 日志化，不停机不重建

这意味着：2013 年到 2025 年创建的无数老集群，PG19 终于可以在线开启 checksum 保护了。

PG19 四值状态机

data_checksums 不再是布尔值 on/off，而是一个四值 enum：

状态含义触发
offchecksums 关闭，页面不携带 checksum初始状态 / disable 完成
inprogress-on后台 worker 正在逐页计算 checksumpg_enable_data_checksums()
on全部页面已完成，checksums 生效worker 完成自动转换
inprogress-off正在关闭（几乎瞬时，只需等所有 backend 停止验证）pg_disable_data_checksums()

关键特性：

  • 不需要停机：业务读写正常进行
  • 中断可恢复：worker 会记住进度，重启后从断点继续
  • WAL 日志化：checksum 计算过程写入 WAL，自动复制到备库
  • 可节流：cost_delay（默认 0ms）和 cost_limit（默认 100）控制 IO 冲击

实测验证：完整状态循环

-- 1. 查看当前状态
SELECT current_setting('data_checksums');
-- on

-- 2. 关闭 checksum（几乎瞬时）
SELECT pg_disable_data_checksums();

-- 3. 确认关闭
SELECT current_setting('data_checksums');
-- off

-- 4. 重新开启（后台 worker 计算）
SELECT pg_enable_data_checksums(cost_delay => 10, cost_limit => 500);

-- 5. 观察中间状态
SELECT current_setting('data_checksums');
-- inprogress-on

-- 6. 等待 worker 完成后
SELECT current_setting('data_checksums');
-- on

从 on 到 off 是瞬时的：只需等待正在进行的 checksum 验证完成（微秒级），无需重写任何页面。从 off 到 on 需要后台计算：逐页读取、计算 checksum、写入页面头部，耗时取决于数据库大小和 IO 带宽。

函数签名

-- 开启（可选参数控制 IO 节流）
pg_enable_data_checksums(
    cost_delay integer DEFAULT 0,   -- 每轮休眠毫秒数
    cost_limit integer DEFAULT 100  -- 每轮 IO 预算
)

-- 关闭（无参数，瞬时完成）
pg_disable_data_checksums()

建议参数：生产环境建议 cost_delay = 10, cost_limit = 500，在 IO 冲击和完成速度之间取得平衡。如果数据库有备库，还需要注意备库的 IO 能力——WAL 重放也会产生对应计算开销。

PG16 对比确认

PG16.11 上不存在 pg_enable_data_checksums 和 pg_disable_data_checksums 函数：

SELECT proname FROM pg_proc
WHERE proname LIKE '%enable%checksum%'
   OR proname LIKE '%disable%checksum%';
-- (0 rows)

PG16 只能通过 pg_checksums --enable（必须停机）来切换。PG19 是首次实现在线切换的版本。

实战建议

哪些集群最需要立即开启

  • PG17/18 之前创建的、checksum = off 的老集群——这些集群已经"盲飞"多年
  • 运行在不可靠存储上的集群（云环境高性能实例也不可掉以轻心）
  • 需要 pg_rewind 的 HA 集群——开启 checksum 比只开 wal_log_hints 多了数据保护

注意事项

  • IO 冲击评估：TB 级数据库的 checksum 计算可能产生可观的读取流量，建议在低峰期启动，配合 cost_delay 参数
  • 无法回退到 PG18 以下：开启 checksum 后，数据文件格式中有 checksum 字段，PG18 之前的版本无法读取（但 PG18 可以读取）
  • 备库 IO 能力：WAL 复制会把相同计算量带到备库，确保备库 IO 有足够余量
  • 监控进度：通过查询 current_setting('data_checksums') 确认状态，inprogress-on 期间持续监控 IO 和 CPU 指标
