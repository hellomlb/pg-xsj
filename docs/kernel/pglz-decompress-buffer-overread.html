pglz_decompress 缓冲区越界：先读后检查，三字节的差距

✅ SQL 验证通过 2026-06-01 内核研究 Bug pglz TOAST 缓冲区溢出 模糊测试 ASAN PG 16.14

概述

PG 16.14 修复了 pglz_decompress() 的一个堆缓冲区溢出：解码 match tag 时先读取 2~3 字节，再检查剩余数据是否足够。如果压缩数据恰好在 tag 中间被截断，读取操作已经越界。

版本 行为 后果
PG 16.11（有 Bug） 先读 2~3 字节，再检查是否越界 越界读后再返回 -1，极端情况崩溃
PG 16.14（已修复） 先检查是否有足够数据，再读取 安全返回 -1

受影响版本：PG 14~18 全部主版本。

现象

pglz 是 PG 的 TOAST 压缩算法，用于压缩存储大字段（TEXT、BYTEA、JSONB 等）。其压缩块由控制字节和一系列 tag 组成：

tag 类型 控制位 占用 说明
Literal 0 1 字节 直接输出
Match tag 1 2~3 字节 len(低4位) + off(高4位+下一字节)，len≥18 需第 3 字节扩展

当压缩数据在 match tag 的第 2 或第 3 字节处被截断，原有代码会先读后检查——此时已经越界。

用 C 测试程序验证（链接 PG 16.11 的 libpgcommon.a 编译运行）：

// 构造被截断的 match tag（控制字节 0x80，后续无数据）
unsigned char corrupt[] = {0x80};  // bit7=1 → 需要 2 字节 match tag，但只有控制字节
char dest[8192];

int rc = pglz_decompress(corrupt, 1, dest, sizeof(dest), false);
// 16.11: rc=0（返回前已越界读取了 sp[0]、sp[1]）
// 16.14: rc=-1（sp+2 > srcend 检查触发，未读取）

如果编译时加上 -fsanitize=address，16.11 版本下 ASAN 会捕获到堆缓冲区越界读，16.14 则干净运行。

根因分析（源码层面）

Bug 出在 src/common/pg_lzcompress.c 的 pglz_decompress() 函数，仅 1 个文件，+19/-8 行。

修复前（PG 16.11）

// match tag 解码路径
len = (sp[0] & 0x0f) + 3;
off = ((sp[0] & 0xf0) << 4) | sp[1];
sp += 2;
if (len == 18)
    len += *sp++;           ← 没有检查 sp 是否越界！
// 边界检查在读取之后
if (unlikely(sp > srcend || off == 0 || ...))  ← 已经读完了才检查

修复后（PG 16.14）

// 读取前先检查
if (unlikely(sp + 2 > srcend))
    return -1;              ← ★★★ 核心修复：先检查再读取

len = (sp[0] & 0x0f) + 3;
off = ((sp[0] & 0xf0) << 4) | sp[1];
sp += 2;
if (len == 18)
{
    if (unlikely(sp >= srcend))
        return -1;          ← ★★★ 扩展长度读取前也检查
    len += *sp++;
}

// 移除旧的 sp > srcend 检查（已被前置检查覆盖）
if (unlikely(off == 0 || off > (dp - (unsigned char *) dest)))
    return -1;

三层根因：

层次 原因
直接原因 match tag 的 2~3 字节在读取前没有边界检查
深层原因 边界检查（sp > srcend）放在读取操作之后，先使用再检查
根本原因 违反安全编程基本原则——"先检查再使用"。任何从不可信来源读取的数据（此处为压缩数据），必须先验证长度再访问

为什么 12 个版本都没发现

发现方式：libFuzzer + AddressSanitizer 模糊测试。

这种"先读后检查"模式的 Bug，正常数据流永远不会触发——因为正常写入的压缩数据格式是正确的，不会在 tag 中间截断。只有被精确截断的恶意数据才会命中。这也是代码审查难以发现的原因——代码看起来逻辑是对的，只是顺序反了。

影响评估

维度 评估
危害类型 堆缓冲区越界读（buffer overread），极端情况崩溃
触发条件 pglz 压缩数据在 match tag 的第 2 或第 3 字节处被截断
影响范围 PG 14~18 全部主版本
实际风险 TOAST 解压是 pglz_decompress 主要调用路径，任何存储为大对象的字段（TEXT、BYTEA、JSONB 等）在极端截断时都可能触发。但需要压缩数据被精确截断，真实生产环境几乎不可能自然发生
修复版本 16.14 / 15.18 / 17.10 / 18.4
修复规模 1 个文件，+19/-8 行
发现方式 libFuzzer + AddressSanitizer 模糊测试

建议

  • 升级到修复版本：虽然生产环境中极难自然触发，但作为内存安全修复，建议随版本升级一并合入
  • 扩展开发者：如果你的扩展处理用户提供的压缩/编码数据，注意"先检查再使用"原则
  • 安全测试：对处理外部输入的解压/解码函数，建议用 libFuzzer + ASAN 做模糊测试覆盖
