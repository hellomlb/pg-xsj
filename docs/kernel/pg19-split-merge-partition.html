PG19 原生分区手术刀：SPLIT PARTITION / MERGE PARTITIONS 语法与实战

2026-06-29 内核研究 PG19 Partition SPLIT MERGE DDL
✅ 已验证版 · PG 19 Beta1 实测

一句话总结

PG19 新增 SPLIT PARTITION 和 MERGE PARTITIONS 两条 DDL，数据自动迁移、索引自动重建、约束自动调整——从 5 步手工操作变成 1 条 SQL，且安全边界完善（不连续、重叠、范围不足全部被拒绝）。

旧方法痛点（PRE-PG19）

在 PG18 及更早版本，要把一个分区拆成两个，你需要：

步骤操作风险
1CREATE TABLE new_part ...范围定义出错 → 数据无法插入
2ALTER TABLE ... DETACH PARTITION old_partDETACH 持有排他锁 → 阻塞所有读写
3ALTER TABLE ... ATTACH PARTITION new_partATTACH 验证扫描 → 大分区耗时长
4INSERT INTO new_part SELECT ... WHERE ...手动迁移数据 → 可能漏行或重行
5DROP TABLE old_part误删 → 数据丢失

5 步操作，每一步都可能出错，且 DETACH/ATTACH 需要排他锁。PG19 的 SPLIT/MERGE 把这 5 步封装成 1 条 SQL。

SPLIT PARTITION 语法与实战

1. 常规 SPLIT（1 → 2）

准备测试表和数据：

-- 创建分区表
CREATE TABLE orders (
    id serial,
    order_date date NOT NULL,
    amount numeric(10,2),
    region text
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
CREATE TABLE orders_2025 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
CREATE TABLE orders_2026 PARTITION OF orders
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
CREATE TABLE orders_default PARTITION OF orders DEFAULT;

-- 插入 505 行数据
INSERT INTO orders (order_date, amount, region)
SELECT
    '2025-01-01'::date + (mod(i,365)::int) * '1 day'::interval,
    (random() * 1000)::numeric(10,2),
    CASE mod(i, 4)
        WHEN 0 THEN 'East'
        WHEN 1 THEN 'West'
        WHEN 2 THEN 'North'
        ELSE 'South'
    END
FROM generate_series(1, 500) i;

-- 2024、2026 各几行
INSERT INTO orders (order_date, amount, region) VALUES
    ('2024-06-15', 150, 'East'),
    ('2024-08-20', 250, 'West'),
    ('2024-03-10', 99, 'North'),
    ('2026-02-10', 800, 'South'),
    ('2026-07-01', 900, 'East');

一条 SQL 拆分 2025 年为上半年和下半年：

ALTER TABLE orders SPLIT PARTITION orders_2025 INTO (
    PARTITION orders_2025_h1 FOR VALUES FROM ('2025-01-01') TO ('2025-07-01'),
    PARTITION orders_2025_h2 FOR VALUES FROM ('2025-07-01') TO ('2026-01-01')
);

500 行自动拆分，无需手工 INSERT。

2. DEFAULT 分区 SPLIT（数据自动迁移）

DEFAULT 分区是最常见的 SPLIT 场景——新年度数据进入 DEFAULT 后，需要拆出专用分区。

-- 先建一个只有 2024 + DEFAULT 的分区表
CREATE TABLE orders2 (
    id serial, order_date date NOT NULL, amount numeric(10,2)
) PARTITION BY RANGE (order_date);

CREATE TABLE orders2_2024 PARTITION OF orders2
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
CREATE TABLE orders2_default PARTITION OF orders2 DEFAULT;

-- 2025 数据落入 DEFAULT
INSERT INTO orders2 (order_date, amount) VALUES
    ('2024-06-15', 100),
    ('2024-08-20', 200),
    ('2025-03-10', 300),
    ('2025-07-22', 400),
    ('2025-11-05', 500);

拆分前数据分布——2025 年的数据全在 DEFAULT：

分区数据
orders2_20242024-06-15 ¥100, 2024-08-20 ¥200
orders2_default2025-03-10 ¥300, 2025-07-22 ¥400, 2025-11-05 ¥500

SPLIT DEFAULT 分区——注意最后一个分区必须显式写 DEFAULT 关键字：

ALTER TABLE orders2 SPLIT PARTITION orders2_default INTO (
    PARTITION orders2_2025 FOR VALUES FROM ('2025-01-01') TO ('2026-01-01'),
    PARTITION orders2_default DEFAULT   -- ⚠️ 必须写 DEFAULT，不能省略
);

拆分后——2025 年数据自动迁移到新分区：

分区数据
orders2_20242024-06-15 ¥100, 2024-08-20 ¥200
orders2_20252025-03-10 ¥300, 2025-07-22 ¥400, 2025-11-05 ¥500
orders2_default空（无不属于任何分区的数据）

关键语法提醒：SPLIT DEFAULT 分区时，如果想保留 DEFAULT 分区，最后一个子分区必须写 DEFAULT 关键字。如果只写分区名不写 DEFAULT，PG19 会报语法错误。

3. 一次 SPLIT 为 N 段

DEFAULT 分区可以一次拆成多个子分区——例如从 DEFAULT 拆出 4 个季度：

ALTER TABLE sales SPLIT PARTITION sales_default INTO (
    PARTITION sales_q1 FOR VALUES FROM ('2026-01-01') TO ('2026-04-01'),
    PARTITION sales_q2 FOR VALUES FROM ('2026-04-01') TO ('2026-07-01'),
    PARTITION sales_q3 FOR VALUES FROM ('2026-07-01') TO ('2026-10-01'),
    PARTITION sales_q4 FOR VALUES FROM ('2026-10-01') TO ('2027-01-01'),
    PARTITION sales_default DEFAULT
);

4 条数据自动按季度分布：

分区数据
sales_q12026-01-15 ¥100
sales_q22026-04-20 ¥200
sales_q32026-07-10 ¥300
sales_q42026-10-05 ¥400

MERGE PARTITIONS 语法与实战

合并两个分区

-- 将上半年 + 下半年合并回完整的 2025 分区
ALTER TABLE orders MERGE PARTITIONS (orders_2025_h1, orders_2025_h2)
    INTO orders_2025;

合并后数据分布恢复如初：

分区行数
orders_20243
orders_2025500
orders_20262

索引与约束行为

MERGE 后，原分区上的索引自动重建到合并分区上：

-- 在分区上建索引
CREATE INDEX mt_idx1 ON mt_p1 (x);
CREATE INDEX mt_idx2 ON mt_p2 (id);

-- MERGE
ALTER TABLE mt MERGE PARTITIONS (mt_p1, mt_p2) INTO mt_p1_p2;

合并后索引自动迁移：

合并前合并后
mt_p1 → mt_idx1mt_p1_p2 → mt_idx1（自动重建）
mt_p2 → mt_idx2mt_p1_p2 → mt_idx2（自动重建）

安全边界（错误场景实测）

PG19 对 SPLIT/MERGE 的安全校验非常严格。以下三种非法操作全部被拒绝：

❌ 不连续范围

ALTER TABLE bad SPLIT PARTITION bad_p1 INTO (
    PARTITION bad_a FOR VALUES FROM (1) TO (50),
    PARTITION bad_b FOR VALUES FROM (60) TO (100)
);
-- ERROR: cannot split to partition 'bad_b' together with partition 'bad_a'

1-50 和 60-100 有个 10 的间隙（50-60），SPLIT 拒绝。

❌ 重叠范围

ALTER TABLE bad SPLIT PARTITION bad_p1 INTO (
    PARTITION bad_a FOR VALUES FROM (1) TO (60),
    PARTITION bad_b FOR VALUES FROM (50) TO (100)
);
-- ERROR: cannot split to partition 'bad_b' together with partition 'bad_a'

1-60 和 50-100 有重叠区间（50-60），SPLIT 拒绝。

❌ 范围无法覆盖所有数据

-- 原分区范围 1-100，数据包含 x=90
ALTER TABLE t3 SPLIT PARTITION t3_p1 INTO (
    PARTITION t3_a FOR VALUES FROM (1) TO (50),
    PARTITION t3_b FOR VALUES FROM (50) TO (80)   -- 80 ≠ 100
);
-- ERROR: upper bound of partition 't3_b' is not equal to upper bound
--        of split partition 't3_p1'

SPLIT 后的子分区范围 必须精确覆盖 原分区的整个范围。上界不能少也不能多。

新旧方法对比

维度SPLIT/MERGE（PG19）DETACH/ATTACH（旧方法）

数据迁移自动，无漏行风险手动 INSERT，可能漏行/重行
索引重建自动手动重建或 ATTACH 时重建
约束调整自动手动检查

限制与注意事项

  • 必须同父表下：MERGE 的两个分区必须是同一个父表的直接子分区
  • 范围必须连续无重叠：SPLIT 的子分区范围必须精确覆盖原分区，无间隙无重叠
  • 仅 RANGE 分区：目前 SPLIT/MERGE 仅支持 RANGE 分区（LIST/HASH 暂不支持）
  • 排他锁：SPLIT/MERGE 是 DDL 操作，需要排他锁，大分区操作期间阻塞读写
  • DEFAULT 关键字：SPLIT DEFAULT 分区时，保留的 DEFAULT 子分区必须显式写 DEFAULT 关键字
  • 子分区名可复用：MERGE INTO 的目标分区名可以与原分区名相同（如 MERGE h1+h2 INTO p1）

总结

PG19 的 SPLIT/MERGE PARTITION 是分区管理的手术刀级改进：

  • 从 5 步手工操作 → 1 条 SQL
  • 数据自动迁移，无漏行风险
  • 索引自动重建，约束自动调整
  • 安全边界完善：不连续、重叠、范围不足全部被拒绝
  • DEFAULT 分区拆分是最常见的实战场景——新年度数据进入 DEFAULT 后，一条 SPLIT 即可拆出专用分区

迁移建议：升级到 PG19 后，所有分区拆分/合并操作优先使用 SPLIT/MERGE，不再使用 DETACH+ATTACH 手工流程。
