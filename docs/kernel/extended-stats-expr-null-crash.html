扩展统计信息表达式空指针崩溃：当 typanalyze 说"我做不到"

✅ SQL 验证通过 2026-06-01 内核研究 Bug 扩展统计信息 typanalyze 空指针 ANALYZE CREATE STATISTICS PG 16.14

概述

PG 16.14 修复了一个隐藏了 12 个版本的 Bug：对扩展类型的表达式创建扩展统计信息时，如果该类型的 typanalyze 回调无法生成有效统计信息，PostgreSQL 服务端会因空指针解引用而崩溃。

版本 行为 后果
PG 16.11（有 Bug） 执行 ANALYZE 时崩溃 服务端 PANIC，连接断开
PG 16.14（已修复） 输出 WARNING 后跳过 正常，统计信息标记为不可用

受影响版本：PG 14~18 全部主版本。自 PG 14（2021 年）引入表达式扩展统计信息起，到 2026 年 16.14 修复，已有 12 个 major 版本。

现象

先安装测试扩展 test_custom_types（定义了一个 int_custom 类型 + 两个有问题的 typanalyze 回调），然后执行以下 SQL：

-- 建表并写入 10000 行
CREATE EXTENSION test_custom_types;
CREATE TABLE test_table (id int, data int_custom);
INSERT INTO test_table SELECT i, i::text::int_custom
FROM generate_series(1, 10000) i;

-- 场景 1：typanalyze 返回 false
ALTER TYPE int_custom SET (ANALYZE = int_custom_typanalyze_false);
CREATE STATISTICS test_stats ON func_int_custom(data), (id) FROM test_table;
ANALYZE test_table;
-- 16.11 → PANIC（服务崩溃）
-- 16.14 → WARNING: statistics object could not be computed

-- 场景 2：typanalyze 返回 true 但 stats_valid=false
ALTER TYPE int_custom SET (ANALYZE = int_custom_typanalyze_invalid);
CREATE STATISTICS test_stats ON func_int_custom(data), (id) FROM test_table;
ANALYZE test_table;
-- 16.11 → PANIC（服务崩溃）
-- 16.14 → 正常

根因分析（源码层面）

Bug 出在 src/backend/statistics/extended_stats.c 和 src/backend/utils/adt/selfuncs.c，共两处缺陷 + 一处连带修复。

缺陷 1：lookup_var_attr_stats() 未检查 NULL 返回值

当 ANALYZE 为扩展统计对象中的表达式构建统计信息时，会调用 examine_attribute()。如果自定义类型的 typanalyze 回调无法分析（返回 false 或 stats_valid=false），该函数返回 NULL。但调用方直接使用返回值，不做检查：

// extended_stats.c — 修复前（PG 16.11）
stats[i] = examine_attribute(expr);
// 后续代码直接用 stats[i] → 空指针解引用 → 崩溃

// 修复后（PG 16.14）— commit 530b6b02
stats[i] = examine_attribute(expr);
if (stats[i] == NULL)
{
    pfree(stats);
    return NULL;  // 分析不了就放弃，不崩溃
}

缺陷 2：statext_expressions_load() 未处理 NULL 数组条目

当从 pg_statistic 系统表加载已存储的表达式统计信息时，如果某个表达式的统计信息为空（NULL 条目），代码直接尝试从 NULL 数据构建 HeapTuple：

// extended_stats.c — 修复前
td = DatumGetHeapTupleHeader(eah->dvalues[idx]);  // dvalues[idx] 可能是 NULL

// 修复后
if (eah->dnulls && eah->dnulls[idx])
{
    ReleaseSysCache(htup);
    return NULL;
}

连带修复：examine_variable()

在 selfuncs.c 中，statext_expressions_load() 返回 NULL 后，examine_variable() 无条件设置了 freefunc = ReleaseDummy，这在 statsTuple 为 NULL 时会导致问题。修复后加了一层判断：

// selfuncs.c — 修复前
vardata->freefunc = ReleaseDummy;  // 即使 statsTuple 是 NULL 也设

// 修复后
if (vardata->statsTuple != NULL)
{
    vardata->freefunc = ReleaseDummy;
}

三层根因：

层次 原因
直接原因 examine_attribute() 返回 NULL 时未检查，导致空指针解引用
深层原因 pg_statistic 数组中的 NULL 条目未被 statext_expressions_load() 处理
根本原因 PG 14 引入表达式扩展统计信息时（commit a4d75c86bf15），未对 typanalyze 回调的所有返回值路径做 NULL 检查。假设所有 typanalyze 都会返回有效数据，但扩展类型的回调可能不遵守这个契约

为什么 12 个版本都没发现

核心原因：没有 in-core 的 typanalyze 回调会触发这组缺陷。

PostgreSQL 自带的所有数据类型（int4、text、jsonb 等）的 typanalyze 实现都正确返回了有效统计信息。只有第三方扩展定义的自定义类型，如果其 typanalyze 实现有问题，才会走进这条空指针路径。这也暴露了 core 代码对扩展行为的一个隐藏假设。

影响评估

维度 评估
危害类型 服务崩溃（PANIC）
触发条件 自定义数据类型 + typanalyze 返回 false/stats_valid=false + CREATE STATISTICS + ANALYZE
影响范围 PG 14~18 全部主版本（需自定义扩展类型触发）
修复版本 16.14 / 15.18 / 17.10 / 18.4
升级风险 无 — 修复只增加 NULL 检查，正常路径不变
隐蔽性 ⭐⭐⭐ — 生产环境几乎不会触发（无 in-core 类型会走到此路径），但扩展开发者可能踩到
扩展生态意义 core 代码对扩展类型的 typanalyze 行为缺乏防御性编程，此修复对第三方扩展开发者释放了一个长期隐藏的坑

建议

  • 普通用户：不受影响，标准类型的 typanalyze 行为正确
  • 扩展开发者：注意 typanalyze 回调的返回值契约——返回 false 或 stats_valid=false 是合法行为，core 代码现在能正确处理
  • 建议升级：虽然生产环境几乎不会触发，但这是潜在的安全隐患，建议随版本升级一并修复
