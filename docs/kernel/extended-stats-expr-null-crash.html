扩展统计信息表达式空指针崩溃：当 typanalyze 说"我做不到"

✅ SQL 验证通过 2026-06-01 内核研究 Bug 扩展统计信息 typanalyze 空指针 ANALYZE PG 16.14

概述

PG 16.14 修复了一个隐蔽的空指针崩溃 Bug：当扩展统计信息作用于自定义数据类型的表达式，且该类型的 typanalyze 回调无法生成有效统计信息时，PostgreSQL 服务端崩溃。

这个 Bug 自 PG 14（2021 年）引入表达式扩展统计信息功能起就已存在，直到 2026 年才被 Michael Paquier 修复。

维度 内容
现象 ANALYZE 在处理扩展统计信息时因空指针解引用而崩溃
触发条件 自定义类型 + typanalyze 返回 false 或 stats_valid=false + CREATE STATISTICS ON expr + ANALYZE
影响范围 PG 14~18 全部主版本（需自定义扩展类型触发，in-core 类型不受影响）
修复版本 16.14 / 15.18 / 17.10 / 18.4
隐蔽性 ⭐⭐⭐ — 生产环境几乎不会触发（没有 in-core 类型会走到这条路径），但扩展开发者可能踩到

一、问题定位

Michael Paquier 在分析中发现了两个独立的缺陷，都在表达式扩展统计信息的边界条件上。

缺陷 1：构建统计信息时未检查 NULL 返回

当 ANALYZE 为扩展统计对象中的表达式构建统计信息时，会调用 examine_attribute() 来获取表达式列的分析信息。如果自定义类型的 typanalyze 回调返回了 false（或虽然返回 true 但未设置有效统计信息），examine_attribute() 会返回 NULL。

但 lookup_var_attr_stats() 中直接使用这个返回值构建 MCV、ndistinct 和 dependencies 统计信息，没有做 NULL 检查：

// 有 Bug：直接使用 examine_attribute() 的返回值
stats[i] = examine_attribute(expr);
// 后续代码未经检查就用 stats[i] 构建统计信息 → 空指针解引用

修复很简单——加一行 NULL 检查：

stats[i] = examine_attribute(expr);
if (stats[i] == NULL)
{
    pfree(stats);
    return NULL;  // 无法分析，及早放弃
}

缺陷 2：加载统计信息时未处理 NULL 条目

当从系统表加载已存储的扩展统计信息时，statext_expressions_load() 从 pg_statistic 数组中读取表达式统计信息。如果某个表达式的统计信息因为收集失败而在数组中是 NULL 条目，代码会尝试从 NULL 数据中构建 HeapTuple，导致错误或崩溃。

// 修复前：未检查 NULL 条目
td = DatumGetHeapTupleHeader(eah->dvalues[idx]);

// 修复后：先检查再使用
if (eah->dnulls && eah->dnulls[idx])
{
    ReleaseSysCache(htup);
    return NULL;  // 无统计数据，告知调用方
}

连带修复了 examine_variable()——只有在 statsTuple != NULL 时才设置 freefunc。

二、为什么 12 个版本都没发现

核心原因：没有 in-core 的 typanalyze 回调会触发这组缺陷。

PostgreSQL 自带的所有数据类型（int4、text、jsonb 等）的 typanalyze 实现都正确返回了有效统计信息。只有第三方扩展定义的自定义类型，如果其 typanalyze 实现有问题，才会走进这条空指针路径。

这也是扩展生态的安全隐患——core 代码假设所有 typanalyze 都会返回有效数据，但扩展类型不一定遵守这个契约。

三、验证方式：编译测试扩展 + 双版本运行

由于没有 in-core 类型能触发，验证需要编写一个自定义 C 扩展，定义一个有"问题"的 typanalyze 回调。

从邮件列表 Patch 中提取的 test_custom_types 扩展定义了 int_custom 类型，提供两个 buggy 回调：

回调 行为 模拟场景
int_custom_typanalyze_false 直接返回 false 类型完全拒绝被分析
int_custom_typanalyze_invalid 返回 true 但设置 stats_valid=false 假装分析成功但实际无效

在两个版本上分别编译安装该扩展，创建测试表并 ANALYZE：

场景 PG 16.11（修复前） PG 16.14（修复后）
typanalyze 返回 false ❌ 服务崩溃 ✅ WARNING + 正常
typanalyze 返回无效数据 ❌ 服务崩溃 ✅ 正常

四、费曼解释

想象你让一个质检员检查一批零件。质检员有一个登记表，上面列了每个零件的检查结果。

原来的代码：质检员在登记表上为每个零件预留位置。但遇到一个"无法检查"的零件时，他不在登记表上做任何标记，而是直接在空的格子位置上盖章、签字、写分析报告——实际上他是在对着一张白纸操作。下一步工序看到这份"检查报告"，以为零件已经分析过了，直接用这个假数据做决策，整个流程就崩溃了。

修复后的代码：质检员看到"无法检查"的零件，直接在登记表上写"此零件无法分析，跳过"。后续工序知道这条记录无效，安全地跳过它。

一句话总结：调用函数拿到 NULL 却不检查，等于默认"所有数据类型都老老实实告诉我统计信息"——扩展类型可不一定。

五、影响与建议

  • 普通用户：不受影响。标准 PostgreSQL 类型的 typanalyze 行为正确，不会触发此 Bug。
  • 扩展开发者：需要注意 typanalyze 回调的返回值契约——返回 false 或 stats_valid=false 时，core 代码已安全处理（升级到修复版本后）。
  • 建议升级：虽然生产环境几乎不会触发，但这是潜在的安全隐患，建议随版本升级一并修复。
