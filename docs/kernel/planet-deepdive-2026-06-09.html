一、pg_background 2.0：自治事务扩展的全面重构

原文： Vibhor Kumar | 2026-06-08

pg_background 通过后台工作者进程实现自治事务（Autonomous Transaction），是社区最常用的解决方案之一。2.0 是该扩展发布以来最大的一次重构。

API 变更全景

核心逻辑：_v2 后缀退役，成为弃用别名（3.0 移除）。

1.x 旧名2.0 正式名破坏性
pg_background_launch_v2(...)pg_background_launch(...)低（别名保留）
pg_background_result_v2(...)pg_background_result(...)低
pg_background_wait_v2(...)pg_background_wait(...)低
pg_background_cancel_v2(...)pg_background_cancel(...)低
pg_background_detach_v2(...)pg_background_detach(...)低

硬重命名（无别名，立即失效）：

  • pg_background_progress(pct, msg) → pg_background_report_progress(pct, msg)
  • 类型 pg_background_progress → pg_background_progress_info
  • pg_background_grant_privileges_v2(...) → pg_background_grant_privileges(...)

V1 API 彻底删除：返回裸 int4 PID 的旧函数不再存在，必须使用 cookie 保护的 handle 版本。

⚠️ 最危险的语义变更

timeout_ms = 0 的含义彻底改变：

  • 1.x：timeout_ms = 0 → 等待默认时长
  • 2.0：timeout_ms = 0 → 无限阻塞（匹配旧版无参数版本的默认行为）

这是升级中最容易被忽略的陷阱。1.x 代码中 timeout_ms => 0 有实际超时行为，升级后变为永不超时。非阻塞检查应使用 timeout_ms => 1。

同时：pg_background_wait 返回类型从 void 变更为 bool（true=完成，false=超时）。

新增 Tier A 便捷函数

函数用途典型场景
run_query(sql, queue_size, timeout_ms, label, col_def)同步 launch→wait→result→detach，返回数据行最常用入口，替代手动四步
drain(handles, timeout_ms)批量等待所有 handle 完成批量异步任务收集
wait_any(handles, timeout_ms)返回第一个完成的 handle事件驱动模式
cancel_by_label(pattern, grace_ms)按 SQL LIKE 模式匹配标签取消批量运维取消
purge()仅 detach 已停止的 worker清理泄漏 worker

安全模型变化

  • PUBLIC 默认无访问：升级后必须显式 GRANT pgbackground_role TO app_role;
  • SECURITY DEFINER 函数：固定 search_path = pg_catalog，防搜索路径劫持

复合类型新增字段

  • pg_background_stats：新增 workers_timed_out（超时取消计数）
  • pg_background_result_info：新增 started_at、finished_at（执行时间范围）
  • pg_background_error：新增 schema_name/table_name/column_name/constraint_name（错误上下文细化）

内部实现改进

  • C 端错误处理修复：EmitErrorReport → AbortCurrentTransaction → FlushErrorState 重排序，防止 abort 在空错误栈上触发
  • SQLSTATE 传播修正：08006 仅保留给基础设施级故障，正常 SQL 错误通过真实 SQLSTATE 传播
  • 并发 cancel 已知问题：cancel_by_label 匹配 ≥2 个 worker 且 grace_ms > 0 时偶发 SIGSEGV，源于 PG 上游 bgworker 启动路径。缓解方案：用 cancel(pid, cookie, 0) 立即 SIGTERM，或逐个取消

升级风险矩阵

风险等级项目影响
🔴 高timeout_ms=0 语义变更可能导致永久阻塞
🔴 高4 个硬重命名标识符代码编译失败
🟡 中wait 返回类型 void→bool依赖返回值的代码异常
🟡 中V1 API 删除使用裸 PID 的代码失效
🟢 低_v2 后缀弃用2.x 仍可用，3.0 移除

白话总结

pg_background 2.0 就像是给"后台跑 SQL"这个功能做了一次大扫除：把以前乱起的名字（_v2 后缀）统一了，把危险的旧门（V1 API）锁了，加了几扇新窗户（drain/wait_any/purge），还贴了个告示牌提醒你"timeout_ms=0 现在是无限等了别搞混"。

二、跨数据库优化器统计对比：7 大数据库横向拆解

原文： Christophe Pettus | 2026-06-08

所有基于成本的优化器解决同一个问题：估计查询的基数。差异在三个维度：

  1. 统计数据的来源（采样/扫描/存储自带）
  2. 统计数据允许的陈旧程度（实时/15分钟/手动）
  3. 可供选择的计划形态（NL/Merge/Hash）

各系统统计哲学

Oracle —— 极繁主义者：

"凡论文中出现过的统计想法，Oracle 都发布过某个版本。" AutoTask 夜间 + 15 分钟高频自动收集，HyperLogLog 草图估算 NDV，Extended Statistics 多列统计，SQL Plan Directives 跨语句持久注释。OPTIMIZER_ADAPTIVE_STATISTICS 默认 FALSE——更多统计 ≠ 更好的计划。

Db2 —— 会编造数字的优化器：

独有武器 AUTO_STMT_STATS：编译查询时当场采样，用新数据做计划——PG 没有对应物。查询优化深度旋钮 0-9（PG 对应 geqo_threshold）。统计视图（Statistical Views）在连接或过滤子集上定义，PG 也没有对应物。

MySQL —— 被拖着前进的极简主义者：

统计沿索引边界分裂为两部分：InnoDB 索引统计（采样 20 页，10% 行变化自动重算）+ 列直方图（永远手动）。两个致命限制：直方图不会自动维护，优化器基本忽略有索引列上的直方图。

SQLite —— ANALYZE，手动执行：

与 PG 关键字相同，但背后几乎一切不匹配。最独特的做法：在开发期用代表性数据运行 ANALYZE，将 sqlite_stat1 行作为 INSERT 语句硬编码进应用程序二进制。这是任何其他数据库都不会以这种方式表述的问题。

DuckDB —— 统计作为存储的副产品：

Row group 元数据在写入时自动记录 min/max + null 计数 + 近似 NDV。最聪明的部分：知道 t1.a ∈ [25, 50]，join t1.a = t2.a → 合成 t2.a >= 25 AND t2.a <= 50 并下推到扫描，跳过不可能匹配的 row group。

Snowflake —— 没有 ANALYZE：

"统计信息永不过时，原因与照片相对于拍摄时刻永不过时相同——存储即统计。" 微分区 50-500MB 不可变列式文件，写入时自动记录元数据。剪枝才是核心——比 join 算法选择更重要。

维度OracleDb2MySQLSQLiteDuckDBSnowflakePG

PostgreSQL 定位：保守中间地带

采样收集 + autoanalyze + MCV/直方图，但不会自动检测多列相关性。

排查优先级（Pettus 原话）：

  1. 统计过期 → ANALYZE 或调紧 autovacuum_analyze_scale_factor
  2. 列独立性假设失效 → CREATE STATISTICS stats_name ON (col1, col2) FROM table_name

PostgreSQL 不会自动收集多列统计，也不会在无数据时编造行数。但它会精确执行你告诉它做的事——长期来看，可控性是保值最好的特性。

三、data_sync_retry：fsyncgate 的遗产

原文： Christophe Pettus | 2026-06-08

PostgreSQL 原以为的契约

  • fsync() 返回成功 → 数据已落盘，WAL 可回收
  • fsync() 返回失败 → 脏页仍在缓存中，可重试

Linux 内核实际行为

写入失败 → 脏页标记为 clean → 记录错误 → 下次 fsync() 报告（仅一次）→ 错误被遗忘 → 页面被回收重用

内核的逻辑：永远无法回收的脏页 = 无界内存泄漏 → 不可接受 → 丢弃是唯一选择。

完整的灾难场景

  1. 检查点运行 → 写回错误发生
  2. 错误被报告给错误的文件描述符 → 被遗忘
  3. 脏页被标记为 clean → 被回收重用
  4. checkpointer 调用 fsync() → 返回"成功"
  5. 检查点被认为完成 → WAL 段被回收
  6. 数据"从已知宇宙中消失"（Andrew Gierth）

时间线

  • 2018 年初：Craig Ringer 将问题带到 pgsql-hackers
  • PgCon 2018：Linux 内核开发者 Matthew Wilcox 演讲（座无虚席）。英国式致歉："我们为此带来的不便致歉"引发全场哄笑
  • 2018年11月（Thomas Munro）：引入 data_sync_retry，默认 off → PANIC
  • 反向移植到所有受支持版本
  • PG 12：额外改进，持续持有脏数据文件的 fd，但未反向移植

data_sync_retry 参数

值行为安全性
off（默认）fsync() 失败 → PANIC → WAL 重放恢复✅ 安全
onfsync() 失败 → 重试 → 继续运行🔴 Linux 上=静默丢数据

官方文档原话：在 Linux 上开启此选项 = "请静默地丢失我的数据，而不是大张旗鼓地丢失"。

残留风险窗口

即使 data_sync_retry = off，仍有窄窗口：文件被关闭→重新打开前→inode 被逐出→错误记录消失。

PG12 缓解：持续持有 fd + 传递给 checkpointer → 避免"关闭→重打开"窗口。

Linux 4.13 errseq_t 修补的局限性：

  1. 数据回不来——可靠报告的错误仍是丢失数据的错误
  2. inode 逐出时错误记录仍然丢失
  3. 这是报告层修补，不是语义层修补

终极方案：Direct I/O

对主数据路径使用 Direct I/O，绕过页面缓存，让数据库自己掌控写回和错误处理。Andres Freund 评价："a metric ton of work"。截至 2026 年仍未完全落地。

共享文件系统特殊风险

NFS/CIFS 上：客户端报告 fsync 成功，但服务端未持久化。多层错误报告路径更易丢。PANIC 是重要信号——文件系统在"说谎"。

白话总结

想象你给银行存了一箱金条，银行说"帮你锁进金库"（fsync 成功）。结果金库门坏了，金条被搬到了走廊（脏页标记为 clean），银行还给你发了张"已安全存入"的收据（重试 fsync 返回成功）。你拿着收据走了，但金条其实已经被别人搬走了（页面被回收）。data_sync_retry=off 就是：金库门一坏就拉警报（PANIC），把所有人叫回来从账本（WAL）重新对账。虽然麻烦，但至少不会让你以为金条还在但其实已经没了。

三篇文章的哲学关联

文章核心哲学
pg_background 2.0宁可让你看到问题，也不静默掩盖：V1 裸 PID 设计隐藏风险 → 2.0 强制 cookie + 撤销 PUBLIC
跨数据库统计保守但可控：不会自作主张收集多列统计，也不会编造数字。说多少，做多少。
data_sync_retryPANIC 是安全网，不是 Bug：知道数据丢了比假装没丢强一万倍。

三条线汇聚到一个设计哲学：PostgreSQL 倾向于让你知道问题存在（PANIC/手动统计/显式API），而不是静默地掩盖它。这种"宁可信其有"的设计哲学，是 PG 作为严肃数据库系统的核心价值。
