PG19 Property Graph 实战：在关系表上声明式叠图，以及三个必须知道的坑

2026-07-10 图查询 PG19 Property Graph SQL/PGQ GRAPH_TABLE 图数据库
✅ 已验证版 · PG 19 Beta1 实测（单版本：本特性为 PG19 独占）

一句话总结

PG19 把 Property Graph 作为一等公民塞进了 SQL：你不用搬数据、不用装图数据库，只要在现有外键表上用 CREATE PROPERTY GRAPH 声明"哪张表是顶点、哪张表是边"，就能用 GRAPH_TABLE(... MATCH ...) 写类 Cypher 的图查询。它编译成普通 JOIN，复用优化器/索引/统计——价值在表达力 + SQL/PGQ 标准兼容，不是性能。但有三个真实限制必须知道，否则上线就踩。

背景：Property Graph 到底是什么

Property Graph 是一种图数据模型：节点（vertex）带 key-value 属性，边（edge）有方向、也带属性。它实现的是 ISO 的 SQL/PGQ（Property Graph Queries） 标准，不是 PG 私有语法。

关键认知——它不是"另起炉灶的图数据库"：

  • Overlay（叠加层）：不搬数据，只在已有表上声明。维度表→顶点，纯连接表→边，事实表→顶点+别名边。
  • 声明式：不用手写 JOIN 链，写 MATCH (a)-[e]->(b) 描述"找什么"，PG 编译成 JOIN。
  • 标准对齐：SQL/PGQ 是跨库标准，写法可移植。

带回去想的问题：既然 property graph 最终编译成 JOIN、且变长路径还得退回递归 CTE，它的真正价值是"性能"还是"表达力 + 标准兼容"？想清楚这个，给客户讲的时候一句话就能戳中要害。

一、建表 + 建图 + 正确查询

准备员工/公司/任职三张表（任职表带主键，让 edge key 可由主键推断）：

CREATE TABLE pg_person  (id int primary key, name text, age int);
CREATE TABLE pg_company (id int primary key, cname text);
CREATE TABLE pg_works_at(person_id int, company_id int, since int,
                         primary key(person_id, company_id));
INSERT INTO pg_person  VALUES (1,'Alice',30),(2,'Bob',40);
INSERT INTO pg_company VALUES (10,'Acme'),(20,'Globex');
INSERT INTO pg_works_at VALUES (1,10,2020),(2,20,2019);

CREATE PROPERTY GRAPH pg_graph
  VERTEX TABLES (
    pg_person  KEY (id) LABEL person  PROPERTIES (id, name, age),
    pg_company KEY (id) LABEL company PROPERTIES (id, cname)
  )
  EDGE TABLES (
    pg_works_at
      SOURCE KEY (person_id) REFERENCES pg_person (id)
      DESTINATION KEY (company_id) REFERENCES pg_company (id)
      LABEL works_at
  );

查询谁在哪家公司、何时入职：

SELECT * FROM GRAPH_TABLE (pg_graph
  MATCH (p IS person)-[w IS works_at]->(c IS company)
  COLUMNS (p.name AS person, c.cname AS company, w.since AS since));

实测输出：

 person | company | since
--------+---------+-------
 Alice  | Acme    |  2020
 Bob    | Globex  |  2019
(2 rows)

关键语法点：EDGE TABLES 不需要 KEY 子句，只写 SOURCE KEY / DESTINATION KEY，edge key 由边表主键自动推断。给边表加主键后即无需显式 KEY。首次建图失败（报 no key specified）的真正原因是边表没建主键，不是特性限制。

二、坑一：KEY 列不自动可查 —— 但能修

建图时如果只写 KEY(id) 却没把 id 列进 PROPERTIES，那 id 就不能在 MATCH 里过滤/投影：

CREATE PROPERTY GRAPH pg_graph_k
  VERTEX TABLES (
    pg_person  KEY (id) LABEL person  PROPERTIES (name, age),  -- 故意不含 id
    pg_company KEY (id) LABEL company PROPERTIES (cname)
  )
  EDGE TABLES ( /* 同上 */ );

SELECT * FROM GRAPH_TABLE (pg_graph_k
  MATCH (p IS person WHERE p.id = 1) COLUMNS (p.name AS n));

实测报错：

ERROR:  property "id" does not exist

避坑方案（实测有效）：把 id 显式写进 PROPERTIES，WHERE p.id=1 立刻成功：

SELECT * FROM GRAPH_TABLE (pg_graph      -- 注意 pg_graph 的 person 已 PROPERTIES (id, name, age)
  MATCH (p IS person)-[w IS works_at]->(c IS company)
  WHERE p.id = 1
  COLUMNS (p.name AS person, c.cname AS company));

实测输出：

 person | company
--------+---------
 Alice  | Acme
(1 row)

白话：KEY 定义"边怎么连"，PROPERTIES 定义"能查什么"。想用某列做条件/投影，它就必须在 PROPERTIES 里——这是声明要求，不是缺陷。

三、坑二：变长路径不支持

想写"跟随 1~3 跳"这类变长路径？直接报错：

SELECT * FROM GRAPH_TABLE (pg_graph
  MATCH (p IS person)-[w IS works_at]*->(c IS company) COLUMNS (p.name AS n));

实测报错：

ERROR:  syntax error at or near "*->"
LINE 2:   MATCH (p IS person)-[w IS works_at]*->(c IS company) COLUMNS (p.name AS n));
                                             ^

PG19 在解析阶段就把量词 * / + 拒了。变通：手写每跳，或退回递归 CTE 做任意深度遍历。

四、坑三：同名列类型冲突

两张顶点表有同名列但不同类型，且都不写 PROPERTIES（PG19 默认把所有列当 property），直接冲突：

CREATE TABLE pg_v1(id int primary key, code int);
CREATE TABLE pg_v2(id int primary key, code text);
-- 边表略

CREATE PROPERTY GRAPH pg_graph_c
  VERTEX TABLES ( pg_v1 KEY (id) LABEL v1, pg_v2 KEY (id) LABEL v2 )
  EDGE TABLES ( /* 同上 */ );

SELECT * FROM GRAPH_TABLE (pg_graph_c
  MATCH (a IS v1), (b IS v2)
  COLUMNS (a.code AS ac, b.code AS bc));

实测报错：

ERROR:  property "code" data type mismatch: integer vs. text
DETAIL:  In a property graph, a property of the same name has to have the same data type in each label.

避坑：用 PROPERTIES (...) 白名单让两表暴露不同的 property 名（如 PROPERTIES (code AS v1_code)），互不干涉。或者统一列类型。

五、元数据：怎么看已建的图

⚠️ 纠正一个常见错误：目录视图不是 pg_property_graphs（查它会 relation "pg_property_graphs" does not exist）。正确名称是 information_schema.property_graphs，列名为 property_graph_name（information_schema 命名风格）：

SELECT property_graph_name FROM information_schema.property_graphs;

实测输出：

 property_graph_name
---------------------
 m_g
(1 row)

底层系统目录在 pg_catalog.pg_propgraph_* 一族，日常查看用上面那个 information_schema 视图即可。

七、给客户的一句话

  • Property Graph 是叠加层，不搬数据；语法编译成 JOIN，复用优化器/索引/统计。
  • 价值在表达力 + SQL/PGQ 标准兼容，不是性能——别当真图数据库吹。
  • 变长路径暂不支持，深图遍历退回递归 CTE。
  • KEY 列须显式进 PROPERTIES 才可查；同名列须同型或用 PROPERTIES 白名单隔离。
  • 适用版本：PG19+。升级前别在 16.x 上试，语法直接报错。
