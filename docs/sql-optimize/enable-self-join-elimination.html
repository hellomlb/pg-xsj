enable_self_join_elimination：主键自连接自动消除（PG18+ 性能特性）

2026-07-13 SQL 优化 self_join_elimination optimizer EXPLAIN PG18 performance
✅ 已验证版 · PG 19 Beta1 实测（PG18+ 特性，PG 16.x 无此 GUC，未做双版本功能验证）

一句话总结

ORM 或视图嵌套经常写出 FROM t a JOIN t b ON a.pk = b.pk 这种"自己和自己主键连"的冗余自连接。PG18 起，优化器能识别它并直接消除——既然主键等值是一对一的，连出来还是原来的行，那个多余的 b 表纯属浪费。PG 19 Beta1 实测：消除后 4.621 ms vs 保留时 21.689 ms，约 4.7x 加速。

白话：这种自连接从哪来？

你大概从没手写过 t a JOIN t b ON a.id = b.id。但它很常见，来源通常是：

  • ORM 懒加载 / 关联映射：框架为了"统一"地拼 SQL，把同一个实体 join 到自己一次。
  • 视图嵌套：视图 V1 基于表 T，视图 V2 又 JOIN 了 V1，而查询同时用 V1 和 V2 并通过主键关联——展开后就是 T 的自连接。
  • CTE / 子查询复用：同一份派生表被引用两次并通过主键相连。

这些自连接结果完全正确，但计划却做了两遍全表扫描再 Hash Join。在 PG17 及更早版本，优化器老老实实把它当真 join 执行。PG18 起多了个"看一眼就知道能省"的能力。

原理：什么条件下能消？

优化器只在「同一张基表自我出现」这一个强前提下沉掉自连接：连接条件是该表自身的主键 = 主键（或等价的唯一约束等值，例如同一表的两个别名、或视图展开后指向同一基表）。因为主键唯一，a.id = b.id 必然是一对一映射，join 结果集和 a 单独扫描的结果集完全一致——那另一侧就没存在意义，直接删掉。

关键闸门是「两侧解析到同一个基表（relid 相同）」，不是「两张表各自有主键」。所以两个不同基表即使都有主键、甚至加了外键保证覆盖，本特性也不消除——它走的是「自连接检测」这条专属路径，不接管「基于外键推论的跨表冗余 join 消除」（PG18/19 实测均不触发）。

条件是否消除原因
主键等值自连接 a.pk = b.pk（同表别名）✅ 消除一对一，结果无损
视图包装同表后自连接 a JOIN v(based_on_same_table)✅ 消除视图展开后仍是同一基表，名字不同无关
不同基表 PK=PK（无 FK）❌ 保留不同 relid，规划器不能假设覆盖，join 可能过滤掉行
不同基表 PK=PK（有 FK 保证覆盖）❌ 保留本特性只走自连接检测路径，不接管 FK 推论消除（PG18/19 实测）
非唯一列自连接 a.val = b.val❌ 保留可能一对多，删了会丢行
非等值 a.id > b.id❌ 保留不是恒等映射
三表主键自连接✅ 全消除逐对都是一对一

注意：这与 enable_seqscan 等"靠加惩罚成本逼优化器换路"的 GUC 不同。self join elimination 是真正重写查询树——Pettus 称它是 enable_* 家族里唯一做查询重写的成员。所以关掉它，计划会真的多出 join 节点，而不只是标个 Disabled。

边界实测：不同基表为什么不消（PG 19 Beta1 原样计划）

为确认「同基表」是闸门，准备两个不同基表 sj_t1 / sj_t2（各有 PK，数据域完全重叠），分别测「无 FK」「加 FK 保证覆盖」「视图包装同表」三种情况：

-- Q2: 不同基表 PK=PK，无 FK → 不消除（Hash Join 仍在）
EXPLAIN (COSTS OFF)
SELECT sj_t1.* FROM sj_t1 JOIN sj_t2 ON sj_t1.id = sj_t2.id;
--   Hash Join
--     Hash Cond: (sj_t1.id = sj_t2.id)
--     ->  Seq Scan on sj_t1
--     ->  Hash  ->  Seq Scan on sj_t2

-- Q3: 加 FK sj_t1.id -> sj_t2.id（理论已保证覆盖） → 仍然不消除
ALTER TABLE sj_t1 ADD CONSTRAINT fk_sj FOREIGN KEY (id) REFERENCES sj_t2(id);
EXPLAIN (COSTS OFF)
SELECT sj_t1.* FROM sj_t1 JOIN sj_t2 ON sj_t1.id = sj_t2.id;
--   Hash Join            ← 依旧在，本特性不接管 FK 推论消除
--     Hash Cond: (sj_t1.id = sj_t2.id)
--     ->  Seq Scan on sj_t1
--     ->  Hash  ->  Seq Scan on sj_t2

-- Q4: 视图包装同表，语法像"不同表" → 消除（视图展开后是同一基表）
CREATE VIEW v_sj AS SELECT * FROM sj_t1;
EXPLAIN (COSTS OFF)
SELECT a.* FROM sj_t1 a JOIN v_sj b ON a.id = b.id;
--   Seq Scan on sj_t1    ← 消除，名字不同但基表相同

Q3 最说明问题：外键已经在语义上证明 join 不丢行、且只选 sj_t1.* 不缺列，理论上完全冗余——但 PG18/19 的 enable_self_join_elimination 依然不消除。因为它只识别「自连接（同基表）」这种模式，跨表的冗余 join 消除是另一条尚未实现的路。

实测验证：测试表准备

DROP TABLE IF EXISTS sje_test;
CREATE TABLE sje_test (id int PRIMARY KEY, name text, val int);
INSERT INTO sje_test SELECT i, 'name_' || i, i * 10 FROM generate_series(1, 10000) i;
ANALYZE sje_test;

SHOW enable_self_join_elimination;   -- on（PG18+ 默认值）

验证结果（PG 19 Beta1）

场景 1：主键自连接（默认 on）→ 期望无 join 节点

EXPLAIN (COSTS OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id;

       QUERY PLAN
------------------------
 Seq Scan on sje_test b
(1 row)

✅ 只剩一个 Seq Scan，a 和 join 全没了。b 被消除，优化器直接从 b 取结果（选哪个别名作基表是实现的细节，不影响正确性）。

场景 2：SET enable_self_join_elimination = off → 期望出现 join 节点

SET enable_self_join_elimination = off;
EXPLAIN (COSTS OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id;

             QUERY PLAN
------------------------------------
 Hash Join
   Hash Cond: (a.id = b.id)
   ->  Seq Scan on sje_test a
   ->  Hash
         ->  Seq Scan on sje_test b
(5 rows)

✅ 关掉后立刻变回 Hash Join + 两次 Seq Scan。说明场景 1 的"单表扫描"确实是被这个特性吃掉 join 的结果。

场景 3：性能对比（on vs off）

-- ON（消除）：Execution Time 4.621 ms
EXPLAIN (ANALYZE, COSTS OFF, TIMING OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id;
 Seq Scan on sje_test b (actual rows=10000.00 loops=1)
   Buffers: shared hit=65 dirtied=1
 Planning Time: 0.196 ms
 Execution Time: 4.621 ms

-- OFF（保留 join）：Execution Time 21.689 ms
SET enable_self_join_elimination = off;
EXPLAIN (ANALYZE, COSTS OFF, TIMING OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id;
 Hash Join (actual rows=10000.00 loops=1)
   Hash Cond: (a.id = b.id)
   Buffers: shared hit=128
   ->  Seq Scan on sje_test a (actual rows=10000.00 loops=1)
         Buffers: shared hit=64
   ->  Hash (actual rows=10000.00 loops=1)
         Buckets: 16384  Batches: 1  Memory Usage: 480kB
         Buffers: shared hit=64
         ->  Seq Scan on sje_test b (actual rows=10000.00 loops=1)
               Buffers: shared hit=64
 Planning Time: 0.950 ms
 Execution Time: 21.689 ms

✅ 消除后省掉一遍全表扫描 + Hash 构建，4.621 ms → 21.689 ms，约 4.7x 加速。Pettus 引用 Dalibo 实测是 2.9x（530ms→185ms），我们的加速比更高是因为测试表更小、join 开销占比更大。

场景 4：非主键自连接（val 无唯一）→ 期望保留 join

EXPLAIN (COSTS OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.val = b.val;

             QUERY PLAN
------------------------------------
 Hash Join
   Hash Cond: (a.val = b.val)
   ->  Seq Scan on sje_test a
   ->  Hash
         ->  Seq Scan on sje_test b
(5 rows)

✅ val 没有唯一约束，优化器不敢消——消了可能丢行。正确行为。

场景 5：三表主键自连接 → 期望全部消除

EXPLAIN (COSTS OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id JOIN sje_test c ON b.id = c.id;

       QUERY PLAN
------------------------
 Seq Scan on sje_test c
(1 row)

✅ 两个 join 都被吃掉，只剩一个 Seq Scan。嵌套视图展开出的多层自连接也能一气呵成地消掉。

场景 6：主键自连接 + 额外过滤 → 仍消除

EXPLAIN (COSTS OFF) SELECT a.* FROM sje_test a JOIN sje_test b ON a.id = b.id WHERE a.val > 50000;

       QUERY PLAN
-------------------------
 Seq Scan on sje_test b
   Filter: (val > 50000)
(2 rows)

✅ 消除后过滤条件下推到那唯一的一次扫描上，逻辑不变。

版本边界（重要，诚实说明）

这是 PG18 引入的特性。我们在 PG 16.14 上验证该 GUC 根本不存在：

-- PG 16.14
SHOW enable_self_join_elimination;
ERROR:  unrecognized configuration parameter "enable_self_join_elimination"

因此本文只在 PG 19 Beta1 做功能验证，未做 PG 16.x 功能验证（它无此能力）。若你的生产库还在 PG16 或更早，这篇对你而言是"未来特性"，升级到 PG18+ 后会自动受益，无需改 SQL。

白话总结：要不要管它？

  • 默认就是 on，你什么都不用做。它只在"主键等值自连接"这种冗余场景下悄悄帮你省一遍扫描。
  • 结果永远正确。消除是恒等变换，不会改变返回行。
  • 诊断时怎么用？怀疑某条慢 SQL 被它影响（或想看"如果不消除会怎样"），SET enable_self_join_elimination = off 临时对比计划即可，看完记得 RESET。
  • 它救不了非唯一自连接。如果真是 a.val = b.val 这种有语义的 join，该留着就得留着——那是业务需要的，不是冗余。
