from_collapse_limit 深度实测：PG 16/19 里它为何很少被触发，真正管用的旋钮是谁

✅ 已验证版 · PG 16.11 + 19 Beta1 双版本实跑

一句话总结

from_collapse_limit 的官方定义是"规划器把 FROM 子查询/括号化连接折叠进父查询、形成一个可整体重排的 join 问题时，允许的最大关系数"，默认 8。但我在 PG 16.11 与 19 Beta1 上实跑验证后发现：对现代版本的日常查询，这个参数很少改变计划——FROM 子查询（聚合、LIMIT、DISTINCT、UNION 等）大多被上拉内联、扁平逗号列表永远合并（单成员），所以它"没机会"生效，而不是"无效"。真正控制连接重排、你调了能立刻看到计划变化的，是它兄弟 join_collapse_limit。

这不是文档写错，而是 PostgreSQL 规划器在 16/19 这代把"子查询上拉"做得更激进了——大多数 FROM 子查询在 pull_up_subqueries 阶段就被拉平，导致 from_collapse_limit 真正作用的 deconstruct_jointree 阶段很少再碰到"多成员子问题"。注意两个阶段的区别：能不能上拉由 pull_up_subqueries 的规则决定，上拉之后是否合并子问题才轮到 from_collapse_limit。下面用实测说话。

二、它到底管什么：源码层面

from_collapse_limit 的真实作用点在 src/backend/optimizer/plan/initsplan.c 的 deconstruct_jointree()，不是很多人以为的"子查询上拉"阶段。关键代码（PG 18.4 源码，16/19 同逻辑）：

/*
 * We collapse subproblems into a single joinlist whenever the resulting
 * joinlist wouldn't exceed from_collapse_limit members.  Also, always
 * collapse one-element subproblems, since that won't lengthen the
 * joinlist anyway.
 */
...
sub_members = list_length(sub_joinlist);
remaining--;
if (sub_members <= 1 ||
    list_length(joinlist) + sub_members + remaining <= from_collapse_limit)
    joinlist = list_concat(joinlist, sub_joinlist);
else
    joinlist = lappend(joinlist, sub_joinlist);

白话翻译：只有当一个"子问题"包含 >1 个关系、且并入父列表会超过 from_collapse_limit 时，规划器才把它单独钉住。反过来，sub_members <= 1（单关系子问题）永远合并——"反正不会拉长 join 列表"。

这两个事实决定了后面的所有现象。

三、实测一：9 表逗号列表，from_collapse_limit 8 vs 1 —— 计划居然完全一样

建 9 张表（t1..t9，链式外键），用逗号列表拼成 9 表连接：

EXPLAIN (COSTS OFF)
SELECT COUNT(*) FROM t1, t2, t3, t4, t5, t6, t7, t8, t9
WHERE t1.id=t2.t1_id AND t2.id=t3.t2_id AND t3.id=t4.t3_id
  AND t4.id=t5.t4_id AND t5.id=t6.t5_id AND t6.id=t7.t6_id
  AND t7.id=t8.t7_id AND t8.id=t9.t8_id AND t1.id=1;

分别设 from_collapse_limit = 8 和 = 1 执行。结果：16.11 与 19 Beta1 上，两种设置输出的计划逐行一致（都是 45 行，Nested Loop + Memoize 驱动 t1，其余 t2..t9 走 Merge/Hash Join）。

原因就是第二节的源码：逗号列表里每个表都是 1 个关系的"单成员子问题"，命中 sub_members <= 1 分支——无论 from_collapse_limit 是 8 还是 1，全部合并。对这种扁平逗号列表形态，参数确实没有调节空间。

四、实测二：9 表显式 JOIN，join_collapse_limit 8 vs 1 —— 真有差别

同样的 9 张表，改用显式 JOIN 语法（无括号）：

-- join_collapse_limit = 8（默认）
EXPLAIN (COSTS OFF)
SELECT COUNT(*) FROM t1
JOIN t2 ON t1.id=t2.t1_id JOIN t3 ON t2.id=t3.t2_id JOIN t4 ON t3.id=t4.t3_id
JOIN t5 ON t4.id=t5.t4_id JOIN t6 ON t5.id=t6.t5_id JOIN t7 ON t6.id=t7.t6_id
JOIN t8 ON t7.id=t8.t7_id JOIN t9 ON t8.id=t9.t8_id
WHERE t1.id=1;

join_collapse_limit = 8 的计划（节选关键结构）：

 Aggregate
   ->  Hash Join
         Hash Cond: (t9.t8_id = t8.id)
         ->  Seq Scan on t9
         ->  Hash
               ->  Nested Loop
                     ->  Index Only Scan using t1_pkey on t1   -- ★ t1 被重排到前面
                           Index Cond: (id = 1)
                     ->  Hash Join
                           Hash Cond: (t8.t7_id = t7.id)
                           ...

join_collapse_limit = 1 的计划（节选关键结构）：

 Aggregate
   ->  Hash Join
         Hash Cond: (t9.t8_id = t8.id)
         ->  Seq Scan on t9
         ->  Hash
               ->  Hash Join
                     Hash Cond: (t8.t7_id = t7.id)
                     ...（t9→t8→t7→…→t2 严格左深链式）
                           ->  Nested Loop
                                 ->  Index Only Scan using t1_pkey on t1  -- ★ t1 被钉在最底层
                                 ->  Seq Scan on t2

差别肉眼可见：= 8 时规划器把高选择性的 t1.id=1（1 行）重排到连接顺序靠前，尽早缩小中间结果；= 1 时显式 JOIN 严格按书写顺序左深连接，t1 被钉在最底层。两版本输出一致，且 = 1 的计划通常更差（不能利用 t1 的选择性提前裁剪）。

这就是关键结论：连接重排的活旋钮是 join_collapse_limit，不是 from_collapse_limit。前者管显式 JOIN 语法的重排，后者管 FROM 子查询/括号子连接的折叠——而后者在现代版本里因为入口闸门（子查询上拉）已处理掉大多数形态，很少再碰到"多成员子问题"这个触发条件。

五、实测三：FROM 子查询在 16/19 大多被上拉，但"无条件下拉"是错的

很多旧资料（包括 Pettus 的 GUC 系列）说"含聚合 / DISTINCT / LIMIT / 窗口函数的子查询是栅栏，不会被上拉"。我在 16.11 与 19 Beta1 上实测，大多数 FROM 子查询确实被上拉内联了——但说"无条件"是错的：pull_up_subqueries 在 16/19 更激进，却仍有明确的拦截条件（最确凿的是窗口函数，实测必保留为 Subquery Scan）。下面是逐形态实测：

子查询形态（顶层 FROM）PG 16.11 / 19 Beta1 实测是否上拉
简单投影 SELECT id,v FROM t2无 Subquery Scan上拉
聚合 GROUP BY ... count(*)内联为 HashAggregate上拉
OFFSET 0（无外层 WHERE）无 Subquery Scan（旧技巧失效）上拉
OFFSET 0（带外层 WHERE）子查询存活为 Subquery Scan，谓词下推为 Filter拦截（保留）
UNION上拉为 Append上拉
普通 LIMIT（无 OFFSET 0）Limit 节点被合并上拉
普通 DISTINCT / 嵌套 DISTINCT两层 HashAggregate，无 Subquery Scan上拉
DISTINCT ON (...)下推为 Unique → Sort上拉
窗口函数 row_number() OVER(...)保留为 Subquery Scan拦截（保留）

注意上表后半部分：在 16/19 实测中，窗口函数子查询必然作为 Subquery Scan 存活；而经典的 OFFSET 0 "栅栏"技巧取决于外层是否有谓词——无外层 WHERE 时被移除（旧技巧失效），带外层 WHERE 时仍存活为 Subquery Scan 屏障（谓词仅下推为 Filter，无法改写子查询）。但这些被拦截存活的子查询折叠后仍是单个 RTE（1 个成员），命中 sub_members <= 1，所以 from_collapse_limit 照样拦不住——我实测 5 表窗口子查询 + 高选择性外表，= 8 与 = 1 计划仍完全一致。换句话说：拦截上拉 ≠ 触发 from_collapse_limit，二者是不同阶段的门槛。

脚注（与老木对齐）：老木提醒"某些嵌套 DISTINCT 形态仍可能保留子问题"。但本实测在 16.11 与 19 Beta1 上，普通嵌套 DISTINCT（验证 3：SELECT DISTINCT a FROM (SELECT DISTINCT a ...)）也被上拉为两层 HashAggregate，未形成 Subquery Scan；DISTINCT ON（补充验证）同样被下推为 Unique → Sort。可见 16/19 对 DISTINCT 的放宽比旧资料描述的更彻底——窗口函数是目前实测唯一确凿的拦截点。对于含不可下推聚合/集合操作等更特殊的子查询，仍可能保留，但那已超出日常 SQL 范畴。此外，老木复核指出经典 OFFSET 0 栅栏并非无条件移除——本实测 A/B 对照确认：无外层谓词时子查询被移除（旧技巧失效），带外层 WHERE 时仍保留 Subquery Scan 屏障、谓词仅下推为 Filter。上表已据实测拆分为"无/有外层 WHERE"两行。

六、那 from_collapse_limit 到底什么时候会"咬人"？概念澄清

先厘清容易被混为一谈的两个阶段：

  • 阶段一 · 入口闸门 pull_up_subqueries：决定一个 FROM 子查询能不能被"拉平"进父查询。这一步在 16/19 非常激进，多数子查询（聚合、LIMIT、DISTINCT、UNION、无外层谓词的 OFFSET 0 等）都被上拉，于是它们根本不会以"子问题"形态存活到下一阶段。
  • 阶段二 · deconstruct_jointree（from_collapse_limit 作用点）：对已经存活的子问题，决定是否把它并入父 joinlist 成为可整体重排的 join 问题。from_collapse_limit 只在这里起作用。

所以老木的审阅点很准：from_collapse_limit 不是"休眠"或"无效"，而是入口闸门在现代 PG 里已经把大多数子查询上拉掉了，导致它很少有机会碰到"多成员子问题"。它该管的场景依然在：

  • FULL OUTER JOIN：源码注释明说它"总是生成独立子问题"，是多成员子问题的典型来源。但当周围表有强过滤（如 t1.id=1）时，规划器常把 FULL JOIN 整体常量折叠掉，反而看不到差异——我本想用 FULL OUTER JOIN 造一个干净的反例，结果被优化器直接化简了，所以这里诚实说明：小数据下我没能在 16/19 上逼出 from_collapse_limit 改变计划的实例。
  • 人工加括号的复杂连接树 / 不可上拉的多表子查询：当子问题以"多成员 sub_joinlist"形态存在（>1 个关系且未被上拉），它才进入 from_collapse_limit 的判断。这在 16/19 日常 SQL 里少见，但在手写括号化连接、或含不可下推谓词的子查询时仍会出现。

所以务实结论：对绝大多数日常查询，调 from_collapse_limit 看不到效果，连接顺序出问题应优先动 join_collapse_limit；但遇到 FULL OUTER JOIN 重连接或手写括号复杂连接树、怀疑被钉死时，它仍是那个能对上的旋钮——只是别把它当成"常用调优项"。

七、geqo_threshold = 12 的隐藏关系（调参铁律）

文档与社区反复强调一个坑：geqo_threshold 默认 12。当单个 join 问题达到 12 个关系，规划器从穷举切换为遗传查询优化器 GEQO（采样而非全覆盖），可能得到非最优计划。而 from_collapse_limit / join_collapse_limit 默认都是 8，故意低于 12，保证默认不会触发 GEQO。

因此如果你为了"更好计划"把某个 collapse limit 调到 >= 12：

-- ❌ 危险：只调高 collapse limit，可能静默触发 GEQO 反而更差
SET from_collapse_limit = 16;

-- ✅ 正确：同一次 SET 里把 geqo_threshold 也调到它之上
SET from_collapse_limit = 16;
SET geqo_threshold = 20;

两者必须配套调，否则你以为在追更优计划、实际可能掉进另一个算法。

八、运维建议（给客户现场）

  1. 连接重排问题 → 调 join_collapse_limit：它是 user 级参数（会话内可改），定位慢查询时优先在单会话实验，对比计划与规划时间，别直接改全局。
  2. from_collapse_limit 默认 8 即可，日常基本不用碰：16/19 里子查询大多已被上拉、扁平逗号列表永远合并，它对日常 SQL 无可见影响；仅在写 FULL OUTER JOIN 重连接或手写括号复杂连接树、且怀疑连接顺序被钉死时，才值得排查。它不是"失效"，是"很少被触发"。
  3. 动 collapse limit 必同动 geqo_threshold：铁律，见第七节。
  4. 验证先于结论：任何连接顺序怀疑，先 EXPLAIN (ANALYZE, BUFFERS) 拿到真实成本，再决定是否调参；本博客所有计划均双版本实跑，未脑补。
