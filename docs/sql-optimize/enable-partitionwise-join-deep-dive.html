enable_partitionwise_join：分区对 Join 的性能提升与 M:N 爆炸陷阱

2026-07-08 SQL 优化 partition join GUC EXPLAIN memory parallel
✅ 已验证版 · PG 16.11 + 16.14 双版本实测

一句话总结

enable_partitionwise_join 默认 off——和 enable_partitionwise_aggregate 一样，它是 enable_* 家族里少数默认 off 的成员。开启后，优化器会把两个分区表的大 Join 拆成按分区对的 N 个小 Join，每对独立选最优算法。但前提条件很严格（两表都按同一分区键分区 + 连接条件包含全部分区键），且有一个致命陷阱：只按分区键 Join 时会触发 M:N 爆炸，产生数亿行中间结果。

白话版：大车间 vs 4 个小车间

假设你有按季度分区的订单表（orders）和订单明细表（order_items），要按 order_date + id 连接：

模式比喻机制内存
off（默认） 把所有季度数据堆到一个大车间做 Join Append → 1 个 Hash Join
所有分区数据汇入后统一连接		 1 × Hash 表
（本例 6528kB）
on 每对季度各自在一个小车间做 Join Append → 4 个 Hash Join（分区对）
Q1 对 Q1、Q2 对 Q2……		 4 × Hash 表
（本例 4×1632kB = 6528kB）

好处是每对小 Join 可以并行处理、各自选最优算法。坏处是规划成本翻倍（4 次规划 vs 1 次），而且触发条件很严格——不是随便两个分区表 Join 就能生效。更危险的是，只按分区键 Join 会产生 M:N 爆炸。

触发条件：三道门槛

partitionwise join 不是你想开就能生效。它有三道门槛：

门槛条件不满足的后果
1. 两表都分区 Join 两侧都必须是分区表 一侧是非分区表 → 不触发（场景 4 验证）
2. 同一分区键 两表按相同分区策略（RANGE/LIST）+ 相同分区键分区 分区键不同 → 不触发
3. 连接条件含分区键 Join 条件中包含分区键列 不含分区键 → 不触发（场景 2 验证）

三个条件同时满足才生效。错过任何一个，ON/OFF 的执行计划完全一样。

场景 1：条件满足——两分区表 + Join 含分区键 + 唯一键

这是 partitionwise join 的理想场景：两个分区表、同一分区键（order_date）、连接条件包含分区键和唯一键（id）。

OFF（默认）——1 个大 Parallel Hash Join：

SET enable_partitionwise_join = off;
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.id, o.customer_id, o.order_date, o.amount, oi.product_id, oi.quantity
FROM orders o JOIN order_items oi
  ON o.order_date = oi.order_date AND o.id = oi.order_id;

场景 2：连接不含分区键——不触发

Join 条件只有 o.id = oi.order_id，不含分区键 order_date——门槛 3 不满足。

SET enable_partitionwise_join = off;
EXPLAIN (ANALYZE)
SELECT o.id, o.customer_id, o.amount, oi.product_id
FROM orders o JOIN order_items oi ON o.id = oi.order_id;

SET enable_partitionwise_join = on;
EXPLAIN (ANALYZE)
SELECT o.id, o.customer_id, o.amount, oi.product_id
FROM orders o JOIN order_items oi ON o.id = oi.order_id;

结果：ON 和 OFF 的执行计划完全一致——都是 1 个 Hash Join（非并行），Hash Cond: oi.order_id = o.id。双版本行为完全一致。

这个场景印证了门槛 3：连接不含分区键 → partitionwise join 不触发，不管 ON 还是 OFF。

场景 3：只按分区键 Join——M:N 爆炸陷阱 ⛔

这是 partitionwise join 的致命陷阱：连接条件只有分区键 order_date，没有唯一键。每个分区对内部，所有同日期的 orders 行和所有同日期的 order_items 行做笛卡尔式连接。

SET enable_partitionwise_join = off;
EXPLAIN (COSTS ON)
SELECT o.id, o.amount, oi.product_id
FROM orders o JOIN order_items oi ON o.order_date = oi.order_date;

-- OFF: 估算行数 = 136,991,671（1.37 亿行）

SET enable_partitionwise_join = on;
EXPLAIN (COSTS ON)
SELECT o.id, o.amount, oi.product_id
FROM orders o JOIN order_items oi ON o.order_date = oi.order_date;

-- ON: 估算行数 = 136,986,330（1.37 亿行，4 个分区对各约 3400 万行）

1.37 亿行是怎么来的？10 万 orders × 50 万 order_items / 365 天 ≈ 每个 date 约 274 × 1369 ≈ 37.5 万行/天 × 365 = 1.37 亿。

ON 时把灾难拆成 4 个小灾难（每分区对 3400 万行），总结果一样糟糕。这是一个SQL 设计问题，不是 partitionwise join 的 bug——但它提醒我们：只按分区键 Join 没有唯一键约束时，行数会爆炸。

场景 4：非分区表 vs 分区表——不触发

一侧是非分区表（普通表），另一侧是分区表——门槛 1 不满足。

CREATE TABLE non_part_orders AS SELECT * FROM orders;
ANALYZE non_part_orders;

SET enable_partitionwise_join = on;
EXPLAIN (COSTS ON)
SELECT npo.id, npo.amount, oi.product_id
FROM non_part_orders npo JOIN order_items oi
  ON npo.order_date = oi.order_date AND npo.id = oi.order_id;

-- 结果：Parallel Hash Join（大 hash），没有 Append 分区对

结论：即使 ON，非分区表 + 分区表仍然是普通 Hash Join。partitionwise join 只在两侧都是分区表时才生效。

内存对比：拆分而非倍增

一个常见的误解是 partitionwise join 会倍增内存。实测数据：

模式Hash 表数量每个 Hash 表总内存
OFF16528 kB (131072 buckets)6528 kB
ON41632 kB (32768 buckets)6528 kB

总内存完全相同——4 个小 Hash 表的总和等于 1 个大 Hash 表。这不是"倍增"，而是"拆分"。每个分区对只需要处理自己分区的数据，Hash 表更小但数量更多。

这与 enable_partitionwise_aggregate 不同——aggregate 的内存确实会 N 倍放大，因为每分区独立做 HashAggregate 需要各自的 work_mem上限。join 的 Hash 表大小由数据量决定，不是 work_mem。

ON 的规划时间反而更短——分区对路径更简单（每个小 Join 只需要考虑自己分区的统计），比一个大 Join 的路径搜索空间小。4 分区场景下，规划成本增加几乎可以忽略。

与 enable_partitionwise_aggregate 的区别

GUC默认触发条件内存影响反面陷阱
enable_partitionwise_aggregate off 分区键 ∈ GROUP BY 或 非 GROUP BY N × work_mem（倍增） 小数据量反慢
enable_partitionwise_join off 两分区表 + 同分区键 + Join 含分区键 总量不变（拆分） 只按分区键 → M:N 爆炸

关键区别：aggregate 的内存会 N 倍放大（每分区独立 HashAggregate 各占 work_mem），join 的内存总量不变（4 个小 Hash 表 = 1 个大 Hash 表）。但 join 有 aggregate 不存在的M:N 爆炸陷阱。

什么时候该开？

✅ 适合开启的场景

条件原因
两表按同一分区键分区 + Join 含分区键和唯一键 分区对 Join 每对是 1:1 或 1:N，不会爆炸
OLAP/报表角色 分区对可并行，大表 Join 提速 10~17%
分区数 ≤ 32 规划成本可控
Join 条件含分区键 + 唯一/近似唯一键 避免 M:N 爆炸

⛔ 不适合开启的场景

条件原因
只按分区键 Join（无唯一键） M:N 爆炸，数亿行中间结果（场景 3 验证）
OLTP 高并发 规划成本翻倍，小查询反而更慢
一侧是非分区表 不触发，白开
分区数 > 100 规划成本过高

配置建议：角色级而非全局

不要全局开启。用角色级配置，只给 OLAP/报表角色开：

-- ✅ 推荐：仅 OLAP 角色开启
ALTER ROLE olap_user SET enable_partitionwise_join = on;

-- ✅ 全局保持默认 off
-- 不需要任何操作，默认就是 off

-- ❌ 不要这样做（全局开启影响所有连接）
-- ALTER SYSTEM SET enable_partitionwise_join = on;

临时验证（会话级）：

-- 单次查询验证
SET enable_partitionwise_join = on;
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.*, oi.*
FROM orders o JOIN order_items oi
  ON o.order_date = oi.order_date AND o.id = oi.order_id;
RESET enable_partitionwise_join;

诊断用法（Christophe Pettus 原则）：所有 enable_* 是诊断探针。关掉它看优化器选什么，诊断完立刻恢复 on。但 enable_partitionwise_join 默认 off，所以反过来——临时开启看效果，验证完恢复 off。

结论

  1. 默认 off 是对的——触发条件严格，且有 M:N 爯炸陷阱
  2. OLAP 角色级开启——ALTER ROLE olap_user SET enable_partitionwise_join = on
  3. 条件满足时提速 10~17%——把大 Join 拆成分区对，每对可并行
  4. 内存总量不变——4 个小 Hash 表 = 1 个大 Hash 表，与 aggregate 的 N 倍放大不同
  5. ⛔ 只按分区键 Join 是致命陷阱——M:N 爯炸产生数亿行，必须加唯一键约束
  6. 三道门槛同时满足才生效——两分区表 + 同分区键 + Join 含分区键
  7. 开启前用 EXPLAIN 验证——确认触发了分区对 Join，确认没有 M:N 爯炸
  8. 双版本执行计划结构完全一致——行数估算差异 <1%，不影响定性结论
