enable_partition_pruning 两阶段分区裁剪深度解析

2026-07-06 SQL 优化 partition pruning GUC EXPLAIN plan-time execution-time
✅ 已验证版 · PG 19 Beta1 实测

一句话总结

enable_partition_pruning 控制分区裁剪的两个不同时刻：计划时（Plan-time）和执行时（Execution-time）。两种裁剪在 EXPLAIN 中发出不同的信号——计划时看 Subplans Removed，执行时看 (never executed)。关掉它只是诊断探针，真正的修复在 WHERE 条件。

实测验证：识别两种裁剪信号

测试表准备

CREATE TABLE sales (
    id serial,
    amount numeric,
    sold_at date NOT NULL,
    region text
) PARTITION BY RANGE (sold_at);

CREATE TABLE sales_2023 PARTITION OF sales
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');
CREATE TABLE sales_2024 PARTITION OF sales
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
CREATE TABLE sales_2025 PARTITION OF sales
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- 每分区 1000 行
INSERT INTO sales (amount, sold_at, region)
SELECT (random() * 10000)::numeric(10,2),
       '2023-01-01'::date + (random() * 1095)::int,
       (ARRAY['North','South','East','West'])[1 + (random() * 3)::int]
FROM generate_series(1, 3000);

ANALYZE sales;

场景 1：计划时裁剪 —— 常量条件

WHERE 分区键 = 常量，优化器直接排除不匹配分区：

EXPLAIN SELECT * FROM sales WHERE sold_at = '2024-06-15';

--                            QUERY PLAN
-- ----------------------------------------------------------------
--  Seq Scan on sales_2024 sales  (cost=0.00..19.71 rows=2 width=19)
--    Filter: (sold_at = '2024-06-15'::date)
-- (2 rows)

❌ 没有 sales_2023，没有 sales_2025。分区从计划中直接消失了。纯常量条件下不需要 EXPLAIN ANALYZE，plain EXPLAIN 就能看到裁剪效果。

场景 2：执行时裁剪 —— Generic Plan

强制通用计划来观察执行时裁剪信号：

SET plan_cache_mode = force_generic_plan;
PREPARE q(date) AS SELECT * FROM sales WHERE sold_at = $1;
EXPLAIN (ANALYZE, COSTS OFF, TIMING OFF) EXECUTE q('2024-06-15');

--                            QUERY PLAN
-- ----------------------------------------------------------------
--  Append (actual rows=1.00 loops=1)
--    Subplans Removed: 2
--    ->  Seq Scan on sales_2024 sales_1 (actual rows=1.00 loops=1)
--          Filter: (sold_at = $1)
--          Rows Removed by Filter: 1016
--  Planning Time: 1.621 ms
--  Execution Time: 0.830 ms
-- (11 rows)

这次看到了关键信号：Subplans Removed: 2——2023 和 2025 分区被执行时裁剪掉了，但只在运行时才知道。计划中仍然包含 Append 节点，因为优化器在计划时不知道 $1 的值。

场景 3：关闭裁剪 — 全线扫描

作为诊断探针，enable_partition_pruning = off 让你看到"没有裁剪的世界"：

SET enable_partition_pruning = off;
EXPLAIN SELECT * FROM sales WHERE sold_at = '2024-06-15';

--                                 QUERY PLAN
-- -------------------------------------------------------------------------
--  Append  (cost=0.00..57.53 rows=6 width=19)
--    ->  Seq Scan on sales_2023 sales_1  (cost=0.00..20.10 rows=2 width=19)
--          Filter: (sold_at = '2024-06-15'::date)
--    ->  Seq Scan on sales_2024 sales_2  (cost=0.00..19.71 rows=2 width=19)
--          Filter: (sold_at = '2024-06-15'::date)
--    ->  Seq Scan on sales_2025 sales_3  (cost=0.00..17.69 rows=2 width=19)
--          Filter: (sold_at = '2024-06-15'::date)
-- (7 rows)

所有三个分区都在，每个都带着永远不会匹配的 Filter。cost=57.53 vs 正常 cost=19.71（单分区），接近 3 倍。这就是裁剪带来的收益。

诊断铁律

  1. SET enable_partition_pruning = off → 重跑 EXPLAIN ANALYZE
  2. 对比两个计划：差异 = 裁剪收益，无差异 = 裁剪根本没生效
  3. 确认问题后 RESET enable_partition_pruning，修查询，不是修 GUC

全局关闭 enable_partition_pruning 会让所有分区查询失去裁剪能力——这几乎一定比任何"裁剪没生效"的问题更糟糕。

裁剪不生效的三种常见原因

原因示例解释
WHERE 未约束分区键 WHERE region = 'North' 分区键是 sold_at，用 region 过滤 → 所有分区都必须扫描
分区键被函数包裹 WHERE EXTRACT(YEAR FROM sold_at) = 2024 优化器无法将函数结果匹配到分区约束
应改写为 WHERE sold_at >= '2024-01-01' AND sold_at < '2025-01-01'
类型不匹配 分区键 timestamp，条件给 date 隐式类型转换阻断分区约束匹配
确保 WHERE 类型与分区键类型一致

实测验证以上三种裁剪失效：

-- ❌ 失效 1: WHERE 不约束分区键
EXPLAIN SELECT * FROM sales WHERE region = 'North';
-- Append: sales_2023 + sales_2024 + sales_2025（三个全扫）

-- ❌ 失效 2: 分区键被函数包裹
EXPLAIN SELECT * FROM sales WHERE EXTRACT(YEAR FROM sold_at) = 2024;
-- Append: sales_2023 + sales_2024 + sales_2025（三个全扫）

-- ✅ 正确: 直接用分区键范围
EXPLAIN SELECT * FROM sales
WHERE sold_at >= '2024-01-01' AND sold_at < '2025-01-01';
-- Seq Scan on sales_2024 only

constraint_exclusion 与 enable_partition_pruning 的关系

这两个 GUC 不是同一个东西：

GUC默认值适用范围作用
constraint_exclusionpartition仅继承式分区（PG10 之前）用表 CHECK 约束排除不匹配的继承子表
enable_partition_pruningon声明式分区（PG10+）用分区边界约束排除不匹配的分区

PG11+ 的声明式分区只用 enable_partition_pruning。调 constraint_exclusion 对你的声明式分区表没有任何作用。这是很多 DBA 误踩的坑。

跨版本一致性

enable_partition_pruning 在 PG16.11 和 PG19 Beta1 上的默认值、参数描述、行为完全一致（on，user 级别，boot_val = on）。这不是 PG19 新特性，而是所有声明式分区用户的通用工具。
