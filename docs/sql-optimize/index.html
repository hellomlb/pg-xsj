from_collapse_limit 深度实测：PG 16/19 里它为何很少被触发，真正管用的旋钮是谁

2026-07-24 SQL 优化

from_collapse_limit 默认 8，官方说是"FROM 子查询折叠阈值"。但 PG 16.11 与 19 Beta1 实跑验证：FROM 子查询大多被上拉（窗口函数仍拦截为 Subquery Scan）、扁平逗号列表永远合并，参数很少改变计划——不是"无效"而是入口闸门（pull_up_subqueries）已把大多数子查询上拉掉。真正控制连接重排、调了能立刻看到计划变化的，是它兄弟 join_collapse_limit；调高任一 collapse limit 必须同步调高 geqo_threshold（默认12），否则静默掉进 GEQO。所有 EXPLAIN 双版本实跑，未脑补。
