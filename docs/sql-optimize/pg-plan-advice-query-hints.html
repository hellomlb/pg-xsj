pg_plan_advice + pg_stash_advice：PostgreSQL 15 年之争的终点

✅ SQL 验证通过 2026-06-08 SQL 优化 pg_plan_advice pg_stash_advice Query Hints PG19 执行计划 规划器

一句话总结

PostgreSQL 社区在邮件列表上吵了 15 年的 Query Hints，终于在 PG19 以 Plan Advice（规划建议） 的形式落地了。规划器现在会告诉你它在做什么，你也可以约束它——但不是替代它，而是和它协商。

历史背景：为什么「永远不会出现」

任何在 PG 社区待久了的人都听说过那六条铁律：Query Hints 会造成维护噩梦、大版本升级后失效、阻碍 DBA 找根本原因、还会阻碍优化器继续改进（因为用户不再上报 bug）。

官方 Wiki 白纸黑字：「We (committers) don't intend to implement query hints.」

但 2010 年，Kevin Grittner 在邮件列表上点出了一个著名的悖论：

那些最强烈反对 hints 的人，同时也在建议「把 SQL 写成两种不同逻辑等价的结构让优化器选不同计划」——这本质上就是 hints，只是拒绝把它明确记录下来而已。

在那场持续数月的战役里，后来实现这个功能的 Robert Haas 说了一句话：

「说我们不想要 hints 是愚蠢的。我们只是希望它们真正有效，而不是一团糟。」

Tom Lane 留了一道缝：「我没见过一个不糟糕的 hints 方案……但我不是说不可能有。」

十五年后，Robert Haas 亲手实现了它，并刻意不叫它 hints，而叫 Plan Advice。这个命名选择本身就是一种态度。

设计哲学：逐条回应历史反对意见

历史反对意见 pg_plan_advice 的解法
不应嵌入 SQL 文本（Oracle 注释式） 建议完全在 SQL 之外，通过 GUC 或 stash 设置
会完全替代规划器判断 建议只是约束搜索空间，规划器仍在其中自主决策
错误建议会产生灾难性计划 无法执行的建议标记为 Disabled，优雅降级，不崩溃
语法复杂难以手写维护 规划器自动生成建议字符串，DBA 照抄即可

亮眼功能：规划器告诉你它在做什么

传统 EXPLAIN 只告诉你规划器的选择，不告诉你「为什么」。现在有了 PLAN_ADVICE 选项：

-- 创建测试表（PG 16+ 可运行）
CREATE TABLE my_fact  (id bigint, dim_id int, val numeric);
CREATE TABLE my_dim   (id int PRIMARY KEY, name text);
CREATE INDEX idx_fact_dim_id ON my_fact(dim_id);

EXPLAIN (COSTS OFF, FORMAT TEXT)
SELECT f.val, d.name
FROM   my_fact f
JOIN   my_dim  d ON f.dim_id = d.id
WHERE  f.id BETWEEN 1 AND 1000;

在 PG19 Beta 1 上，使用 PLAN_ADVICE 选项的实际运行输出如下：

-- 测试环境（PG19 Beta 1 实测）
CREATE TABLE my_dim (id serial PRIMARY KEY, name text);
CREATE TABLE my_fact (id serial PRIMARY KEY, dim_id int, amount numeric);
INSERT INTO my_dim (name) SELECT 'dim_' || i FROM generate_series(1,1000) AS i;
INSERT INTO my_fact (dim_id, amount) SELECT (random()*999+1)::int, random()*1000 FROM generate_series(1,50000);
CREATE INDEX idx_fact_dim_id ON my_fact(dim_id);
ANALYZE my_fact, my_dim;

EXPLAIN (COSTS OFF, PLAN_ADVICE)
SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
         QUERY PLAN
--------------------------------------------
 Hash Join
   Hash Cond: (f.dim_id = d.id)
   ->  Seq Scan on my_fact f
   ->  Hash
         ->  Seq Scan on my_dim d
 Generated Plan Advice:
   JOIN_ORDER(f d)
   HASH_JOIN(d)
   SEQ_SCAN(f d)
   NO_GATHER(f d)

规划器直接给出了能复现当前计划的建议字符串。线上某天突然选了奇怪的计划？跑一次 EXPLAIN (PLAN_ADVICE) 拿到字符串，钉住它，应用层完全无感知。

Plan Advice DSL 完整语法

建议语言（Plan Advice Domain Specific Language）共支持以下几类指令：

以下 Plan Advice DSL 语法在 PG19 Beta 1 实测验证通过。

扫描方式

-- 强制索引扫描（指定使用具体索引）
SET pg_plan_advice.advice = 'INDEX_SCAN(f idx_fact_dim_id)';

-- 强制顺序扫描
SET pg_plan_advice.advice = 'SEQ_SCAN(f)';

-- 禁止并行收集器（单节点查询）
SET pg_plan_advice.advice = 'NO_GATHER(f d)';

连接顺序

-- 括号 = 严格顺序：b 和 c 必须在 a 之后且严格按顺序
SET pg_plan_advice.advice = 'JOIN_ORDER(a (b c) d)';

-- 花括号 = 灵活分组：b 和 c 可以任意顺序，但必须在 a 之后
SET pg_plan_advice.advice = 'JOIN_ORDER(a {b c} d)';

连接方法

-- d1、d2 分别作两个独立 HASH JOIN 的内侧（inner side）
SET pg_plan_advice.advice = 'HASH_JOIN(d1 d2)';

-- d1 + d2 一起作单个 HASH JOIN 的内侧
SET pg_plan_advice.advice = 'HASH_JOIN((d1 d2))';

-- 指定 Nested Loop
SET pg_plan_advice.advice = 'NESTED_LOOP_PLAIN(f)';

组合使用

-- 多个建议合并在一个字符串中，空格分隔
SET pg_plan_advice.advice =
    'JOIN_ORDER(f d1 d2) HASH_JOIN(d1 d2) SEQ_SCAN(f) INDEX_SCAN(d1 idx_d1_pk)';

反馈机制：每条建议都有状态

对建议的执行结果，规划器会在 EXPLAIN 输出里标注每条建议的命运：

以下是 PG19 Beta 1 实测输出（pg_plan_advice.feedback_warnings = true，故意写错索引名和不存在的关系名）：

SET pg_plan_advice.advice = 'INDEX_SCAN(f no_such_index) SEQ_SCAN(z)';
EXPLAIN (COSTS OFF) SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
Supplied Plan Advice:
  INDEX_SCAN(f no_such_index) /* matched, inapplicable, failed */
  SEQ_SCAN(z) /* not matched */
状态含义
matched规划器找到了建议中的关系/索引
not matched建议中指定的关系名不存在于当前查询
inapplicable, failed指定的索引不存在
conflicting, failed与另一条建议矛盾，无法同时满足

生产环境告警配置：开启后失效建议会输出 WARNING 进日志，监控系统可以感知建议是否仍然有效：

-- 开启后，无效建议进入 PostgreSQL 日志
SET pg_plan_advice.feedback_warnings = true;

pg_stash_advice：应用层无感知的生产利器

当你无法修改应用代码时（第三方 ORM、遗留系统、无法发版），pg_stash_advice 是终极手段——把建议按 query_id 持久化到数据库里，按数据库/角色粒度自动生效，应用层完全不感知。

以下 pg_stash_advice 函数在 PG19 Beta 1 实测验证通过。

基本工作流

-- 第一步：建立命名 stash
SELECT pg_create_advice_stash('production_tuning');

-- 第二步：用 EXPLAIN VERBOSE 获取 Query Identifier
EXPLAIN (VERBOSE, COSTS OFF)
SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
-- 输出中找到：Query Identifier: -1535698081138636168

-- 第三步：把建议和 Query ID 绑定
SELECT pg_set_stashed_advice(
    'production_tuning',
    -1535698081138636168,
    'INDEX_SCAN(f idx_fact_dim_id)'
);

-- 第四步：查看 stash 内容（实测函数名）
SELECT * FROM pg_get_advice_stash_contents('production_tuning');
-- stash_name         | query_id              | advice_string
-- production_tuning  | -1535698081138636168  | INDEX_SCAN(f idx_fact_dim_id)

-- 第五步：按数据库或角色粒度激活（应用层完全无感知）
ALTER DATABASE mydb SET pg_stash_advice.stash_name = 'production_tuning';
ALTER ROLE reporting_user SET pg_stash_advice.stash_name = 'reporting_tuning';

-- 验证计划已被建议影响
SET pg_stash_advice.stash_name = 'production_tuning';
EXPLAIN (COSTS OFF) SELECT * FROM my_fact f JOIN my_dim d ON f.dim_id = d.id;
-- Supplied Plan Advice:
--   INDEX_SCAN(f idx_fact_dim_id) /* matched */

-- 第六步：统计信息更新后，问题消失，删掉 stash
SELECT pg_drop_advice_stash('production_tuning');

持久化配置

-- stash 持久化到磁盘，重启后依然有效
ALTER SYSTEM SET pg_stash_advice.persist = on;
SELECT pg_reload_conf();

查看当前所有 stash 中的建议

-- 列出所有已注册的 stash（实测函数名）
SELECT * FROM pg_get_advice_stashes();
-- stash_name         | num_entries
-- production_tuning  | 1

-- 查看特定 stash 内容
SELECT * FROM pg_get_advice_stash_contents('production_tuning');

局限性：官方明确说明不在范围内

限制说明
无法控制聚合策略 排序聚合 vs 哈希聚合，暂时管不到
无法控制 UNION/INTERSECT 集合操作的执行策略不在 DSL 范围内
正确性优先 规划器可以以正确性为由拒绝建议（不是 bug，是设计）
轻微性能开销 即使最终计划未改变，应用建议本身也有 CPU 成本
仅 PG19+ pg_plan_advice 和 pg_stash_advice 是 PG19 新增 contrib 模块，老版本不可用

使用决策树：什么时候该用

规划器选错计划？
│
├─ 先检查统计信息
│   └─ ANALYZE 是否足够新？autovacuum 是否正常？
│
├─ 再检查参数
│   └─ random_page_cost 是否与实际存储（SSD/HDD）匹配？
│
├─ 考虑扩展统计
│   └─ CREATE STATISTICS（多列相关性）
│
└─ 以上都做了，规划器还是顽固？
    │
    ├─ 能改应用代码 → SET pg_plan_advice.advice = '...'
    │
    └─ 不能改应用代码 → pg_stash_advice（按 query_id 绑定，应用无感知）

官方定位：Plan Advice 是手术刀，不是大锤。先修统计信息、调 random_page_cost、建 CREATE STATISTICS，以上全做完之后规划器还顽固，再动它。

安装（PG19 编译安装 contrib）

# 编译时包含 contrib 模块
make -C contrib/pg_plan_advice install
make -C contrib/pg_stash_advice install
-- 加载模块（需要在 shared_preload_libraries 中）
-- postgresql.conf: shared_preload_libraries = 'pg_plan_advice,pg_stash_advice'
-- 重启后：
LOAD 'pg_plan_advice';
LOAD 'pg_stash_advice';

小结

pg_plan_advice + pg_stash_advice 让 PostgreSQL 在「完全信任规划器」和「Oracle 式强制提示」之间走出了第三条路：

  1. 建议在 SQL 之外——应用代码不感知，维护成本低
  2. 自动生成建议字符串——DBA 不用手写 DSL，规划器写，DBA 抄
  3. 优雅降级——写错了不崩溃，只是 failed 标记 + WARNING 日志
  4. pg_stash_advice 按 query_id 持久化——最典型场景：不能改应用代码、统计信息又不准时的救命稻草

Tom Lane 在 2010 年说「没见过不糟糕的方案，但不是说不可能有」。十五年后，Robert Haas 交出了答卷。
