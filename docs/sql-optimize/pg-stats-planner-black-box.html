pg_stats：PostgreSQL 规划器是怎么「脑补」你的数据的

2026-06-23 SQL 优化 pg_stats pg_statistic MCV 直方图 多列统计 CREATE STATISTICS 查询规划器 Richard Yen POSETTE 2026

一句话总结

PostgreSQL 查询规划器不读你的数据，它读 pg_stats 里的「数据户口本」。户口本错，决策就错。 本文基于 Richard Yen 在 POSETTE 2026 的演讲，从 5 大核心机制到 6 步排错法，全链路实战，所有示例在 PG 19 Beta 1 的 1M 行 customers 表上跑出真实数据。

"The Query Planner is only as smart as the statistics you feed it." — Richard Yen

为什么这事重要：一个真实的"反直觉"案例

假设我们有这样一张表（Richard 演讲原例 + 我重新造数据）：

CREATE TABLE customers (
    id          bigserial PRIMARY KEY,
    city        text NOT NULL,
    state       text NOT NULL,
    signup_date date NOT NULL
);

我们给 state 建了索引，但查 state = 'CA' 时，规划器选了 Bitmap Heap Scan 而不是 Index Scan。这违反了很多人的直觉——"明明有索引为什么不用？"

答案藏在 pg_stats 里。CA 占 6.5% ≈ 65000 行，Bitmap Heap Scan 实际更快。规划器没有错，是我们看不懂它。

核心机制 1：pg_statistic vs pg_stats

1.1 真相与人类视图

维度 pg_statistic pg_stats
存储格式优化给规划器用（数组/二进制）人类可读
谁能看superuserpublic（默认）
更新方式ANALYZE 表时pg_statistic 的视图

1.2 四大核心字段

SELECT attname, n_distinct, null_frac, correlation
FROM pg_stats
WHERE tablename = 'customers'
ORDER BY attname;

在 PG 19 Beta 1 的 1M 行 customers 表上：

attnamen_distinctnull_fraccorrelation
city100050-0.29
id-101
signup_date182500.01
state500-0.07

关键解读：

  • n_distinct = -1 表示该列所有行值唯一（PG 内部用负数标记"唯一"）。
  • correlation 是 [-1, +1] 区间的数。+1 表示物理顺序 = 逻辑顺序（id 是 bigserial 自增插入，所以 1.0 完美排序）。0 表示完全乱序。
  • city 的 correlation = -0.29 是负的：意味着物理顺序大致按 city 倒序排列（因为我们用伪随机种子造数据）。

1.3 实战经验：n_distinct 估算精度

-- 实际 distinct 值
SELECT COUNT(DISTINCT city) FROM customers;  -- 10005

-- pg_stats 估算
SELECT n_distinct FROM pg_stats WHERE tablename='customers' AND attname='city';  -- 10005

误差 0%。但注意：n_distinct 是基于 采样 估算的，10M+ 行的表可能误差 1-5%。

核心机制 2：MCV（最频值）—— 等值查询的"快查表"

2.1 看 MCV

SELECT
    unnest(most_common_vals::text::text[]) AS state,
    ROUND(unnest(most_common_freqs)::numeric, 4) AS frequency
FROM pg_stats
WHERE tablename = 'customers' AND attname = 'state'
LIMIT 8;

PG 19 Beta 1 上的 MCV 输出（造数据时把 5 个州各设 6.5%）：

statefrequency
NY0.0655
FL0.0651
CA0.0647
TX0.0646
WY0.0642
IA0.0163
KY0.0160
ID0.0158

注意：5 个州各占 6.5% 进入 MCV，其余 45 个州因为都是 1.5% 或更低，**MCV 只列高频**，不常见的值进直方图。

2.2 MCV 怎么影响规划

查 state = 'CA' 时，规划器：

  1. 看 most_common_vals 找到 'CA'
  2. 读对应 most_common_freqs = 0.0647
  3. 行数估算 = 100万 × 0.0647 ≈ 64700 行
  4. 对比 Bitmap Heap Scan vs Seq Scan 的 cost，选便宜的

这就是为什么规划器「知道」CA 多——它读的是 MCV，不是你的数据。

数据越倾斜，MCV 越值钱。 如果某列所有值都均匀（每个 1%），MCV 反而没意义，直方图更管用。

核心机制 3：直方图 —— 范围查询的"分布地图"

3.1 默认 100 桶是什么样

SELECT (unnest(histogram_bounds::text::timestamp[]))::date AS bucket_bound
FROM pg_stats
WHERE tablename = 'customers' AND attname = 'signup_date'
LIMIT 8;

PG 19 Beta 1 上的直方图输出（默认 STATISTICS=100）：

bucket_bound
─────────────
 2017-12-31
 2018-01-18
 2018-02-05
 2018-02-24
 2018-03-15
 2018-04-01
 2018-04-20
 2018-05-08

桶宽 ≈ 17 天。1M 行 ÷ 100 桶 = 1万行/桶。我们的数据是均匀分布 5 年（1825 天），所以 17 天/桶完全合理。

3.2 调到 1000 桶：精度提升 8.5 倍

ALTER TABLE customers ALTER COLUMN signup_date SET STATISTICS 1000;
ANALYZE customers;

同样前 8 个桶：

bucket_bound
─────────────

 2018-01-03
 2018-01-05
 2018-01-07
 2018-01-09
 2018-01-10
 2018-01-12

桶宽 ≈ 2 天。精度提升 8.5 倍。

3.3 实战建议

⚠️ 不要全局调高 default_statistics_target！ANALYZE 会变慢，pg_statistic 会膨胀。只调"估算问题大的列"。

场景推荐
OLTP 表，< 1000 万行默认 100 够用
DW/分析型表500-1000 全局
时间范围查询慢对时间列单独 SET STATISTICS 1000
列基数极高单列 2000+

核心机制 4：Correlation —— 物理顺序的"提示"

回到 1M 数据集：

  • id：correlation = 1.0（bigserial 顺序插入，物理 = 逻辑）
  • city：correlation = -0.29（伪随机，物理顺序与 city 倒序弱相关）
  • signup_date：correlation = 0.01（伪随机，物理乱序）

这影响什么？ Index Scan 的成本估算。

如果 correlation = 1.0，走 WHERE id > 500000 时，规划器认为「物理读是连续的」，random_page_cost 接近 seq_page_cost，Index Scan 几乎免费。

如果 correlation ≈ 0，物理读 = 东一榔头西一棒槌，Index Scan 成本暴涨，Seq Scan 反而更划算。

调优技巧：经常 ORDER BY 的列，定期 CLUSTER table USING index 重建表的物理顺序，让 correlation 趋近 1.0。

核心机制 5：多列统计 —— 解决"列独立性谎言"

5.1 问题的本质

默认规划器假设列之间统计独立：

P(city='Cheyenne' ∧ state='WY') = P(city) × P(state)

但 Cheyenne 99% 都在 Wyoming。这假设在地理数据上是假的。

5.2 50000 行 vs 估算偏差

我们造数据时设计：Cheyenne 99% 都在 WY。查询：

EXPLAIN ANALYZE SELECT * FROM customers WHERE city = 'Cheyenne' AND state = 'WY';

4a. 没有多列统计时（PG 19 Beta 1）：

Bitmap Heap Scan on customers
  (cost=1250.18..6959.86 rows=3167 width=24)
  (actual time=3.897..8.460 rows=49999.00 loops=1)
  Recheck Cond: ((city = 'Cheyenne'::text) AND (state = 'WY'::text))

规划器估算 3167 行，实际 49999 行，偏差 15.8 倍。

5.3 解决：CREATE STATISTICS

CREATE STATISTICS customers_city_state (dependencies, ndistinct)
    ON city, state FROM customers;
ANALYZE customers;

4b. 创建多列统计后（PG 19 Beta 1）：

Bitmap Heap Scan on customers
  (cost=1349.24..7376.84 rows=52767 width=24)
  (actual time=3.629..8.202 rows=49999.00 loops=1)
  Recheck Cond: ((city = 'Cheyenne'::text) AND (state = 'WY'::text))

规划器估算 52767 行，实际 49999 行，偏差 1.05 倍。

状态规划器估算实际行数偏差
无多列统计31674999915.8x
有多列统计52767499991.05x

15.8 倍 → 1.05 倍。一行 DDL 解决问题。

5.4 何时该用 CREATE STATISTICS？

判断标准：

  1. 查询 WHERE 子句里同时过滤两列或更多列
  2. 实际行数 / 规划器估算 > 10x
  3. 业务上知道这些列强相关（如 city↔state, product_id↔category_id）

注意：(dependencies, ndistinct) 是组合写法，等价于创建两条统计。如只要一种，写 dependencies 即可。

核心机制 6：错估会"向上传染"—— 找最深的估算偏差节点

6.1 树形计划的解剖

EXPLAIN ANALYZE 输出是棵树，自底向上执行：

        Limit           ← 第 1 层（最浅，根节点）
          │
        Sort            ← 第 2 层
          │
        Hash Join       ← 第 3 层
          │
     ┌────┴────┐
   Scan      Hash          ← 第 4 层
  orders    (build)
     │
   orders
   表       ← 叶子（最深）

"最深的节点"= 离叶子最近的 = 数据离开存储层的第一站 = Seq Scan / Index Scan / Bitmap Heap Scan。

6.2 错估的"滚雪球"效应

规划器是串联反应——底层的行数估算错了，会像滚雪球一样乘给上层。

举例：

  • 底层 Seq Scan 估 1000 行 → 实际 100 万行
  • 上层 Hash Join 不知道底层翻车 → 选错 join 算法
  • 上上层 Sort 内存爆
  • 顶层 Limit 拿不到数

但根因只是底层那个 Scan 把行数估错了 1000 倍。

6.3 实操：找"最深的估算偏差节点"

从 EXPLAIN 输出从上往下翻，找第一个出现 Scan 关键字的节点：

节点类型是不是"最深"？
Seq Scan / Index Scan / Index Only Scan / Bitmap Heap Scan✅ 是
Hash Join / Nested Loop / Merge Join❌ 不是（上层）
Sort / Aggregate / Limit❌ 不是（更上层）
Append (UNION ALL)看里面

找到第一个 Scan 节点 → 看它的 rows=N vs actual rows=M → 是！这就是病灶。

实战排错 6 步法（Richard 钦定顺序）

当查询计划看起来不对时，按这个顺序来：

步骤动作何时用
1️⃣ EXPLAIN ANALYZE 对比估算行数 vs 实际行数 永远是第一步
2️⃣ 找最深的估算偏差节点 那里才是"病灶"
3️⃣ 看 pg_stats 那列的 n_distinct / MCV 数据真如规划器认为的那样？
4️⃣ 跑 ANALYZE 大 batch load / 迁移后，autovacuum 没追上
5️⃣ 单列仍偏差 → ALTER COLUMN SET STATISTICS 桶数不够
6️⃣ 多列联合偏差 → CREATE STATISTICS 列相关

⛔ 最后才考虑改 SQL 或设 enable_seqscan=off，别本末倒置。

成本参数速查：什么影响 cost 估算

规划器最后算 cost 时，用三个核心参数：

参数含义默认值何时调
random_page_cost 随机读一页的成本（Index Scan 走 heap fetch） 4.0 用 SSD 调到 1.1
seq_page_cost 顺序读一页的成本（Seq Scan） 1.0 通常不动
cpu_tuple_cost 处理每行的 CPU 成本 0.01 几乎永远不动

PG 调优的第一原则：先查统计信息，再调 random_page_cost。调 cost 参数治标不治本。

PG 19 vs PG 16 行为差异：Bitmap Heap Scan 成为主流

注意：Richard 演讲里 CA（17%）走的是 Seq Scan，但 PG 19 Beta 1 的 1M 数据上 CA（6.5%）走的是 Bitmap Heap Scan。

原因：

  • PG 19 优化器更倾向于 Bitmap Heap Scan，因为它能批量读 page
  • 我数据是 6.5% 而不是 Richard 的 17%，规模差
  • PG 19 的 cost 模型对 SSD 场景有微调

结论：在 PG 19 上，"没有索引时 Seq Scan"是常规；但只要有索引，规划器经常选 Bitmap Heap Scan 而不是 Index Scan。这是好事——现代硬件下 Bitmap 通常更快。
