enable_partitionwise_aggregate：分区聚合的内存放大与性能权衡

2026-07-07 SQL 优化 partition aggregate GUC EXPLAIN memory parallel
✅ 已验证版 · PG 16.11 + 16.14 双版本实测

一句话总结

enable_partitionwise_aggregate 是 enable_* 家族的异类——几乎所有 enable_* 参数默认都是 on（它们是优化器探针，关掉只用于诊断），唯独它默认 off。原因很简单：开了会吃内存。每个分区各自做一个 HashAggregate，内存占用 = 分区数 × work_mem。本文在 PG 16.11 和 16.14 双版本实测 8 分区 80 万行表，用数据告诉你什么时候该开、什么时候别碰。

白话版：大锅 vs 小锅

假设你有 8 个分区的订单表，要按地区做 SUM 汇总：

模式比喻机制内存
off（默认） 把 8 个分区的数据全堆到一起，用一口大锅煮 Append → 1 个 HashAggregate
所有分区数据汇入后统一聚合		 1 × work_mem
on 每个分区各自用一口小锅先煮，最后把 8 锅结果倒一起再搅 Append → 8 个 HashAggregate → 上层合并
每分区独立聚合后结果再汇总		 N × work_mem
N = 分区数

好处是每口锅可以并行煮（分区并行），坏处是内存吃 N 倍。这就是它默认 off 的原因——不是因为它不好，而是因为在 OLTP 环境下，8 分区 × 4MB = 32MB 一个查询，太贵了。

场景 1：GROUP BY 分区键（region）

分区键 region 出现在 GROUP BY 中——这是 partitionwise aggregate 的理想场景，因为每分区聚合后结果直接是最终结果的一部分，不需要再合并相同 key。

OFF（默认）——1 个大 HashAggregate 汇总所有分区：

SET enable_partitionwise_aggregate = off;
EXPLAIN (ANALYZE, BUFFERS)
SELECT region, SUM(amount) AS total
FROM sales_2026
GROUP BY region
ORDER BY region;

--                          QUERY PLAN
-- ----------------------------------------------------------------
--  Finalize GroupAggregate  (actual time=260.767..260.840 rows=8)
--    Group Key: sales_2026.region
--    Buffers: shared hit=5706
--    ->  Gather Merge  (actual time=260.758..260.824 rows=10)
--          Workers Planned: 2  Workers Launched: 2
--          ->  Sort  (actual time=201.936..201.939 rows=3 loops=3)
--                Sort Key: sales_2026.region
--                ->  Partial HashAggregate  (actual time=201.890..201.895 rows=3 loops=3)
--                      Group Key: sales_2026.region
--                      Batches: 1  Memory Usage: 24kB
--                      ->  Parallel Append  (actual rows=266667 loops=3)
--                            ->  8× Parallel Seq Scan on sales_2026_* 
--  Planning Time: 0.735 ms
--  Execution Time: 260.982 ms

ON——8 个小 HashAggregate 各自处理分区：

SET enable_partitionwise_aggregate = on;
EXPLAIN (ANALYZE, BUFFERS)

FROM sales_2026
GROUP BY region
ORDER BY region;

--    Sort Key: sales_2026_1.region
--    ->  Gather  (actual time=58.035..236.318 rows=8)
--          Workers Planned: 2  Workers Launched: 2
--          ->  Parallel Append  (actual rows=3 loops=3)
--                ->  HashAggregate  (actual time=48.469..48.470 rows=1)
--                      Group Key: sales_2026_1.region
--                      Batches: 1  Memory Usage: 24kB
--                      ->  Seq Scan on sales_2026_north  (actual rows=100000)
--                ->  HashAggregate  (actual time=57.669..57.669 rows=1)

场景 2：GROUP BY 非分区键（sale_date）

分区键是 region，但 GROUP BY 的是 sale_date——分区键不在 GROUP BY 中。这种情况下 partitionwise aggregate 还有效吗？

有，但机制不同。每分区仍然做 partial aggregation（10 万行 → 181 行），但上层需要 Finalize GroupAggregate 合并——因为同一个 sale_date 分布在多个分区。

场景 3：GROUP BY 分区键 + 非分区键

GROUP BY region, sale_date——分区键和非分区键混合。这是最常见的 OLAP 报表场景。

场景 4：小数据量反例（work_mem=4MB）

不是所有场景 ON 都更快。同样 GROUP BY 分区键（8 个 group，每分区只 1 个 group），但本组把 work_mem 调小到 4MB 后跑一次：

执行时间184.241 ms198.437 ms807.024 ms775.082 ms

16.11 上 ON 反而更慢 7.7%——因为每分区只有 1 个 group（region 固定），HashAggregate 几乎没有聚合效果，但 8 个 HashAggregate node 的初始化和调度开销超过了收益。16.14 上 ON 反而快 3.9%——双版本结果不一致，说明 partitionwise aggregate 的收益在边界场景不稳定，取决于并行 worker 调度、IO 缓存、I/O 子系统状态等环境因素。

工程结论：数据量小 + group 数少时，partitionwise aggregate 是否值得开，没有稳定答案，必须 EXPLAIN ANALYZE 实测后再决定，不能想当然。

关键发现

1. 无并行条件下，ON 的收益明显缩水。

对比有并行（10 万行/分区）和无并行（1 万行/分区）的场景 1：

测试条件16.11 场景 116.14 场景 1
有并行（8P，10 万行/分区）+9.4%+9.7%
无并行（8P，1 万行/分区）+17.5%+6.3%
无并行（32P，1 万行/分区）+22.1%+11.2%

16.11 无并行时收益反而更高（17.5% vs 9.4%），这是因为禁用并行后 OFF 的基准变慢了（没有 Parallel Append 加速），而 ON 的 per-partition 聚合优势更突出。但 16.14 的收益从 9.7% 降到 6.3%——两台服务器配置不同（16.14 的 shared_buffers 仅 128MB，绝对执行时间是 16.11 的 5-6 倍），不能简单归因于版本差异。

2. 16.14 32P 场景 2 出现真实反例（慢 3.77%）。

3 次执行中只有 1 次 ON 比 OFF 快。GROUP BY 非分区键时，每分区做 partial aggregation 后仍需上层 Finalize 合并——当分区数多（32）、group 数多（365 天）、且无并行时，Finalize 合并的开销可能抵消 partial aggregation 的收益。这个反例是真实的，但受 16.14 服务器 IO 不足影响，不能下"PG 16.14 的 partitionwise 有 bug"这种结论。

3. "分区越多收益越大"不是普遍规律。

16.11 上 32P 提速 ≥ 8P（3 个场景全部成立），但 16.14 上 32P 提速反而低于 8P（3 个场景中 2 个下降）。分区数对收益的影响取决于数据量、group 数、服务器资源，不能想当然。

什么时候该开？

✅ 适合开启的场景

条件原因
分区键 ∈ GROUP BY 每分区聚合后结果直接是最终结果的一部分，不需要上层 Finalize 合并
分区数 ≤ 32 且 work_mem ≤ 16MB 总内存 ≤ 512MB，可控范围
OLAP/报表角色 低并发、大查询，内存充裕
每分区行数 >> group 数 聚合压缩比高，减少传输量收益大

⛔ 不适合开启的场景

条件原因
OLTP 高并发 N × work_mem × 并发数 可能 OOM
分区数 > 100 内存放大失控，100 分区 × 4MB = 400MB/查询
work_mem > 64MB 即使 8 分区也是 512MB，风险太高
GROUP BY 的 group 数很少（< 10） 聚合压缩比低，node 开销 > 收益（场景 4 实测验证）

配置建议：角色级而非全局

不要全局开启。用角色级配置，只给 OLAP/报表角色开：

-- ✅ 推荐：仅 OLAP 角色开启
ALTER ROLE olap_user SET enable_partitionwise_aggregate = on;
ALTER ROLE olap_user SET work_mem = '32MB';  -- 配合适度提高 work_mem

-- ✅ 全局保持默认 off
-- 不需要任何操作，默认就是 off

-- ❌ 不要这样做（全局开启影响所有连接）
-- ALTER SYSTEM SET enable_partitionwise_aggregate = on;

与 enable_partition_pruning 的区别

GUC默认作用内存影响
enable_partition_pruning on 排除不相关分区（减少扫描量） 无（只减少工作）
enable_partitionwise_aggregate off 每分区独立聚合（减少传输量） N × work_mem

两者可以同时使用——pruning 先减少分区数，partitionwise aggregate 再对剩余分区做并行聚合。pruning 是免费的，partitionwise aggregate 有代价。

结论

  1. 默认 off 是对的——内存放大 N 倍在 OLTP 环境是灾难
  2. OLAP 角色级开启——ALTER ROLE olap_user SET enable_partitionwise_aggregate = on
  3. 分区键 ∈ GROUP BY 时收益最大——每分区结果直接可用，不需要上层 Finalize
  4. 小数据量别开——node 初始化开销 > 聚合收益（场景 4 实测验证）
  5. 配合 partition pruning 使用——pruning 先减分区数，降低 partitionwise 的内存放大倍数
  6. 开启前用 EXPLAIN ANALYZE 验证——不是所有场景都更快
  7. 并行查询是收益放大器——有并行时 ON 提速 9-31%，无并行时缩水到 1-22%；16.14 无并行 32P 场景 2 甚至慢 3.8%。如果你的 OLAP 查询没有并行 worker，partitionwise 的收益可能不如预期
  8. 分区数翻倍不等于收益翻倍——16.11 无并行下 32P ≥ 8P，但 16.14 无并行下 32P 反而不稳定。"分区越多越赚"不是普遍规律
