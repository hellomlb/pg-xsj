非分区键查询实现分区裁剪：CHECK 约束 + constraint_exclusion 实战

2026-07-13 SQL 优化 partition pruning constraint_exclusion CHECK EXPLAIN gaps-and-islands
✅ 已验证版 · PG 16.11 + PG 19 Beta1 双版本实测

一句话总结

声明式分区表按 A 列分区，但业务总用 B 列查——默认会扫全部分区。当 B 与 A 强相关（如 session_id 随时间单调递增）时，给每个分区加一条声明其 B 列取值边界的 CHECK 约束，constraint_exclusion（默认 partition）就会利用这些 CHECK 把不匹配的分区直接裁掉。本文所有 SQL 在 PG 16.11 与 PG 19 Beta1 上实际执行，两个版本输出逐字节一致。

问题：非分区键查询命中所有分区

经典事件/日志表按时间分区：

CREATE TABLE events (
    event_time timestamp NOT NULL,   -- 分区键
    session_id int      NOT NULL,   -- 业务查询键（非分区键）
    payload    text
) PARTITION BY RANGE (event_time);

业务要"查某个 session 的全部事件"。它用 session_id 过滤，根本不碰分区键。enable_partition_pruning 只能拿分区键边界去匹配 WHERE，对 session_id 无能为力——于是优化器只能把三个分区全扫一遍。这是我们之前在《enable_partition_pruning 两阶段分区裁剪》里讲过的"裁剪失效原因 1：WHERE 未约束分区键"。

误区：constraint_exclusion 对声明式分区"没用"？

我们那篇文章里写过一句：

"PG11+ 的声明式分区只用 enable_partition_pruning，调 constraint_exclusion 对你的声明式分区表没有任何作用。"

这句话对分区键裁剪完全正确——分区边界匹配由 enable_partition_pruning 接管，constraint_exclusion 在那里纯冗余。但对本文的非分区键 + CHECK 约束场景，它需要被修正：constraint_exclusion（默认 partition）正是让 CHECK 约束参与分区裁剪的总开关。下面的场景 6 会直接证明——把它设为 off，三个分区立刻全部出现。

解法：CHECK 约束编码"非分区键 → 分区"的映射

核心前提：session_id 与 event_time 强相关。session 是递增分配的，2026-07-01 的 session 落在 1..1000，07-02 落在 1001..2000……既然每个分区的 session_id 范围是确定的，就把这个范围写死成 CHECK 约束。优化器在计划时把 WHERE 的 session_id 条件和各分区的 CHECK 比对，能证明不匹配的就裁掉。

要素作用
CHECK (session_id BETWEEN lo AND hi)声明"本分区的 session_id 只可能在这个区间"
constraint_exclusion = partition（默认）允许优化器用 CHECK 约束排除分区
session_id 与分区键强相关保证区间不重叠、不漏数据，裁剪才有意义

实测验证：测试表准备

DROP TABLE IF EXISTS events_np;
CREATE TABLE events_np (
    event_time timestamp NOT NULL,
    session_id  int NOT NULL,
    payload    text
) PARTITION BY RANGE (event_time);

CREATE TABLE events_np_d1 PARTITION OF events_np FOR VALUES FROM ('2026-07-01') TO ('2026-07-02');
CREATE TABLE events_np_d2 PARTITION OF events_np FOR VALUES FROM ('2026-07-02') TO ('2026-07-03');
CREATE TABLE events_np_d3 PARTITION OF events_np FOR VALUES FROM ('2026-07-03') TO ('2026-07-04');

-- session_id 与日期强相关（越大越新）
INSERT INTO events_np SELECT '2026-07-01'::timestamp + (g % 86400 || ' s')::interval, g, 'p'||g FROM generate_series(1,1000) g;
INSERT INTO events_np SELECT '2026-07-02'::timestamp + (g % 86400 || ' s')::interval, 1000+g, 'p'||g FROM generate_series(1,1000) g;
INSERT INTO events_np SELECT '2026-07-03'::timestamp + (g % 86400 || ' s')::interval, 2000+g, 'p'||g FROM generate_series(1,1000) g;

-- 关键：把每个分区的 session_id 边界写死成 CHECK 约束
ALTER TABLE events_np_d1 ADD CONSTRAINT ck_d1 CHECK (session_id BETWEEN 1    AND 1000);
ALTER TABLE events_np_d2 ADD CONSTRAINT ck_d2 CHECK (session_id BETWEEN 1001 AND 2000);
ALTER TABLE events_np_d3 ADD CONSTRAINT ck_d3 CHECK (session_id BETWEEN 2001 AND 3000);

ANALYZE events_np;

SHOW constraint_exclusion;   -- partition（默认值，无需修改）

逐场景 EXPLAIN（PG 16.11 与 19 Beta1 完全一致）

场景 1：分区键查询（基线，本来就会裁）

EXPLAIN SELECT * FROM events_np
WHERE event_time >= '2026-07-02' AND event_time < '2026-07-03';

--                         QUERY PLAN
-- ----------------------------------------------------------------
--  Seq Scan on events_np_d2 events_np
--    Filter: ((event_time >= '2026-07-02 00:00:00'::timestamp without time zone)
--             AND (event_time < '2026-07-03 00:00:00'::timestamp without time zone))
-- (2 rows)

这是 enable_partition_pruning 的功劳，仅作基线对照。

场景 2：非分区键点查 → 只扫 d1

EXPLAIN SELECT * FROM events_np WHERE session_id = 500;

--           QUERY PLAN
-- ------------------------------
--  Seq Scan on events_np_d1 events_np
--    Filter: (session_id = 500)
-- (2 rows)

events_np_d2、events_np_d3 从计划里直接消失了。d1 的 CHECK 是 1..1000，500 落在其中；d2 是 1001..2000、d3 是 2001..3000，优化器证明 500 不可能在那两个分区，直接裁掉。

场景 3：非分区键范围 → 只扫 d3

EXPLAIN SELECT * FROM events_np WHERE session_id BETWEEN 2500 AND 2800;

--                      QUERY PLAN
-- -------------------------------------------------
--  Seq Scan on events_np_d3 events_np
--    Filter: ((session_id >= 2500) AND (session_id <= 2800))
-- (2 rows)

场景 4：IN 跨两区 → 扫 d1 + d3，d2 被裁

EXPLAIN SELECT * FROM events_np WHERE session_id IN (500, 2500);

--                        QUERY PLAN
-- --------------------------------------------------
--  Append
--    ->  Seq Scan on events_np_d1 events_np_1
--          Filter: (session_id = ANY ('{500,2500}'::integer[]))
--    ->  Seq Scan on events_np_d3 events_np_2
--          Filter: (session_id = ANY ('{500,2500}'::integer[]))
-- (5 rows)

优化器对 IN 列表里每个值分别判断归属分区，500→d1、2500→d3，d2 两边都不沾，被裁。

场景 5 & 6：两个对照（证明机制）

-- 对照 A：去掉所有 CHECK 约束后，点查退化为全扫
ALTER TABLE events_np_d1 DROP CONSTRAINT ck_d1;
ALTER TABLE events_np_d2 DROP CONSTRAINT ck_d2;
ALTER TABLE events_np_d3 DROP CONSTRAINT ck_d3;
EXPLAIN SELECT * FROM events_np WHERE session_id = 500;
--  Append: events_np_d1 + events_np_d2 + events_np_d3（三个全扫）

-- 对照 B：恢复 CHECK，但关掉 constraint_exclusion → 仍然全扫
ALTER TABLE events_np_d1 ADD CONSTRAINT ck_d1 CHECK (session_id BETWEEN 1    AND 1000);
ALTER TABLE events_np_d2 ADD CONSTRAINT ck_d2 CHECK (session_id BETWEEN 1001 AND 2000);
ALTER TABLE events_np_d3 ADD CONSTRAINT ck_d3 CHECK (session_id BETWEEN 2001 AND 3000);
SET constraint_exclusion = off;
EXPLAIN SELECT * FROM events_np WHERE session_id = 500;
--  Append: events_np_d1 + events_np_d2 + events_np_d3（三个全扫）
RESET constraint_exclusion;

两个对照说明：CHECK 约束和 constraint_exclusion 缺一不可。对照 A 证明没有 CHECK 就无从裁剪；对照 B 证明开关一关，CHECK 约束被闲置。

场景 7：部分重叠 → 跨 d1 + d2

EXPLAIN SELECT * FROM events_np WHERE session_id BETWEEN 900 AND 1500;

--                        QUERY PLAN
-- --------------------------------------------------
--  Append
--    ->  Seq Scan on events_np_d1 events_np_1
--    ->  Seq Scan on events_np_d2 events_np_2
-- (5 rows)

900 在 d1（1..1000）内、1500 在 d2（1001..2000）内，区间横跨两个分区，两个都保留，但 d3 仍被裁。

真实扫描量对比（EXPLAIN ANALYZE）

EXPLAIN (ANALYZE, COSTS OFF, TIMING OFF) SELECT * FROM events_np WHERE session_id = 500;

--                         QUERY PLAN
-- ------------------------------------------------------------
--  Seq Scan on events_np_d1 events_np (actual rows=1 loops=1)
--    Filter: (session_id = 500)
--    Rows Removed by Filter: 999
-- (3 rows)

裁剪生效后，只扫 d1 的 1000 行（999 被过滤、1 返回）。若未裁剪，需要扫三个分区共 3000 行——扫描量降到 1/3。表越大、分区越多，收益越夸张（100 个分区 → 扫描量降到 1/100）。

注：PG 19 Beta1 在此计划末尾额外输出 Buffers: shared hit=7，是 19 对 EXPLAIN ANALYZE 的默认增强，不影响裁剪结论；两版本核心计划完全一致。

进阶：用 Gaps and Islands 推导 CHECK 边界

上面的边界是我"已知"的。真实环境里，你往往需要从数据反推每个分区的 session_id 范围。最简单的是 min/max：

SELECT pgc.relname,
       min(e.session_id) AS sid_min,
       max(e.session_id) AS sid_max,
       count(*)          AS rows
FROM events_np e
JOIN pg_class pgc ON pgc.oid = e.tableoid
GROUP BY pgc.relname
ORDER BY pgc.relname;

--    relname    | sid_min | sid_max | rows
-- --------------+---------+---------+------
--  events_np_d1 |       1 |    1000 | 1000
--  events_np_d2 |    1001 |    2000 | 1000
--  events_np_d3 |    2001 |    3000 | 1000

但当某个分区的 session_id 有空洞（如 d1 只插了 1..500 和 800..1000，缺 501..799），min/max 会给出错误的 1..1000 边界，导致 CHECK 比真实数据更宽。这时用 Gaps and Islands 算法求最大连续区间：

WITH ranked AS (
  SELECT tableoid, session_id,
         session_id - row_number() OVER (PARTITION BY tableoid ORDER BY session_id) AS grp
  FROM events_np
)
SELECT pgc.relname, min(session_id) AS sid_min, max(session_id) AS sid_max
FROM ranked r JOIN pg_class pgc ON pgc.oid = r.tableoid
GROUP BY pgc.relname, grp
ORDER BY pgc.relname, sid_min;

--    relname    | sid_min | sid_max
-- --------------+---------+---------
--  events_np_d1 |       1 |     500
--  events_np_d1 |     800 |    1000   ← 空洞 501..799 被识别为两段
--  events_np_d2 |    1001 |    2000
--  events_np_d3 |    2001 |    3000

带空洞的分区要用单个 OR CHECK 约束覆盖多段（不能拆成两条独立 CHECK 分别 ADD，否则加第一条时既有数据就违反它）：

ALTER TABLE events_np_d1 ADD CONSTRAINT ck_d1
  CHECK (session_id BETWEEN 1 AND 500 OR session_id BETWEEN 800 AND 1000);

EXPLAIN SELECT * FROM events_np WHERE session_id = 500;   -- → d1
EXPLAIN SELECT * FROM events_np WHERE session_id = 900;   -- → d1（落在 800..1000 段）
EXPLAIN SELECT * FROM events_np WHERE session_id = 1500;  -- → d2

优化器能正确处理 OR 形式的 CHECK，裁剪依然精准。

生效前提与陷阱（必读）

  1. 非分区键必须与分区键强相关。如果是随机 user_id，CHECK 范围会覆盖所有分区 → 裁剪彻底失效，白加约束还拖慢写入。
  2. CHECK 约束必须是"真"的。CHECK 会拦下越界插入——但如果你为写入性能放松 CHECK（或在应用层"保证"），一旦数据越界，查询会漏掉本应返回的行。这是正确性事故，不只是性能问题。
  3. 分区重组要同步维护 CHECK。分裂/合并/重命名分区时，边界漂移，CHECK 不跟着更新就会出上面第 2 条的问题。建议用迁移脚本或触发器强约束。
  4. 适合读多写少、分区稳定的日志/事件/历史表。不适合高频 DDL 或混乱数据。
  5. 写入与 DDL 开销。每行插入都过一遍 CHECK 校验；ALTER 加 CHECK 在大表上可能长时间持锁。

与 enable_partition_pruning 的关系（修正前文）

机制裁剪依据适用查询
enable_partition_pruning（默认 on）分区边界约束（分区键）WHERE 含分区键
constraint_exclusion（默认 partition）分区上的CHECK 约束（含用户自定义）WHERE 含被 CHECK 覆盖的任意列

两者互补不互斥：分区键裁剪交给前者，非分区键裁剪靠后者。我们此前"constraint_exclusion 对声明式分区没用"的说法，仅在"分区键裁剪"语境下成立；一旦你用 CHECK 约束显式声明非分区键的取值范围，constraint_exclusion 立刻变成刚需。

跨版本一致性

constraint_exclusion 在 PG 16.11 与 PG 19 Beta1 上的默认值（partition）、行为、以及本文所有 EXPLAIN 输出逐字节一致。这不是 PG19 新特性，而是 PG 老祖宗就有的能力——只是大多数人没想到它能这么用。
