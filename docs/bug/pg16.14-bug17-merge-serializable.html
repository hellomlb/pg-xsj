Bug #17：MERGE 在 RR/SERIALIZABLE 下静默违反隔离性

✅ SQL 验证通过 2026-05-25 Bug 分析

概述

PG 16.14 修复了一个后果严重却难以发现的 Bug：MERGE 在 REPEATABLE READ / SERIALIZABLE 隔离级别下，遇到并发更新时静默产生错误结果，不报序列化失败。

版本 行为 后果
PG 16.11（有 Bug） MERGE 静默成功 违反隔离性保证，返回错误数据
PG 16.14（已修复） 抛出 could not serialize access due to concurrent update 正确报错，应用可重试

受影响版本：PG 15.0~15.17、16.0~16.13、17.0~17.9、18.0~18.3 中所有 MERGE 语句在 RR/SERIALIZABLE 隔离级别下均有此问题。

现象

用两个并发 psql 窗口复现：

-- 建表（任意窗口）
DROP TABLE IF EXISTS target;
CREATE TABLE target (key int PRIMARY KEY, val text);
INSERT INTO target VALUES (1, 'before');
步骤 窗口 A 窗口 B
1 BEGIN;
2 UPDATE target SET val = 'a' WHERE key = 1;
3 BEGIN ISOLATION LEVEL REPEATABLE READ;
4 MERGE INTO target t USING (SELECT 1 AS key, 'b'::text AS val) s ON t.key = s.key WHEN MATCHED THEN UPDATE SET val = s.val; ← 阻塞
5 COMMIT; ← 解除阻塞
6 16.11: MERGE 0（静默成功 ❌）/ 16.14: ERROR: could not serialize access due to concurrent update ✅
白话：窗口 B 在 REPEATABLE READ 下执行 MERGE，应该看到 key=1, val='before' 的快照。但窗口 A 先提交了 val='a'，窗口 B 的 MERGE 发现 tuple 被改过，16.11 选择假装没事，用新版本继续干活。16.14 正确抛出序列化错误，让应用知道"这个世界在我开始之后变了，你得重试"。

根因分析（源码层面）

Bug 出在 src/backend/executor/nodeModifyTable.c 的 MERGE 执行路径。

PG 的 UPDATE/DELETE 在 RR/SERIALIZABLE 下遇到并发修改的元组时，会调用 IsolationUsesXactSnapshot() 检查：如果是快照隔离，抛出序列化失败。

但 MERGE 路径漏写了这个检查：

// nodeModifyTable.c — 修复前（PG 16.11）
// EvalPlanQual (EPQ) 路径中：
if (IsolationUsesXactSnapshot())  // ← MERGE 路径缺少这行！
    ereport(ERROR,
            (errcode(ERRCODE_T_R_SERIALIZATION_FAILURE),
             errmsg("could not serialize access due to concurrent update")));

// UPDATE/DELETE 路径有这行，MERGE 路径没有

三层根因：

层次 原因
直接原因 MERGE 漏写 IsolationUsesXactSnapshot() 守卫
深层原因 MERGE 的 EPQ 路径从 UPDATE/DELETE 复制代码时遗漏了隔离性检查
根本原因 MERGE 语义复杂（MATCHED / NOT MATCHED 多分支），EPQ 路径代码有 3 处不同入口，漏改其一

修复极简 — +6 行，在 MERGE 的 EPQ 入口补上检查：

// 修复后（PG 16.14）— commit 39b3d2b7d
+ if (IsolationUsesXactSnapshot())
+     ereport(ERROR,
+             (errcode(ERRCODE_T_R_SERIALIZATION_FAILURE),
+              errmsg("could not serialize access due to concurrent update")));

升级影响（重要！）

这是那种"修复后反而报错"的 Bug：

场景 升级前（16.11） 升级后（16.14）
MERGE + RR + 无并发冲突 正常 正常 ✅
MERGE + RR + 有并发冲突 静默成功（错误结果！） 报错 serialization failure

对应用层的影响：

  • 如果你的应用在 RR/SERIALIZABLE 下使用 MERGE，升级后可能突然开始报序列化错误
  • 这不是坏消息 — 老版本在这些场景下静默返回了错误数据，16.14 才是正确的
  • 处理方式：在应用代码中为 MERGE 加入重试逻辑（跟 SELECT FOR UPDATE / UPDATE 一样的模式）
    • # 示例：Python 重试包装
import psycopg2
from psycopg2 import errors

def merge_with_retry(conn, max_retries=3):
    for attempt in range(max_retries):
        try:
            with conn.cursor() as cur:
                cur.execute("""
                    MERGE INTO target t
                    USING (SELECT 1 AS key, 'b'::text AS val) s
                    ON t.key = s.key
                    WHEN MATCHED THEN UPDATE SET val = s.val
                """)
            conn.commit()
            return
        except errors.SerializationFailure:
            conn.rollback()
            if attempt == max_retries - 1:
                raise

    影响评估

    维度 评估
    危害类型 静默数据错误（不报错、不警告、日志无异常）
    触发条件 MERGE + RR/SERIALIZABLE 隔离级别 + 并发更新同一行
    影响范围 PG 15~18 全部主版本中所有使用 MERGE 的应用
    修复版本 16.14 / 15.18 / 17.10 / 18.4
    升级风险 低 — 修复只增加错误抛出，正常路径不变
    隐蔽性 ⭐⭐⭐⭐⭐ 极高 — 结果看起来完全正常，只有对比数据才能发现错误
    国产化迁移意义 从 Oracle MERGE 迁移到 PG MERGE 的应用需额外关注，Oracle 在 SERIALIZABLE 下的行为可能不同

    建议

  • 立即升级：所有使用 MERGE 的 PG 实例应升级到 16.14 / 15.18 / 17.10 / 18.4
  • 审计应用：检查所有 MERGE 语句的隔离级别设置，确认是否有 RR/SERIALIZABLE
  • 加重试逻辑：升级后在 MERGE 调用处加入 SQLSTATE 40001（serialization_failure）重试
  • 监控：关注升级后 serialization_failure 错误率上升 — 这表示之前有静默数据错误被修复了
