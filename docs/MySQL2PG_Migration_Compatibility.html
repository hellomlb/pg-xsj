4.2 PG 独有索引详解

部分索引 — 索引大小减小 90%+

-- 只索引活跃用户
CREATE INDEX idx_active_users ON users (email) WHERE status = 'active';

-- 过滤已删除数据
CREATE INDEX idx_current_orders ON orders (order_date) WHERE deleted_at IS NULL;

BRIN 索引 — 时序大表的救星

-- MySQL: B-tree 索引可能 500GB+
CREATE INDEX idx_created ON logs (created_at);

-- PG: BRIN 索引仅块级元数据，占用 < 1GB
CREATE INDEX idx_created ON logs USING BRIN (created_at) WITH (pages_per_range = 32);

GIN 索引 — JSONB 毫秒级检索

CREATE INDEX idx_doc ON t USING GIN (doc);
SELECT * FROM t WHERE doc @> '{"name": "Alice"}';  -- 毫秒级

4.3 索引并发创建

-- PG: CREATE INDEX CONCURRENTLY 不阻塞读写
CREATE INDEX CONCURRENTLY idx_c ON t (c);

-- PG 16 增强：max_parallel_maintenance_workers 并行创建
SET max_parallel_maintenance_workers = 4;
CREATE INDEX CONCURRENTLY idx_big ON big_table (col);

5. 分区表兼容性

5.1 分区语法对比

-- MySQL 8.0 范围分区
CREATE TABLE orders (
    id INT, order_date DATE, amount DECIMAL(10,2)
) PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- PostgreSQL 16 范围分区
CREATE TABLE orders (
    id INT, order_date DATE, amount NUMERIC(10,2)
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2022 PARTITION OF orders
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');
🔴 最大陷阱：PG 分区键必须是原始列，不能是函数/表达式。
MySQL: PARTITION BY RANGE (YEAR(order_date)) ✅
PG: PARTITION BY RANGE (YEAR(order_date)) ❌ 不支持！
🧪 实测确认：需改为 PARTITION BY RANGE (order_date) + FOR VALUES FROM (date) TO (date)

这是迁移工作量最大的模块。MySQL 和 PG 的过程语言语法完全不同，需要完整重写。

6.1 存储过程代码对比

-- MySQL 存储过程
DELIMITER //
CREATE PROCEDURE get_user(IN user_id INT, OUT user_name VARCHAR(100))
BEGIN
    SELECT name INTO user_name FROM users WHERE id = user_id;
END //
DELIMITER ;

-- PostgreSQL 函数
CREATE OR REPLACE FUNCTION get_user(user_id INT)
RETURNS TEXT AS $$
DECLARE
    user_name TEXT;
BEGIN
    SELECT name INTO user_name FROM users WHERE id = user_id;
    RETURN user_name;
END;
$$ LANGUAGE plpgsql;

-- 调用：MySQL → CALL get_user(1, @name);
-- 调用：PG    → SELECT get_user(1);

6.2 触发器对比

-- MySQL 触发器（一行搞定）
CREATE TRIGGER before_insert_user
BEFORE INSERT ON users FOR EACH ROW
SET NEW.created_at = NOW();

-- PG 触发器（需要先建函数）
CREATE OR REPLACE FUNCTION set_created_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.created_at := NOW();
    RETURN NEW;  -- ⚠️ 必须 RETURN
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER before_insert_user
    BEFORE INSERT ON users
    FOR EACH ROW
    EXECUTE FUNCTION set_created_at();

9. 序列与自增

9.1 三种实现方式对比

-- MySQL：AUTO_INCREMENT
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100)
);

-- PG：SERIAL（向后兼容）
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT
);

-- PG：GENERATED AS IDENTITY（推荐，SQL 标准）
CREATE TABLE users (
    id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name TEXT
);

9.2 ID 迁移关键

-- 迁移时保留 MySQL ID
CREATE TABLE users (id SERIAL PRIMARY KEY, name TEXT);

-- 设置序列起始值 = MySQL 中 MAX(id) + 1
SELECT setval('users_id_seq', 1000000);

-- GENERATED ALWAYS 下，手动插入 ID 需要：
INSERT INTO users (id, name) OVERRIDING SYSTEM VALUE

12.2 高优先级风险项 🚨

  • 1

    字符串大小写敏感

    影响所有 WHERE name = 'xxx' 查询。
    🧪 复验：MySQL SELECT 'Alice' = 'alice' → 1（不区分），PG SELECT 'Alice' = 'alice' → false（区分）
    缓解：统一 LOWER() 或使用 citext 扩展

  • 2

    MySQL 专有函数

    IF() / DATE_FORMAT() / GROUP_CONCAT() 等需逐个改写为 PG 等价函数

  • 3

    LIMIT 参数顺序

    LIMIT offset, count vs LIMIT count OFFSET offset——90% 开发者会踩的坑

  • 4

    GROUP BY 严格性

    MySQL 8.0 默认也是严格的（ONLY_FULL_GROUP_BY=ON），但旧版本可能宽松。迁移前确认客户 sql_mode 设置。

  • 5

    分区键不能是表达式

    PARTITION BY RANGE (YEAR(d)) 在 PG 中不支持，需改为范围边界分区

12.3 PG 迁移优势 🆕

部分索引

索引大小减少 50-90%，查询性能大幅提升

BRIN 索引

时序大表索引体积从 GB 级降到 MB 级

GIN / GiST

JSONB 毫秒级检索、全文搜索、模糊匹配

物化视图

报表直接从物化结果读取，无需实时计算

RETURNING

一次查询获取 INSERT/UPDATE/DELETE 行

JSONB

支持索引、丰富运算符，比 MySQL JSON 强大

RLS 行级安全

多租户场景无需应用层过滤

CTE 递归

树形查询（组织架构、分类树）一行 SQL

并行查询

全表扫描/聚合/JOIN 自动并行化

结论：MySQL 8.0 → PG 16.11 迁移可行，总体兼容度约 85%（需适配 + 完全兼容）。
核心风险集中在 字符串比较语义、专有函数改写、存储过程翻译 三个领域。
迁移收益（部分索引 / JSONB / 物化视图 / BRIN）对大数据量和复杂查询场景的价值往往超过迁移成本。
