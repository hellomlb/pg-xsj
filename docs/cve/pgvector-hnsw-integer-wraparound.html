pgvector 并行 HNSW 整数回绕：一次代码审计发现的理论缓冲区溢出

2026-05-28 CVE 验证 pgvector HNSW 整数回绕 缓冲区溢出 代码审计 安全加固

一、这是什么？

pgvector 是 PostgreSQL 最流行的向量相似性搜索扩展，HNSW（Hierarchical Navigable Small World）是其核心索引算法。在并行构建 HNSW 索引时，共享内存分配逻辑存在整数回绕（Integer Wraparound）漏洞。

项目 内容
漏洞类型 整数回绕导致缓冲区溢出
影响组件 src/hnswbuild.c — 并行 HNSW 索引构建
影响版本 pgvector v0.8.0 ~ v0.8.1
修复 Commit b7e680d
发现者 chungkn (OneMount Group)
严重程度 中危 — 理论可证明，实际触发条件极端
为什么 SQL 不可复现？ 触发整数回绕需要 memoryUsed 接近 SIZE_MAX（64位系统约 18EB），远超实际可分配内存。但通过代码审计 + C 语言逻辑验证，可严格证明漏洞存在。

二、漏洞原理：一根稻草压垮骆驼

2.1 正常流程

并行 HNSW 索引构建的内存分配分三步：

  • HnswBeginParallel() 预估内存 → maintenance_work_mem * 1024
  • InitGraph() 设置 memoryTotal（总配额）和 memoryUsed = 0（已用）
  • 工作进程通过 HnswSharedMemoryAlloc() 递增 memoryUsed，InsertTuple() 检查 memoryUsed >= memoryTotal 时 flush 到磁盘

    • 2.2 问题出在哪？

    核心数据结构 HnswGraph 中两个字段都是 size_t（uint64_t）：

    Size memoryUsed;   // 当前已分配
Size memoryTotal;  // 总内存配额

    漏洞链分三步触发：

    环节 代码 问题
    ① 无上限 graph->memoryTotal = memoryTotal 直接赋值，无 SIZE_MAX/2 保护
    ② 无溢出检查 memoryUsed += MAXALIGN(size) 加法溢出→回绕到小值
    ③ 检查被绕过 memoryUsed >= memoryTotal 回绕后始终为 false，继续分配→越界

    用大白话说：计数器从 99 加到 100 应该报警，但它直接翻回了 0，系统以为还有空间，继续往里塞数据——这时候已经写到别人的地盘了。

    2.3 那个"鸵鸟式" Hack

    当前代码第 977-978 行藏了一段注释：

    /* Report less than allocated so never fails */
InitGraph(&hnswshared->graphData, hnswarea, esthnswarea - 1024 * 1024);

    人为少报 1MB 内存，试图"预留缓冲"。这不是修复——它只是让正常场景不暴露问题，整数回绕的风险依然存在。

    三、四个漏洞点逐帧拆解

    漏洞点 #1：InitGraph 无上限（第 606 行）

    // v0.8.1（有漏洞）
graph->memoryTotal = memoryTotal;  // 可能接近 SIZE_MAX！

// b7e680d（修复后）
graph->memoryTotal = Min(memoryTotal, HNSW_MAX_GRAPH_MEMORY);

    漏洞点 #2：内存分配无溢出检查（第 649 行）

    // v0.8.1（有漏洞）
buildstate->graph->memoryUsed += MAXALIGN(size);  // 可能回绕！

// b7e680d（修复后）— 配合 #1 的上限保护，不再需要单独溢出检查（SIZE_MAX/2 保证不会溢出）

    漏洞点 #3：裸内存检查无安全边距（第 513 行）

    // v0.8.1（有漏洞）
if (graph->memoryUsed >= graph->memoryTotal)

// b7e680d（修复后）
memoryMargin = base == NULL ? 0 : 1024 * 1024;
if (graph->memoryUsed + memoryMargin >= graph->memoryTotal)

    漏洞点 #4：Hack 规避代码（第 977-978 行）

    // v0.8.1（有漏洞）
InitGraph(&hnswshared->graphData, hnswarea, esthnswarea - 1024 * 1024);

// b7e680d（修复后）
InitGraph(&hnswshared->graphData, hnswarea, esthnswarea);

    四、修复对比一览

    漏洞点 修复前 (v0.8.1) 修复后 (b7e680d) 修复方式
    内存上限 无 SIZE_MAX / 2 新增宏常量，硬限制
    上限保护 无 Min(memoryTotal, HNSW_MAX_GRAPH_MEMORY) InitGraph 入口拦截
    预估限制 无 Min(esthnswarea, HNSW_MAX_GRAPH_MEMORY) BeginParallel 入口拦截
    内存检查 memoryUsed >= memoryTotal memoryUsed + memoryMargin >= memoryTotal 加 1MB 安全边距
    Hack 代码 esthnswarea - 1MB 已移除 根因修复后不再需要

    修复的核心思路简单直接：SIZE_MAX/2 一劳永逸。任何合法内存分配都不可能超过这个值，整数回绕从物理上被杜绝。

    五、验证：不能跑 SQL，但能跑逻辑

    5.1 为什么 SQL 不可复现

    限制层 说明
    内存要求 需要约 18EB，64位地址空间的理论上限
    PG 参数 maintenance_work_mem 上限通常不超数 GB
    OS 限制 共享内存（shm）远小于 18EB
    实际场景 "减 1MB" hack 在正常场景下足够掩盖问题

    5.2 三种逻辑验证

    方法一：代码审计（已执行）

    逐行确认 v0.8.1 代码中不存在任何修复 commit 的保护措施：

  • ❌ HNSW_MAX_GRAPH_MEMORY 宏不存在
  • ❌ Min(memoryTotal, ...) 上限检查不存在
  • ❌ memoryMargin 安全边距不存在
  • ❌ Hack 代码仍在第 977-978 行

    • 方法二：C 语言整数回绕 PoC

    #include <stdio.h>
#include <stdint.h>
#include <stddef.h>

int main() {
    size_t memoryTotal = SIZE_MAX;
    size_t memoryUsed  = SIZE_MAX - 100;
    size_t allocSize   = 200;

    memoryUsed += allocSize;  // 回绕！

    printf("=== v0.8.1 模拟 ===\n");
    printf("memoryTotal = %zu\n", memoryTotal);
    printf("memoryUsed (回绕后) = %zu\n", memoryUsed);
    printf("检查: memoryUsed >= memoryTotal ? %s\n",
           memoryUsed >= memoryTotal ? "true" : "false ← 被绕过！");
    return 0;
}

    输出确认：memoryUsed 从 SIZE_MAX - 100 + 200 = 100 回绕成 100，远小于 memoryTotal，检查被绕过。

    通过 git diff b7e680d 确认当前代码缺少 5 项修复。

    六、风险评估与建议

    维度 评估
    可利用性 低 — 需要 18EB 内存，实际不可触发
    影响范围 中 — 影响所有并行 HNSW 索引构建场景
    数据安全 中 — 理论上的缓冲区溢出可能导致数据损坏或崩溃
    综合评级 中危 — 防御性修复，建议升级

    建议

  • 升级 pgvector 至包含 commit b7e680d 的版本
  • 如无法立即升级，限制 max_parallel_maintenance_workers 降低并行构建暴露面
  • 设置合理的 maintenance_work_mem 上限，避免极端配置
